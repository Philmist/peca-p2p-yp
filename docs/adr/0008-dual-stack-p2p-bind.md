# ADR-0008: P2P 待受のデュアルスタック化(`p2p_bind` カンマ区切り複数バインド)

**Status**: Accepted
**Date**: 2026-07-05
**Principles**: Principle II (Security by Design), Principle VI (Principle Traceability)
**関連**: ADR-0006 決定 4(`p2p_bind` は唯一の外部露出ポート)、contracts/p2p-gossip.md
(HELLO `listen_port` の意味論)、FR-016(待受なし=外向きのみ)

## 背景

現状の P2P 待受は単一の `SocketAddr` に単一の `TcpListener` をバインドする構造で、
IPv4 と IPv6 を同時に待ち受けられない。

- `p2p_bind` は単一アドレス文字列(既定 `0.0.0.0:7147`)で、`Settings::p2p_addr()` は
  `Option<SocketAddr>` を返す(空文字=待受無効=`None`)
- `main.rs` は `TcpListener::bind(addr)` を 1 回だけ呼ぶ
- `P2pRuntime::spawn(listener: Option<TcpListener>, …)` は単一リスナーを受け取り、
  `run_listener` が 1 本の accept ループを回す

IPv6 到達性のある環境で IPv6 ピアからの着信を受けるには、IPv4 と IPv6 の両方で
待ち受ける必要がある。

なお `[::]`(IPv6 ワイルドカード)1 本でのデュアルスタック待受(IPv4-mapped 受信)は
OS 依存であり、特に **Windows は既定で `IPV6_V6ONLY=1`** のため `[::]` は IPv6 専用となり
IPv4 着信を受けられない。tokio の `TcpListener::bind` はこのソケットオプションを制御できず、
移植性のあるデュアルスタックにはならない。

## 決定 1: `p2p_bind` をカンマ区切りの複数アドレスとして解釈する

`p2p_bind` を **カンマ区切りの `SocketAddr` リスト**として解釈し、各要素に対して個別に
`TcpListener` をバインドする(アドレスファミリごとに 1 本ずつ、計 N 本のリスナーを持つ)。

- 既定値は `0.0.0.0:7147,[::]:7147`(IPv4 全 IF + IPv6 全 IF を同一ポートで待受)
- 空文字は従来どおり待受無効(外向きのみ — FR-016)。`None` 相当を維持する
- 各要素は前後空白をトリムして個別に `SocketAddr` としてパースする。空要素(連続カンマ・
  末尾カンマ)は無視する
- `Settings::p2p_addr()` の戻り値を単一アドレスから **アドレスの列**へ拡張する
  (空文字は空の列)。`P2pRuntime::spawn` は複数リスナーを受け取り、リスナーごとに
  `run_listener` タスクを起動する(`run_listener` 本体は引数のリスナーに対して汎用のため無変更)
- **検証**: 非空の各要素はパース可能であること。`pcp_bind` / `http_bind` の loopback 強制
  (ADR-0006 決定 4)は本 ADR の対象外で従来どおり(`p2p_bind` は外部露出可)

## 決定 2: 一部バインド失敗は残りで続行する(全滅のみ起動失敗)

指定された複数バインドのうち **一部が失敗しても、成功したリスナーだけで待受を継続する**。

- **理由**: IPv6 未対応・アドレス割当なしの環境では `[::]:7147` のバインドが失敗し得る。
  既定値をデュアルスタックにする以上、片方の失敗で起動不能になっては可用性を損なう。
  「使える方だけで動く」ことが最小権限・堅牢性の観点で安全側
- **全滅時の扱い**: 指定が非空(=待受を意図)であるにもかかわらず **全てのバインドに失敗**した
  場合は、従来の「待受不能=起動失敗」を維持し、非ゼロ終了する。「待受を意図したのに 1 本も
  上がっていない」状態を待受無効(外向きのみ)と黙って同一視しない
- **失敗の可観測性**: 個別バインド失敗は `tracing`(WARN)で当該アドレスとともに記録し、
  黙って握り潰さない。成功が 1 本でもあれば起動を続行する

## 決定 3: HELLO `listen_port` は単一ポート — 全バインドで同一ポートを前提とする

HELLO / PEX が申告する `listen_port` は **単一のポート番号**であり、この意味論は変更しない
(contracts/p2p-gossip.md、ADR-0006 決定 2)。着信したピアは「観測した接続元 host + 申告
`listen_port`」で候補アドレスを合成するため、IPv4 と IPv6 が同一ポートで待ち受けていれば
どちらのファミリからのダイヤルバックも成立する。

- 申告 `listen_port` は **成功したバインドの先頭要素のポート**を採用する
- 複数バインドで異なるポートを指定する運用は、ダイヤルバックの一意性が崩れるため**非推奨**
  (既定値も同一ポート)。異なるポートを禁止はしないが、申告されるのは先頭ポートのみである
  ことを設定説明に明記する
- UPnP は IPv4 NAT のみを対象とする現行仕様を維持する(IPv6 は NAT 前提でないため対象外)。
  マッピング対象ポートは申告 `listen_port` と同一で整合する

## 否定した選択肢

- **`[::]` 単一バインドでのデュアルスタック(`IPV6_V6ONLY=0`)** — Windows 既定と噛み合わず、
  移植性のためには `socket2` 等で明示的にソケットオプションを設定する必要がある。依存追加と
  プラットフォーム分岐を招くうえ、IPv4-mapped アドレス(`::ffff:a.b.c.d`)で着信元が観測され
  PEX 候補生成・自己接続判定・ログの取り扱いが複雑化する。2 本のリスナーを持つ方が単純で
  ファミリの扱いが明快
- **`p2p_bind_v6` を別キーで追加** — キーが増え、「複数バインド」という一般形の特殊ケースに
  なる。カンマ区切りリストなら 1 キーで N バインドを表現でき、将来的に複数 IF への個別バインド
  等にも拡張が効く
- **一部バインド失敗で起動失敗(全 or nothing)** — 決定 2 の理由により却下。IPv6 が使えない
  環境で既定起動が失敗するのは受け入れがたい
