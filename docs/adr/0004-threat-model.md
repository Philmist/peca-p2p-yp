# ADR-0004: 脅威モデル(Sybil・汚染・Eclipse・PEX 毒入れの多層緩和)

**Status**: Accepted
**Date**: 2026-07-03
**Principles**: Principle I (Safety First), Principle II (Security by Design), Principle VI (Principle Traceability)
**Task**: T007(Phase 2 実装前ゲート)。constitution Follow-up TODO: THREAT_MODEL / PRIVACY を本 ADR で解消する

## 背景

本ソフトウェアはインターネットに露出する P2P ノードであり、悪意ある参加者からの
偽登録(Sybil)・一覧汚染・ピアリスト毒入れ・資源枯渇攻撃に晒される
(constitution §Security Requirements「P2P 脅威の緩和」)。
[contracts/p2p-gossip.md §脅威と対応範囲](../../specs/001-nostr-p2p-yp/contracts/p2p-gossip.md)
の表を入力とし、同表および spec(FR-008/FR-012/FR-013)が ADR スコープに委譲した事項を確定する。

## 多層緩和の基本構成(research R8 — v1 の 5 層)

| # | 層 | 内容 | 根拠 |
|---|-----|------|------|
| 1 | 署名検証必須 | 検証失敗イベントは不可視・不格納・不再伝搬 | FR-005 / Principle II |
| 2 | ミュート | pubkey / channel 単位、ローカル保存のみ・非公開 | FR-008 |
| 3 | ピア選別 | 追加・削除・無効化による汚染ピアの切り離し | FR-008 / FR-010 |
| 4 | 任意 PoW(NIP-13) | 既定 0(無効)。閾値未満は保持も再伝搬もしない | FR-008(下記で確定) |
| 5 | 資源上限 | 受信レート・イベントサイズ・EventStore 容量・ログローテーション | FR-007 / R16 |

純粋 P2P では自ノードが再伝搬の担い手になるため、「保持しない = 拡散に加担しない」という
フィルタの二重効果がある(R8)。

## 委譲事項の確定

### 1. Eclipse 攻撃 — 接続先の多様性確保(サブネット分散等): **v1 では採用しない**

- **v1 の緩和(既存)**: manual ピアの優先維持(候補順 manual → last_ok_at → fail_count)、
  外向き(8)/着信(32)枠の分離、PEX 候補の実接続検証(verified まで再接続候補・共有対象に
  しない)。利用者が信頼して手動登録したピアが接続枠から追い出されない構造が v1 の主防壁
- **見送りの理由**: サブネット分散(同一 /16 の接続数制限等)は、攻撃者が複数サブネットに
  ノードを分散配置すれば回避できる一方、同一 ISP に集中する正当な利用者(国内コミュニティは
  地理的に偏る)を誤って排除しうる。5,000 ノード規模・手動シード文化では manual アンカーの
  効果が支配的であり、複雑性に見合わない
- **残余リスク**: 攻撃者が多数の着信可能ノードを長期運用し、PEX 経由で候補リストを占有した
  うえで manual ピアの停止を待つシナリオ。コストは高い(実接続に応答し続ける必要がある)が
  ゼロではない。**利用者側の対処**: 信頼できる manual ピアを複数維持すること(README/T057 に記載)
- **再評価トリガ**: beta 実測、または視界遮断(自分の掲載が他ノードから見えない等)の報告

### 2. EventStore 追い出し — クォータ: **pubkey 単位クォータを採用、ピア単位は見送り**

- **決定**: 同一 pubkey が EventStore に保持できるイベント数の上限を **64** とする
  (`event_store_max` 4,096 の 1/64)。超過時は当該 pubkey の `created_at` が古い順に破棄する。
  1 ペルソナ 64 チャンネル同時掲載まで許容 — 正当利用(数チャンネル)に対し十分な余裕
- **理由**: 単一の鍵が `d` タグを変えながら大量発行して EventStore 全体(古い順破棄)を
  追い出す最安価な攻撃を、鍵あたりのコスト一定で遮断する。実装は挿入時の件数カウントのみで安価
- **ピア単位クォータを見送る理由**: gossip ではイベントが正当な中継ピアから多数届くのが
  正常動作であり、受信元ピア単位の保持クォータは正当な再伝搬経路を罰する。ピアからの
  **流入**はレート制限(256KB/秒・200 msg/秒)が既に束縛している
- **ログ**: クォータ超過による破棄は容量管理(event_store_max 超過破棄)と同種の
  ストア内部方針であり、セキュリティイベントとしない(SecurityEvent 12 カテゴリは変更しない)
- **残余リスク**: 多数の鍵を生成する Sybil には効かない(鍵生成は無料)。多鍵型は
  レート上限(流入束縛)・PoW オプトイン(コスト賦課)・ミュート/ピア切離し(自衛)で受ける
- **実装先**: T016(EventStore)。tasks.md に反映済み

### 3. PEX への第三者アドレス注入(反射・負荷集中): **残余リスクを評価のうえ受容**

- **採用済みの緩和**(contracts/p2p-gossip.md): 未検証ピアの再共有禁止(二次拡散の遮断 —
  増幅率が 1 を超えない)、新規(未検証)候補への接続試行 1 件/秒以下、指数バックオフ
  (5s/×2/300s)、fail_count 8 で降格
- **評価**: 攻撃者が被害者アドレスを PEX 応答に注入しても、(a) 受信ノードはそれを再共有
  しないため拡散は攻撃者の直接接続先に限られる、(b) 各ノードから被害者への反射は
  TCP SYN 1 本/秒未満で、失敗が続けばバックオフ・降格により漸減する。網全体(5,000 ノード)に
  注入できた最悪ケースでも分散した低強度 SYN であり、増幅型 DoS として成立しない
- **受容**: 上記により v1 では追加対策なし。再評価トリガ: 実被害報告、またはノード数が
  想定(5,000)を大きく超えた場合

### 4. イベントの選択的隠蔽・不完全な SYNC 応答: **検出は非目標と確定(残余リスクを記録)**

- 単一ピアの隠蔽は接続度 8 の多重経路により他ピアからの受信で補完される。全接続先が結託して
  遮蔽する状況は Eclipse 攻撃(§1)に帰着する
- 隠蔽の**検出**(応答の完全性監査)は行わない。純粋 P2P では「あるべき全量」の基準が
  存在せず、検出機構自体が偽陽性と複雑性を持ち込むため、意図的な非目標とする
- **残余リスク**: 接続先の大半が結託した場合の部分的な視界欠落。manual ピア維持(§1)が対処

### 5. FR-008 の確定事項(スパム・Sybil)

- **追加緩和手法の要否**: v1 は上記 5 層構成のままとし、**追加しない**。
  Web of Trust・招待制・身元登録はオープン型既定(既定非表示の禁止 — FR-008 MUST NOT)に
  抵触するため採用しない(R8)
- **PoW の既定閾値**: **0(無効)を維持**する。理由: (a) オープン型既定と新規参加障壁ゼロの
  維持、(b) 低スペック環境の配信者を既定で排除しない、(c) スパムが顕在化していない段階での
  コスト賦課は正当利用者の UX 悪化のみ。**帰結**: 既定構成の耐性は資源上限(層 5)による
  減衰と利用者の自衛操作(層 2・3)で提供される(spec FR-008 に明記済み)。
  スパム顕在化時の目安として **8〜16 bits** を設定画面・README(T057)で案内する
- **「一覧の実用性維持」の判定基準**: 大量偽登録(1,000 件規模の注入)下で以下がすべて
  成立すること。検証は T055(セキュリティシナリオ cucumber「大量の偽チャンネル登録への耐性」)
  1. 署名検証済みの正当チャンネルが一覧から**失われない**(偽登録による誤除去ゼロ — SC-005 と同根)
  2. 利用者がミュート(pubkey/channel)と発信元ピアの切り離しを操作した後の一覧で、
     正当チャンネルを識別・選択できる
  3. ノードが資源枯渇(メモリ・CPU・ログディスク)で停止しない(層 5 + ログ集約 —
     data-model §SecurityEvent)

### 6. FR-012 の確定事項(コンテンツ警告)

- **URL 以外の内容ベース警告(名称・説明のヒューリスティック検査)**: **v1 では実装しない**。
  ローカル完結のパターン検査は誤検知・回避が容易で、誤った安心を与える害が利益を上回る。
  v1 の防衛線は (a) URL スキーム検査(http/https 以外に警告 — `url_warning`)、
  (b) UI の「未検証リンクを開く前の確認」(明示操作なしに開かない)の 2 点
- **再評価トリガ**: 実被害パターン(フィッシング誘導等)の蓄積

### 7. FR-013 ペルソナ間リンク推定 — 観測可能情報の限界と対策(ADR-0003 §6 と分担)

プロトコル層(イベント・タグ・鍵)の非リンク保証は [ADR-0003 §6](./0003-persona-key-management.md)。
本節はネットワークメタデータ層を扱う。**接続ピアが観測できる情報**:

| 観測情報 | リンク推定への寄与 |
|----------|--------------------|
| 接続元 IP(TCP) | 同一ノードからの複数ペルソナのイベント初出 |
| 発行タイミング | 再発行周期(60 秒)の位相が揃う |
| 最初の伝搬元ピア | 発行元ノードの推定(接続トポロジ観測) |
| トラッカー IP(`tip` タグ — 公開情報) | **ペルソナを切り替えても同一 IP で配信すれば直接リンク可能** |

- **発行タイミングのジッタ**: **v1 では採用しない**。支配的なリンク信号は接続元 IP と
  トラッカー IP であり、タイミングをぼかしてもこれらは遮断できない。効果が従属的な対策に
  実装・検証コストを払わない
- **根本対策(発行経路の分離 — Tor 等)**: v1 スコープ外
- **v1 で採用する対策(開示)**: UI のペルソナ管理画面に「同一ノード・同一トラッカーからの
  複数ペルソナ掲載は、ネットワーク観測によりペルソナ間の関連を推定されうる」旨の注意を
  常設する(実装先: T032)。誤った匿名性の期待を与えないことが Principle I の要請
- spec FR-013 の MUST(技術的紐づけの禁止)は**プロトコル上の情報に限る**保証であり
  (spec ※但し書き)、本節の観測可能情報はその保証の外であることを確定する

### 8. プライバシー方針 — 配信者トラッカー接続先の公開(constitution TODO: PRIVACY の解消)

- **方針**: 配信者のトラッカー接続先(IP:port)は、現行 YP と同様に**公開情報として扱う**
  (`tip` タグ・index.txt TIP フィールドで全参加者に供給される)。これは PeerCast の
  視聴方式(視聴者がトラッカーへ直接接続)に固有の前提であり、本ソフトウェアが新たに
  導入する露出ではない
- **帰結**:
  1. 配信者のグローバル IP は全参加者(悪意ある者を含む)から収集・記録されうる
  2. firewalled 配信者は `tip` 省略で IP 非公開になるが、v1 では直接視聴不可
     (UI に明示 — contracts/local-api.md)
  3. §7 のとおり、トラッカー IP はペルソナ間リンク推定の最強の信号であり、
     FR-013 の匿名性保証はトラッカー IP には**及ばない**
- **視聴者側メタデータ**: ミュートリストはローカル保存のみで非公開(data-model §MuteEntry)。
  一覧の閲覧は gossip 受信の副産物であり、視聴興味を示すリクエストをネットワークに発しない
  (何を見ているかは接続ピアから観測できない)。視聴開始後のトラッカー接続は PeerCast 本体の
  挙動であり本ソフトウェアのスコープ外

## 残余リスク一覧(v1 受容分)

| リスク | 受容理由 / 対処 | 再評価トリガ |
|--------|-----------------|--------------|
| 多鍵 Sybil による一覧ノイズ | レート上限+自衛操作+PoW オプトイン(§5) | スパム顕在化 |
| 長期運用型 Eclipse | manual ピア維持で対処(§1) | 視界遮断の報告 |
| PEX 反射の低強度 SYN | 増幅率 < 1・レート制限(§3) | 実被害報告 |
| 結託ピアによる選択的隠蔽 | 検出非目標・多重経路で緩和(§4) | Eclipse と同枠 |
| トラッカー IP によるペルソナリンク | 公開情報方針の帰結として UI 開示(§7・§8) | 発行経路分離の要望 |
