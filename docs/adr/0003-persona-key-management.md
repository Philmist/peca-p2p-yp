# ADR-0003: ペルソナ鍵管理(DPAPI 保管・エクスポート・破棄)

**Status**: Accepted
**Date**: 2026-07-03
**Principles**: Principle I (Safety First), Principle II (Security by Design), Principle VI (Principle Traceability)
**Task**: T006(Phase 2 実装前ゲート)

## 背景

匿名文化の維持は「ペルソナ = nostr 鍵ペア」の複数保持・切替・破棄で実現する
(spec Clarifications 2026-07-02、FR-013)。秘密鍵はなりすまし防止(FR-005)の根であり、
その保管・エクスポート・破棄・喪失/漏洩時の扱いを実装開始前に確定する(research R6)。

## 決定

### 1. 保管: Windows DPAPI(ユーザースコープ)

- 秘密鍵は `CryptProtectData`(`windows` クレート)で暗号化した BLOB(`secret_enc`)として
  SQLite に保存する。**平文で保存してはならない (MUST NOT)** — data-model §Persona
- スコープはユーザー単位(`CRYPTPROTECT_LOCAL_MACHINE` は使わない — 同一 PC の他ユーザーから
  復号できないこと)
- 追加パスフレーズは要求しない(自己完結・インストール物を増やさない制約 — research R6)。
  パスフレーズ派生鍵(argon2)は将来のオプションとして排除しない

### 2. エクスポート(nsec 表示)の受け入れ基準

`POST /api/v1/personas/{pubkey}/export`(contracts/local-api.md)に従う:

1. 要求ボディに `{"confirm":true}` 必須 — 欠落は 400
2. 応答前に UI は「秘密鍵を知る者はこのペルソナとして掲載できる」
   「破棄後は復元できず、これが唯一のバックアップ手段である」旨の警告と明示確認を経る
3. nsec は応答本文でのみ返す。**ログ・セキュリティイベントに記録してはならない (MUST NOT)**

### 3. 破棄の非可逆性

- 破棄 = `secret_enc` を含む行の完全削除。**本ソフトウェアからの復元手段は提供しない**
  (匿名性優先 — 破棄したペルソナが後から復元可能だと「破棄した」という利用者の期待を裏切る)
- API は確認フラグ必須(`DELETE /personas/{pubkey}`)
- 状態遷移: `created → active ⇄ archived → deleted(行削除)`(data-model §Persona)

### 4. DPAPI 復号失敗時の挙動

BLOB 破損・別 Windows ユーザープロファイル・OS 再インストール後などで復号に失敗した場合:

- 当該ペルソナを「利用不可」として UI に明示し、**起動と他ペルソナ・閲覧機能は継続する**
  (全体を停止させない — 可用性)
- 復元手段は提供しない。事前の nsec エクスポート(§2)が唯一のバックアップ手段である旨を
  UI で案内する(data-model §Persona で設計済み。実装先: T028/T032)

### 5. 鍵漏洩時の失効表明・視聴者側の回避手段: v1 非目標と確定

spec Edge Cases が本 ADR へ委譲した事項。**v1 では実装しない**と確定する。

- **理由 1(意味論の破綻)**: 失効を表明できるのは当該秘密鍵の署名だけだが、漏洩後は
  攻撃者も同じ鍵で署名できる。「失効イベント」は漏洩者・攻撃者のどちらからも発行でき、
  さらに攻撃者が正当な所有者の鍵を奪って偽の失効を流す DoS(正当ペルソナの抹殺)にも使える
- **理由 2(伝搬保証の不在)**: 純粋 P2P には失効情報を永続保持・保証配布する場所がない。
  鮮度窓(600 秒)で消えるイベントモデルの上では「過去に失効した」ことを新規参加ノードへ
  伝える手段がなく、部分的にしか効かない対策は誤った安心を与える(Principle I に反する)
- **運用ガイダンス(v1 の対処)**: 漏洩が疑われる場合は当該ペルソナを破棄し、新しい
  ペルソナで再掲載する(spec Edge Cases「喪失時は新しいペルソナでの再掲載が可能」と同じ経路)。
  ペルソナの継続性(積み上げた信頼)は失われるが、これはペルソナモデルが受容するコストである
- **再評価トリガ**: 実際の漏洩被害の発生、または相互運用可能な失効表明が NIP として
  確立した場合に本 ADR を改訂する

### 6. ペルソナ間リンク推定への対策(プロトコル層 — 本 ADR の担当範囲)

FR-013 の「システムはペルソナ同士を技術的に紐づけてはならない (MUST NOT)」を
鍵管理・イベント構築の層で保証する:

- `label`(ローカル表示名)・ペルソナの内部 id はネットワークに送出しない(data-model §Persona)
- イベントに他ペルソナの情報を混入させない(contracts/nostr-events.md 発行規則。
  契約テスト T024 で検証)
- 鍵生成は `nostr` クレートの乱数生成に委ね、ペルソナ間で導出関係のある鍵
  (シード共有・HD 導出等)を**使わない** — 導出関係は将来の暗号解析・実装バグで
  リンクの証拠になりうる

**ネットワークメタデータ層の限界**(接続元 IP・発行タイミング・最初の伝搬元ピア・
トラッカー IP を通じたリンク推定)は本 ADR のスコープ外とし、その評価と対策要否は
[ADR-0004(脅威モデル)§FR-013](./0004-threat-model.md) で確定する(spec FR-013 ※但し書き)。

## 否定した選択肢

- **`keyring` クレート(Windows 資格情報マネージャー)** — 保存サイズ制限と列挙性の扱いが
  煩雑(research R6)
- **パスフレーズ派生鍵(argon2)の必須化** — 起動毎の入力が UX を損なう。将来オプション
- **鍵のクラウド/ファイルバックアップ機能** — バックアップ経路自体が漏洩面になる。
  明示的な nsec エクスポート(警告付き)のみとする
- **失効リスト(CRL 相当)の gossip 配布** — §5 理由 1・2 のとおり v1 では成立しない

## 原則参照

- Principle I: 破棄・喪失・漏洩時の挙動を利用者の期待(匿名性・復元不可)に一致させる
- Principle II: OS 標準の鍵保護(DPAPI)・既存暗号ライブラリのみ使用・秘密情報のログ禁止
- FR-005 / FR-013、research R6、data-model §Persona、contracts/local-api.md
