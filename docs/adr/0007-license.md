# ADR-0007: 許容的ライセンス(MIT)の確定と GPL 結合回避の根拠

**Status**: Accepted
**Date**: 2026-07-05
**Principles**: Principle VI (Principle Traceability)
**Task**: T063(Phase 7 Polish、T057 README の前提)。constitution Governance
「ライセンス方針(暫定)」と Follow-up TODO(LICENSE)を本 ADR で解消する

## 背景

constitution Governance は「本プロジェクトは PeerCastStation 等と TCP 経由の API 連携に
留まり GPL の結合著作物には当たらないとの判断のもと、公開前に許容的ライセンス(MIT 相当を
想定)を別 ADR で確定する」と暫定方針を置き、Follow-up TODO に「⏳ LICENSE: 公開前に
許容的ライセンスを ADR で確定(MIT 相当想定、GPL 結合回避)」を残している。research R9 は
PCP 実装をクリーンルームで行う方針を、pcp-announce.md / http-yp.md は事実源の性質を
それぞれ記録済みである。本 ADR はこれらを集約し、ライセンスを確定する。

本ソフトウェアを許容的ライセンスとする目的は、プロトコルとその実装を**クローズドな
プログラムや MIT 等の許容的なプログラムからも利用可能にする**ことである。コピーレフトの
強制はこの目的に反するため採らない。

## 決定 1: 本ソフトウェアは MIT License で公開する

- リポジトリ直下に `LICENSE`(MIT 本文、opensource.org の正規テンプレート準拠)を置く。
  著作権表示は `Copyright (c) 2026 Philmist`(利用者確認済み — 本件の実装関与者は
  利用者(Philmist)と Claude Code のみ)
- ソース先頭へのライセンスヘッダ付与は要求しない(MIT は必須としない)

## 決定 2: PeerCastStation(GPLv3-or-later)との関係で GPL は本ソフトウェアに及ばない

本ソフトウェアが PeerCastStation の派生的著作物(結合著作物)に当たらない根拠を、
**独立した 2 系統**で記録する。いずれか一方だけでも GPL 不伝播は成立する。

### 根拠 A: 連携はプロセス間 TCP のみ —— 結合著作物を構成しない

- 本ソフトウェアは PeerCastStation とアドレス空間を共有せず、静的・動的リンクも
  共有ライブラリ結合も行わない。交換するのは PCP メッセージ(データ)のみである
  (pcp-announce.md: loopback TCP `127.0.0.1:7146`、掲載先 YP として振る舞う)
- 別プロセスが TCP/ソケットで通信する構成は「独立した別プログラム」であり、GPL の
  結合著作物(リンク・同一プロセス実行によって成立)には当たらない。これは FSF 自身の
  GPL 解釈とも一致する
- 却下した選択肢: PeerCastStation へのプラグイン追加(GPL 結合となるため — research R9)

### 根拠 B: 用いたのはプロトコルの「事実」のみ —— 著作権的派生物を構成しない

- 著作権は**表現**を保護し、事実・アイデア・方法・規約(interface / protocol)は
  保護しない。atom 構造・HELO/OLEH のハンドシェイク手順・フィールド配置・index.txt の
  フィールド定義は、相互運用のための**機能的事実**であって著作物ではない
- 本ソフトウェアはこれらの事実のみを用いてクリーンルーム実装した。PeerCastStation の
  ソースコード・独自コメント・創作的なコード構造は参照も複製もしていない(research R9)
- **GPL は著作権ライセンスであり、著作権的な意味で派生物でない成果物には及ばない。**
  「GPL ソース → 誰かが事実を文書化 → その事実から独立実装」という連鎖では、どの段階でも
  著作権的表現の複製が生じないため、GPL は下流へ伝播しない。仮に中間文書が GPL ソースを
  読んで書かれた派生文書だと最大限譲歩しても、そこから本ソフトウェアへ流れるのは依然
  として事実であり、事実に著作権はないため本ソフトウェアは当該文書の派生物にもならない
- **マージ(merger)理論**: 相互運用のためにバイト列・識別子を一致させる必要がある箇所は、
  「そのアイデアを表現する方法が一つしかない」ため表現がアイデアに融合し保護されない。
  4 文字の atom 名(`helo`/`oleh`/`bcst` 等)やフィールド名は短い語・定型句であり著作物性がない

## 決定 3: 事実源の性質と、逐語コピー不在の維持

参照した事実源とその性質を記録する。いずれも GPL 対象のソースコードではない。

| 事実源 | 性質 | 備考 |
|--------|------|------|
| 参考資料 gist(PCP YellowPage protocol) | プロトコルの解説文書(事実の記録) | PeerCastStation 実装準拠の解説。ソースコードではない(research R9) |
| kumaryu/peercaststation wiki「index.txt の仕様」 | 参考仕様文書 | GPL 対象文書ではない参考仕様として利用 — 利用者確認済み 2026-07-04(http-yp.md) |
| 実機検証(2026-07-04 / 07-05)で観測した挙動 | 外部観測された動作の事実 | OLEH の sid/rip 挙動など。ソース由来ではなく観測事実(pcp-announce.md 改訂履歴) |

**維持すべき実装上の線引き**(公開後も遵守する):

- リポジトリの GPL ライセンスと wiki 文書の著作権は別建てであり、「リポジトリが GPL だから
  wiki も GPL」ではない。かつ本ソフトウェアは wiki の**テキストを複製しておらず**、
  記載された事実(フィールド定義等)を利用しているにすぎない
- 相互運用に機能的に必要な範囲を超えて、PeerCastStation 由来のコメント・エラーメッセージ・
  非自明なデータテーブルを**逐語コピーしない**。互換に必要な一致はコピーではなく必然である
- 実装者は GPL ソースコードを複製しない(クリーンルーム姿勢の継続)

## 決定 4: 商標は著作権/GPL と分離して扱う

- 「PeerCast」「PeerCastStation」は商標であり、著作権・GPL とは別の論点である
- 「PeerCastStation 互換」等の**指名的言及(nominative use)は可**とするが、公認・提携・
  出所を示唆する表現は避ける
- agent 名を `peca-p2p-yp/<semver>` と固定で名乗る設計(pcp-announce.md)は、出所の明確化の
  点でも本方針と整合する

## 否定した選択肢

- **GPLv3 / コピーレフト系ライセンス** — クローズド/許容的ライセンスからの利用を可能にする
  という本プロジェクトの目的(決定 1)に反する。GPL 結合の実体がない以上、コピーレフトを
  自発的に課す理由もない
- **Apache-2.0** — 特許条項を伴う点で MIT より保護的だが、本 v1 では最小・広範な互換性を
  優先し MIT を選ぶ(将来必要が生じれば再検討可能)
- **デュアルライセンス** — 単一目的(広く許容的に使わせる)に対して不要な複雑性

## 留保

本 ADR は法律専門家の助言ではなく、一般的なライセンス解釈に基づく作者の判断の記録である。
決定 2 の根拠 A・B は現在の一般的解釈の範囲で GPL 不伝播が成立するとの立場に立つ。残余リスクは
「プロトコル事実の連鎖で GPL が伝播すると認めた判例が存在しない」という不存在リスクにとどまり、
インターフェース再実装はむしろ好意的に扱われてきた経緯がある。実務上守るべきは決定 3 の
「事実の利用」と「表現の複製」の線引きであり、これを継続する限り MIT 公開に支障はない。
