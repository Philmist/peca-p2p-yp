# セキュリティレビュー観点チェックリスト

**Date**: 2026-07-03
**Principles**: Principle II (Security by Design), Principle III (Code Quality and Review)
**位置づけ**: [ADR-0004(脅威モデル)](./0004-threat-model.md) の成果物(T011)。
constitution **実装中ゲート 6**(セキュリティ観点のコードレビュー)が参照するチェックリスト。
番号なしファイル名は ADR 連番との衝突回避のため(tasks.md T011)。

## 適用範囲

**セキュリティに関わる PR** = 次のいずれかに該当する変更を含む PR:

- 信頼境界のコード(`src/p2p/` / `src/pcp/` / `src/web/` / `src/yp/`)の追加・変更
- 入力検証・レート制限・容量上限・ログ出力の追加・変更
- 鍵・ペルソナ・トークン(`src/identity/`、`X-Api-Token` 周辺)の扱いの変更
- 依存クレートの追加・メジャー更新
- `src/security/` の変更

該当しない PR(ドキュメントのみ、UI 文言のみ等)は適用不要。判断に迷う場合は適用する。

## チェックリスト

| # | 観点 | 確認内容 | 根拠 |
|---|------|----------|------|
| 1 | 入力検証(サイズ) | ネットワーク・API から受けるデータに上限(フレーム 64KB / イベント 16KB / atom 64KB / JSON ボディ 64KB / ヘッダ 8KB 等)が適用されているか。新しい受信経路に上限漏れがないか | Principle II / FR-007 / 各契約 |
| 2 | 入力検証(形式・内容) | パース→形式→内容の多段検証が契約の順序どおりか。検証前のデータを後段(格納・再伝搬・表示)に渡していないか | Principle II / contracts 受信検証 |
| 3 | 信頼境界ごとの強度 | 違反時の扱いが境界の露出度に一致しているか(p2p = 破棄+切断+ログ、PCP(loopback)= 切詰め許容、HTTP = 定型エラー)。強度を弱める変更に契約上の根拠があるか | contracts/pcp-announce.md「切詰め許容の根拠」 |
| 4 | エラー応答の情報漏洩 | エラー・CLOSE reason・ログにパス・スタックトレース・内部状態を含めていないか。応答は定型コードのみか | Principle II / Security Requirements |
| 5 | 最小権限・バインド | 新しい待受・エンドポイントが既定 loopback か(外部露出は p2p_bind のみ)。`pcp_bind`/`http_bind` の loopback 強制(ADR-0006 決定 4)を迂回していないか。変更系 API に `X-Api-Token` + `Host` 検証が適用されているか | ADR-0006 / contracts/local-api.md |
| 6 | 暗号 | 署名・検証・鍵生成が既存ライブラリ(`nostr` クレート・DPAPI)経由か。自前の暗号処理・乱数の独自実装を追加していないか | Principle II (MUST NOT) |
| 7 | 秘密情報の取り扱い | 秘密鍵(nsec / secret_enc)・トークンをログ・セキュリティイベント・エラー応答・イベントタグに出力していないか。ペルソナのローカル情報(label 等)をネットワークに送出していないか | ADR-0003 / FR-013 |
| 8 | セキュリティイベントログ | 拒否・違反が data-model §SecurityEvent の 12 カテゴリのいずれかに対応付いているか。カテゴリの新設・変更は data-model を先に更新しているか。高頻度イベントの集約・ローテーション(ログ DoS 耐性)を損なう変更でないか | FR-007 / data-model §SecurityEvent |
| 9 | レート制限・資源上限 | 新規/変更エンドポイント・受信経路にレート制限が適用されているか。無制限に成長するメモリ構造(キャッシュ・キュー・リスト)を導入していないか(上限 + 破棄規則があるか) | Security Requirements / R16 |
| 10 | P2P: 未検証情報の不使用 | 他ノード由来の情報(PEX アドレス・HELLO 申告値)を検証(実接続)前に使用・再共有していないか。`ts`・`listen_port` 等の申告値をイベント検証・接続判断に使っていないか | FR-015 / contracts/p2p-gossip.md |
| 11 | P2P: 伝搬の不変条件 | 重複抑制・置換規則・「格納成功イベントのみ再伝搬」「受信元を除く」を変更していないか。変更する場合、PlusCal モデル(ADR-0005)を先に更新・再検査したか。DedupCache 保持期間 ≥ 鮮度窓の連動(ADR-0005 設計制約)を壊していないか | Principle V / ADR-0005 |
| 12 | 依存クレート | 新規依存は必要最小限か(監査面積)。`cargo audit` / Trivy を通過するか。`nostr-sdk` 等の禁止依存(ADR-0002)を持ち込んでいないか | Principle III / ADR-0001 / ADR-0002 |
| 13 | テスト対応 | 変更に対応するネガティブシナリオ(不正入力・境界値・攻撃パターン)がテストに追加されているか。セキュリティ挙動の変更が Gherkin シナリオ(spec)と乖離していないか | Principle IV |
| 14 | コメント | セキュリティに関わる複雑なロジック(検証順序・境界条件・レース回避)に意図のコメントがあるか | Principle III (MUST) |

## 運用(constitution 実装中ゲート 6)

1. セキュリティに関わる PR の作成者は、PR 説明に本チェックリストの**適用結果を記録する
   (MUST)**。形式は次のとおり(該当しない項目は「非該当」で可):

   ```markdown
   ## セキュリティレビューチェックリスト適用結果
   - [x] 1 入力検証(サイズ): p2p フレーム上限 64KB を維持(変更なし)
   - [x] 4 エラー応答: 新規エラーは定型コード `invalid_value` のみ
   - [ ] 7 秘密情報: **要レビュー** — export 応答のログ抑制を変更したため重点確認
   - 2,3,5,6,8..14: 非該当(理由: ...)
   ```

2. レビュアは記録された適用結果を確認し、重点項目(上例の「要レビュー」)を検証する
3. 本チェックリストの項目追加・変更は ADR-0004 の改訂(または本ファイルの PR + ADR 参照更新)による
