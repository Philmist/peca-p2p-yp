# ADR-0010: DNS 解決が必要なピアの限定サポート(manual 登録のみ)

**Status**: Accepted
**Date**: 2026-07-07
**Principles**: Principle I (Safety First), Principle II (Security by Design), Principle VI (Principle Traceability)
**Issue**: #3(設計判断は grill セッションで確定。本 ADR は決定と受容した残余リスクを記録する)

## 背景

現状のピア登録は生 IP を前提とする。`parse_addr`(`src/p2p/peers.rs`)はブラケットなし
ホストを寛容に素通しし、`TcpStream::connect(&str)`(`src/p2p/runtime.rs`)が偶然 DNS 解決
するため、ホスト名ピアは**中途半端に半サポート**の状態にあった(意図的設計ではない)。

DDNS で規定される家庭ノードや、VPS で提供する初期シードを `seed.example.org:7147` の形で
登録したいという要求がある。とりわけ [ADR-0004](./0004-threat-model.md) §1 が Eclipse 防壁の
中核に据える「manual アンカーの優先維持」は、動的 IP の家庭ノードが安定アンカーになれないと
痩せる。DNS/DDNS 対応はこの防壁を実効化する。

## 信頼境界との整合(再検証)

本 ADR の決定が [CONTEXT.md](../../CONTEXT.md) §信頼境界・ADR-0004・ADR-0006 と整合することを
再検証した:

- **P2P gossip が唯一の外部露出**(CONTEXT.md)であり、最大の攻撃面である。DNS 解決の導入で
  新たに増える露出は「自ノードが自分の manual リストのホスト名を OS リゾルバへ問い合わせる」
  経路のみ。網から受け取った値でホスト名解決は**行わない**(名前空間分離。下記決定 3)。
- ADR-0006 決定 2:制御メッセージ(PEERS 含む)の経路上改竄は署名保護外で、実接続検証で
  緩和する。DNS ピアも同じ枠組みに載る(未検証 IP を触れ回らない。下記決定 4)。
- ADR-0004 §4「検出は非目標」:DNS 解決失敗を新たな攻撃信号として検出しない(下記決定 6)。
- **ADR-0004 は変更しない**。固定 12 種の `SecurityCategory` を維持する。

## 決定

### 1. サポート方針

- DNS 解決ピアは **manual 登録に限り**許容する。PEX/自動獲得経路では許容しない。
- DNS 固有リスク(解決経路の乗っ取りによる接続先誘導)は **生 IP manual ピアと同格**として
  受容する(生 IP も BGP/経路乗っ取りの同種リスクを既に受容済み)。
- **逆引き(PTR)は実装しない。**

### 2. 同一性・解決タイミング

- gossip の `nonce` は自己接続検出専用のノード単位乱数で、リモートピアの安定 ID ではない。
  HELLO に pubkey は無く、**ピア同一性 = アドレス文字列**。
- DNS ピアの canonical キー = **ホスト名文字列**。**ダイヤル毎にフレッシュ解決**する
  (登録時 IP 固定は DDNS の意味を消すため不可)。
- 着信 IP との統合(順引き突合含む)はしない。→ 同一物理ピアへの**二重接続**(外向き枠+
  着信枠を各 1 消費)を**受容**する。DedupCache が内容重複を吸収するため正しさは壊れない。

### 3. 名前空間の分離(中核 invariant)

> **ホスト名は「利用者ローカルの manual リスト」にのみ存在する。PEX/gossip の名前空間は
> IP リテラルのみ。**

双方向で強制する(`src/p2p/pex.rs`):

- **送出**:`select_peers_for_pex` はホスト名ピアを IP に射影して共有する(決定 4)。
  ホスト名自体は網に出さない。
- **受信**:`validate_incoming_peers` は**ホスト名形式を拒否**し、IP リテラルのみ候補化する
  (既存の「ホスト名候補を受理してしまう」穴の厳格化)。

### 4. PEX での能動共有(解決済み IP を共有)

- DNS manual ピアも PEX で共有するが、中身は**接続成立時の実ソケット IP
  (`stream.peer_addr()`、検証済み)**とする。
- **PEX 送出時にホスト名を再解決した IP を載せてはならない**(未検証 IP を verified として
  触れ回るのは research R14 / MUST NOT 違反)。共有するのは「直近に自ノードが実際に握手した
  IP」に限る。まだ外向き成功していないホスト名ピアは共有しない。
- 効果:VPS シードのブートストラップ伝播が加速する。IP は ADR-0004 §8 で公開情報扱いのため
  新たな漏洩は生まない。

### 5. ストア表現(単一エントリ + 派生属性)

- `peers` に **`resolved_ip TEXT NULL` 列を追加**(冪等マイグレーション。旧 DB は
  `ALTER TABLE ADD COLUMN`)。ホスト名 1 エントリ(`source=manual, addr=ホスト名`)に対し、
  外向き成功時の実 IP を canonical 表記で記録する。
- `resolved_ip` は **PEX 射影専用**。**ダイヤルには一切使わない**(外向きは常にホスト名を
  再解決 → 自己二重接続を作らない)。
- 永続化し成功毎に上書きする。再起動直後に再検証前の `resolved_ip` を共有しうるが、これは
  既存の「verified IP ピアを再起動後に再検証せず共有する」挙動と同格。
- 削除で行ごと一括消去(単一エントリ)。
- 2 エントリ案(manual ホスト名 + 自動生成 verified IP)は却下:候補選定が両方を掴んで
  自己二重接続を常態化させ、DDNS の IP 変化で古い IP 行の churn/削除漏れを生むため。

### 6. 解決の実行方式と失敗の扱い

- 解決は**暗黙のまま**(`TcpStream::connect(&addr)` に OS リゾルバを委ねる)。成功時に
  `stream.peer_addr()` で**実際に繋がった IP**を捕捉し `resolved_ip` に記録する(複数 A/AAAA は
  OS が順に試行、繋がったものが検証済み IP)。
- **DNS 解決失敗と TCP 接続失敗は区別しない**。どちらも `Err(_)` → `failure_backoff`
  (fail_count+1 → 指数バックオフ → 降格)で同一扱い。
- **新しい `SecurityCategory` は追加しない**(固定 12 種を維持 — ADR-0004 は変更しない)。
  解決失敗は攻撃信号でなく運用/設定ミスで、DNS 改竄と NXDOMAIN をローカル区別できず偽陽性の
  山になる(ADR-0004 §4「検出は非目標」と整合)。通常接続失敗と同じく debug ログ + fail_count。
- 受容する残余リスク:暗黙解決はブロッキング getaddrinfo を tokio のブロッキングプールで
  走らせるため、ハングする多数ホスト名でプール圧迫の理論的余地がある(manual・数個規模で
  無視できる)。複数レコードの happy-eyeballs は OS 順序任せ(自前実装しない)。

### 7. ホスト名バリデーション(manual 経路のみ)

- **分類器**:host が `Ipv4Addr` としてパース可 → IP リテラル。不可 → ホスト名検証へ
  (IPv6 はブラケット経路で既に IP リテラル)。この分類器を決定 3/4 の「IP そのまま/ホスト名は
  resolved_ip 差し替え」判定に用いる(`PeerAddr::is_hostname`)。
- **RFC 1123**:各ラベル `[a-z0-9-]`、先頭/末尾ハイフン禁止、ラベル ≤63、ホスト全体 ≤253。
- **ASCII のみ**(IDN は利用者が punycode `xn--` へ変換して入力)。非 ASCII は拒否(既存の
  ASCII 小文字化正規化を安全に保つ)。
- **アンダースコア拒否**(厳格 RFC 1123)。
- **末尾ドットは正規化時に 1 個除去**(`seed.example.org` と `seed.example.org.` の二重登録防止)。
- **ドット必須にしない**(単一ラベル `localhost` 等を許容。hosts 等で解決しうる。引けなければ
  resolution 失敗 → fail_count で自然降格)。
- 適用範囲は **manual 登録経路のみ**。PEX 経路は分類器で IP リテラル以外を弾く。

### 8. スコープ外(別 Issue)

- 「特定ペルソナが裏書きするノードか」を検証する **npub 対**は本 ADR から切り離す。認証付き
  シード配布(署名記録のインポート形式拡張)として別 Issue で扱う。接続認証(ハンドシェイクでの
  ノード↔pubkey 束縛)は**採らない**(ノードとペルソナを結びつけ FR-013 / ADR-0004 §7 の
  匿名性保証を破壊しうるため)。

## 残余リスク一覧(v1 受容分)

| リスク | 受容理由 / 対処 | 再評価トリガ |
|--------|-----------------|--------------|
| DNS 解決経路の乗っ取りによる接続先誘導 | 生 IP manual ピア(BGP/経路乗っ取り)と同格。manual 限定・PEX 非伝播で影響を局所化(決定 1・3) | 実被害報告 |
| 同一物理ピアへの二重接続(外向き+着信) | DedupCache が内容重複を吸収し正しさは不変。枠消費のみ受容(決定 2) | 枠圧迫の実測 |
| 再起動直後の古い resolved_ip 共有 | 既存の verified IP ピア再共有と同格(決定 5) | — |
| ブロッキング getaddrinfo によるプール圧迫 | manual・数個規模で無視。暗黙解決を維持(決定 6) | 大量ホスト名登録の需要 |

## 原則参照

- Principle I: 誤った安全性の期待を作らない(DNS リスクを生 IP と同格として明示受容)
- Principle II: trust nothing(実接続検証・未検証 IP の不共有・名前空間分離)、攻撃面最小化
  (網由来値でのホスト名解決を行わない)
- Principle VI: Issue #3 の確定判断と残余リスクの ADR 化
