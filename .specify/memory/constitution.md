# peca-p2p-yp Constitution

## 規範キーワード

本文書の MUST / MUST NOT / SHOULD / SHOULD NOT / MAY は RFC 2119 の定義に従う。
SHOULD 要求は、正当な理由が記録される限り満たさないことを許容する。

## Core Principles

### I. Safety First (NON-NEGOTIABLE)

ユーザーの安全はすべての技術的決定に優先する。安全性を損なう可能性がある変更は、
代替手段が存在する限り採用してはならない。

- すべての機能はユーザーへのリスク評価を伴わなければならない (MUST)
- 安全性に影響する変更は、実装前に明示的なレビューを経なければならない (MUST)
- ユーザーデータの機密性・完全性・可用性を保護しなければならない (MUST)
- 未緩和の High または Critical 深刻度の既知脆弱性を含むコードはリリースしてはならない (MUST NOT)
- Low / Medium の既知脆弱性は、影響評価と緩和方針をイシューに記録した上で許容してよい (MAY)
- ユーザーを危険にさらす可能性のあるP2Pネットワーク上の悪意あるコンテンツを
  検出または警告する仕組みを持たなければならない (MUST)

**Rationale**: P2Pネットワーク環境では、悪意ある参加者からユーザーを保護することが
根本的な責務である。イエローページサービスは多数のユーザーが依存するインフラであり、
ユーザーの安全を後付けにすることは許容されない。

### II. Security by Design

セキュリティは実装の後追いではなく、設計段階から組み込まれなければならない。

- すべてのネットワーク通信は入力検証を経なければならない (MUST)
- 外部から受け取るデータは信頼してはならない (MUST) — trust nothing from the network
- 最小権限の原則に従って実装しなければならない (MUST)
- セキュリティ上の設計決定はADRに記録しなければならない (MUST)
- 暗号化・認証・認可の実装は既存の暗号論的に安全なライブラリに基づかなければならない (MUST)
- 自前の暗号アルゴリズムを実装してはならない (MUST NOT)

**Rationale**: P2Pイエローページサービスはネットワーク境界に位置するコンポーネントであり、
攻撃対象領域が広い。設計時のセキュリティ組み込みにより、後からのパッチ適用コストと
リスクを大幅に削減できる。Principle Iへの直接的な技術的根拠となる。

### III. Code Quality and Review

コードの品質はユーザー安全の技術的基盤である。低品質なコードはバグを生み、
バグはセキュリティ上の脆弱性となりうる。

- すべての変更はレビューを経なければならない (MUST)
- レビューは作者以外の開発者によって行われることが望ましい (SHOULD)
- セキュリティに関わる変更は、レビュー観点チェックリストの適用結果を記録しなければならない (MUST)
- 静的解析ツールによるチェックをCIに組み込まなければならない (MUST)
- 依存ライブラリのセキュリティ更新は速やかに適用しなければならない (MUST)
- 技術的負債は意識的に管理し、ADRまたはイシューに文書化しなければならない (MUST)
- コードは読みやすく、保守可能でなければならない (MUST)
- セキュリティに関わる複雑なロジックは必ずコメントで意図を記述しなければならない (MUST)

**Rationale**: 高品質なコードベースは、バグの発見と修正を容易にし、
セキュリティ脆弱性の混入を防ぐ基盤となる。Principle Iのユーザー安全を
実現するための技術的土台である。

### IV. Behavior-Driven Testing with Gherkin

実装上重要なすべての要求はGherkin記法による振舞い記述で定義されなければならない。

- ユーザーストーリーはGherkin (Given/When/Then) で記述しなければならない (MUST)
- セキュリティ要件はGherkinシナリオとして記述し、自動テストで検証しなければならない (MUST)
- ネガティブシナリオ（不正入力・攻撃パターン・境界値）をシナリオに含めなければならない (MUST)
- BDDシナリオはspec.mdに記録し、実装との乖離を防がなければならない (MUST)
- テストはGherkinシナリオに対応付けられ、シナリオが失敗する状態を確認してから
  実装を開始しなければならない (MUST)
- 受け入れテストはGherkinシナリオから自動生成または対応付けられなければならない (SHOULD)

**Rationale**: Gherkinによる振舞い記述は、要求の曖昧さを排除し、
セキュリティ要件が実装に確実に反映されることを保証する。
ネガティブシナリオの明示的な記述により、攻撃シナリオへの対応漏れを防ぐ。

**Gherkin セキュリティシナリオ例**:

```gherkin
Feature: Yellow Pages エントリ登録のセキュリティ
  Scenario: 過度に大きいペイロードの拒否
    Given ネットワークに悪意ある参加者が存在する
    When 上限を超えるサイズのエントリ登録リクエストが送信された
    Then リクエストは拒否されなければならない
    And エラーは内部情報を漏洩してはならない
    And セキュリティイベントとしてログに記録されなければならない
```

### V. Formal Verification for Critical Paths

新規に設計する非自明な並行アルゴリズムおよびプロトコル状態機械は、
PlusCal または TLA+ による形式的検証で設計段階のバグを発見することが強く推奨される。

**「クリティカル」と判断する基準（すべてを満たすもの）**:
- 新規設計であり、既存の実績ある仕様・ライブラリの単純な利用ではない
- 競合状態・デッドロック・プロトコル違反がテストで再現困難な非自明さを持つ
- 失敗がユーザー安全（Principle I）またはデータ整合性に直接影響する

**要件**:
- 上記すべてを満たすアルゴリズムはPlusCalモデルを作成するべきである (SHOULD)
- モデルを作成する場合、デッドロック・ライブロック・不変条件違反をチェックしなければならない (MUST)
- .tlaファイルと検証結果は `docs/formal/` ディレクトリに保存しなければならない (MUST)
- クリティカルと判断した／しなかった理由は、該当機能の設計時にADRで明示しなければならない (MUST)
- 既存 PeerCast YP プロトコルに準拠する状態機械は、状態遷移が既知のため対象外とする

**Rationale**: P2Pプロトコルの並行処理部分は、テストのみでは発見できない
微妙な競合状態やプロトコル違反が存在しうる。PlusCalによる形式的検証は
設計時にこれらのバグを根絶でき、Principle Iのユーザー安全を強力に支援する。

### VI. Principle Traceability

すべての技術的決定と実装の選択は、本コンスティテューションのどの原則に基づくかを
明示しなければならない。

- ADRは参照する原則番号（例: Principle II）を記載しなければならない (MUST)
- plan.mdのConstitution Checkセクションは具体的な原則参照を含まなければならない (MUST)
- 原則に反する決定は正当化の理由を文書化し、代替案が存在しないことを示さなければならない (MUST)
- spec.mdのGherkinシナリオはどの原則を検証するかを示さなければならない (SHOULD)

**Rationale**: 追跡可能性がなければ、原則は形骸化する。
すべての決定が原則に遡及できることで、コンスティテューションが生きた文書であり続け、
Principle Iのユーザー安全が実際の実装に反映されることを保証できる。

## Security Requirements

以下のセキュリティ要件はすべての実装に適用される。Principle IおよびIIから導出される。

- **入力検証**: ネットワークから受け取るすべてのデータをサイズ・形式・内容の面で検証しなければならない
- **エラーハンドリング**: エラーメッセージは内部実装の詳細・スタックトレース・パスを漏洩してはならない
- **セキュリティログ**: 接続拒否・不正なリクエスト・認証失敗はログに記録しなければならない
- **依存関係**: サードパーティライブラリはセキュリティ監査された版を使用し、定期的に更新しなければならない
- **レート制限**: イエローページエントリの登録・取得はレート制限を設けてDoS攻撃を緩和しなければならない
- **コンテンツ検証**: 登録されるエントリの内容は許可されたフォーマットに準拠しているか検証しなければならない
- **P2P 脅威の緩和**: 大量の偽登録（Sybil）や虚偽・なりすましエントリによる YP 情報の汚染に対する緩和策を設計に含めなければならない。具体的手法は脅威モデル ADR で定める

## Development Workflow & Quality Gates

### 実装前ゲート (Principle IV, V, VI)

1. **振舞い定義**: Gherkinシナリオの作成と承認 (Principle IV)
2. **形式的検証**: クリティカルコンポーネントのPlusCalモデル作成と検証 (Principle V)
3. **Constitution Check**: plan.mdの該当セクションで原則適合を確認 (Principle VI)
4. **セキュリティリスク評価**: 変更がユーザー安全に与える影響を評価 (Principle I)

### 実装中ゲート (Principle III, IV)

5. **テストファースト**: Gherkinシナリオに対応するテストが失敗する状態で実装開始 (Principle IV)
6. **セキュリティ観点のコードレビュー**: Principle IIのチェックリストを使用 (Principle II, III)
7. **CI通過**: 静的解析・依存関係チェック・自動テストすべてがパス (Principle III)

### リリース前ゲート (Principle I, II)

8. **Gherkinシナリオ完全通過**: すべてのシナリオ（ネガティブシナリオ含む）が自動テストでパス
9. **セキュリティスキャン**: 既知の脆弱性チェックの通過
10. **ドキュメント更新**: ADRおよびdocs/formal/の最新化

## Governance

本コンスティテューションはすべての他のプラクティス・慣習に優先する。

**改正手順**:
1. 変更内容・理由・参照する原則番号をADR草案に記す
2. ユーザー安全（Principle I）への影響を評価する
3. 本ファイルと依存テンプレートを同時に更新し、バージョンを上げる (MUST)
4. 改正は独立したレビュー（作者以外による）を経ることが望ましい (SHOULD)

**バージョニングポリシー**:
- MAJOR: 原則の削除・後方互換性のない再定義
- MINOR: 新しい原則・セクションの追加または実質的な拡張
- PATCH: 明確化・表現の修正・非意味的な改訂

**コンプライアンスレビュー**:
- すべてのPRはplan.mdのConstitution Checkセクションを通過しなければならない (MUST)
- 原則に反する実装が発見された場合は、即座に是正しなければならない
- コンスティテューションの有効性は、beta リリース時および以降の各マイナーリリース時にレビューする

**ライセンス方針（暫定）**:
本プロジェクトは PeerCastStation 等と TCP 経由の API 連携に留まり GPL の結合著作物には
当たらないとの判断のもと、公開前に許容的ライセンス（MIT 相当を想定）を別 ADR で確定する。

ランタイム開発ガイダンスは `docs/agents/domain.md` を参照。
