# Feature Specification: 掲載前のペルソナ選択(選択中ペルソナの明示的な切り替え)

**Feature Branch**: `feature/persona-selection`

**Created**: 2026-07-07

**Status**: Draft

**Input**: User description: "配信を開始する前にユーザーが「名乗るペルソナ(グローバルな選択中ペルソナ = selected)」を選択できる機能"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 配信を始める前に名乗るペルソナを選ぶ (Priority: P1)

配信者は、上流(PeerCast)で配信を開始する前に、YP 上でどのペルソナ(公開鍵アイデンティティ)として名乗るかを能動的に選択したい。現状は「選択中ペルソナ」を切り替える操作がUIに存在せず、複数ペルソナを持っていても意図したペルソナで名乗れているか確信を持てない。

**Why this priority**: これがこの機能の中核。掲載は PCP チャンネル到着で自動発行されるため、「どのペルソナで名乗るか」は掲載が起きる前に決めておく以外に選択の機会がない。ここが欠けていると、ユーザーは意図しないペルソナで公開鍵アイデンティティをネットワークに晒してしまう(誤爆)。

**Independent Test**: ペルソナ管理画面で複数の有効ペルソナを用意し、片方を「選択」→ もう片方を「選択」と切り替え、その都度「現在選択中」表示とチャンネル一覧画面の選択中ペルソナ表示が即座に追随することを確認するだけで、単独で価値(=意図したペルソナで名乗れる確信)を検証できる。

**検証する原則**: Principle I(意図したペルソナで名乗れることは誤爆=プライバシー事故の防止)。選択可能ガードは Principle II(バックエンド強制)。

**Acceptance Scenarios**:

1. **Given** 有効(active かつ利用可能)なペルソナが2つあり何も配信していない状態、**When** 一覧で片方の「選択」を押す、**Then** そのペルソナが選択中になり、「現在選択中」バナーとチャンネル一覧画面の選択中表示が即座に更新される。
2. **Given** ペルソナが1つも無い状態、**When** ペルソナ管理画面を開く、**Then** まず作成を促す導線が示され、チャンネル一覧画面の選択中表示は「未選択(ペルソナを作成してください)」と表示される。
3. **Given** ペルソナを初めて1つ作成した直後、**When** 一覧を見る、**Then** その最初のペルソナが自動的に選択中になっている(追加操作不要で名乗れる状態)。
4. **Given** 既に選択中ペルソナがある状態で2つ目以降のペルソナを作成した、**When** 作成完了後の状態を見る、**Then** 選択中は自動では移らず、元のペルソナのままである。

---

### User Story 2 - 配信中は名乗っているペルソナを凍結する (Priority: P1)

配信者は、配信(掲載)が進行している最中に、名乗っているペルソナを誤って切り替えたり、そのペルソナを破棄・アーカイブしたりしてしまわないよう保護されたい。配信中にペルソナが入れ替わると、ネットワーク観測者に「旧ペルソナと新ペルソナが同一人物」というリンクを推定させる強いシグナルを与えてしまう。

**Why this priority**: プライバシーの中核的な保護。複数ペルソナを使い分ける唯一の目的は「同一人物だと結び付けられないこと」であり、配信中のペルソナ入れ替えはその目的を根本から破壊する。P1 の User Story 1 と対で初めて安全な選択機能が成立する。

**Independent Test**: 1つ以上のチャンネルを掲載中(配信中)にした状態で、選択中ペルソナの切り替え・破棄・アーカイブがUIで無効化され、かつ直接APIを叩いても拒否されることを確認。配信を停止して全チャンネルが終了した後は再び操作できることを確認。

**検証する原則**: Principle I(配信中のペルソナ入替=リンク推定シグナルの構造的防止)。直接 API バイパス試行の拒否(ネガティブシナリオ)は Principle II/IV。

**Acceptance Scenarios**:

1. **Given** 選択中ペルソナで1つ以上のチャンネルを配信中、**When** 別ペルソナへの切り替えを試みる、**Then** UIでは操作が無効化されており、API直接呼び出しでも拒否され、選択中ペルソナは変わらない。
2. **Given** 選択中ペルソナで配信中、**When** その選択中ペルソナの破棄またはアーカイブを試みる、**Then** いずれも拒否され、配信は継続する。
3. **Given** 選択中ペルソナで配信中、**When** その選択中ペルソナの表示名(ラベル)変更を試みる、**Then** 許可される(署名アイデンティティは変わらないため)。
4. **Given** 配信中、**When** 配信に関与していない別のペルソナを作成・アーカイブ・破棄する、**Then** いずれも許可される(選択中ペルソナ以外は制限対象外)。
5. **Given** 配信中で選択操作がロックされている、**When** 上流で配信を停止し全チャンネルが終了する、**Then** ロックが解除され、再びペルソナを選択・破棄・アーカイブできる。

### User Story 3 - 誤ったペルソナで名乗っていないか常に自覚できる (Priority: P2)

配信者は、今どのペルソナで名乗っているのかを、掲載中のチャンネルを見る画面で常に確認できたい。プライバシー上の誤爆を未然に防ぐには、選択中ペルソナが常時見えていることが重要。

**Why this priority**: User Story 1/2 が「正しく選ぶ・凍結する」を担うのに対し、これは「気付ける」を担う補完。確認ダイアログを設けない代わりの誤爆防止手段であり、単体でも自覚向上の価値がある。

**Independent Test**: チャンネル一覧画面を開き、現在の選択中ペルソナが人間可読(表示名+短縮した識別子)で読み取り専用表示されること、選択を切り替えると追随することを確認。

**検証する原則**: Principle I(常時明示による誤爆の未然防止)。

**Acceptance Scenarios**:

1. **Given** 選択中ペルソナがある状態、**When** チャンネル一覧画面を開く、**Then** 現在名乗っているペルソナが人間可読に読み取り専用で表示される。
2. **Given** 選択中ペルソナが後からアーカイブまたは利用不可(鍵復号失敗)になった、**When** 各画面を見る、**Then** 「現在選択中」表示が警告状態になり、再選択を促され、掲載は保留状態に落ちる。

---

### Edge Cases

- **保留中(未発行)チャンネルがあるだけの状態**: PCP チャンネルは到着しているが選択中ペルソナが無く、まだネットワークへ発行していない場合、これは「配信中」に含めない。よって最初のペルソナ選択は常に許可され、選択によって保留が解ける。
- **配信の異常終了**: PCP の異常切断・強制終了でも、当該チャンネルは終了として最終発行され配信中状態から除去されるため、ロックは確実に解除される(配信中状態が取り残されない)。
- **選択中ペルソナが利用不可化(OS 再インストール等で鍵復号が失敗)**: 未選択相当として扱い、警告表示と再選択導線を出し、掲載は保留に落ちる。復元手段は提供しない(事前 nsec エクスポートが唯一のバックアップ)。
- **UI の状態が古いまま制限操作を送信(競合)**: 画面表示上はロックされていなかったが送信時点では配信中だった場合、操作は拒否され、ユーザーには理由が示され、画面状態が最新へ更新される。
- **選択中ペルソナが破棄済み**: 既に存在しないペルソナが選択中として残っていた場合は未選択相当に扱う。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: ユーザーは、有効(active)かつ利用可能(鍵復号が成功する)なペルソナを、明示的な操作で「選択中ペルソナ」に設定できMUST。
- **FR-002**: システムは、アーカイブ済み・利用不可のペルソナを選択中に設定する操作を、UI 上でもバックエンド上でも拒否しMUST(UI のみの防御にしてはならない)。
- **FR-003**: システムは、選択中ペルソナが常に高々1つであることを保証しMUST(単一選択)。
- **FR-004**: システムは、最初のペルソナ作成時にそれを自動的に選択中とすMUST。2つ目以降の作成では選択中を自動変更してはならないMUST NOT。
- **FR-005**: システムは、1つ以上のチャンネルをネットワークへ発行中(配信中)である間、選択中ペルソナに対する「別ペルソナへの切り替え」「破棄」「アーカイブ(有効→アーカイブの状態変更)」をすべて拒否しMUST。拒否は UI の無効化とバックエンドの拒否の両方で行うMUST。
- **FR-006**: システムは、配信中であっても選択中ペルソナの表示名(ラベル)変更を許可しMUST(署名アイデンティティに影響しないため)。
- **FR-007**: システムは、配信中の制限を選択中ペルソナのみに適用しMUST。配信に関与していない他のペルソナの作成・アーカイブ・破棄は配信中でも許可しMUST。
- **FR-008**: システムは「配信中」を「1つ以上のチャンネルが実際にネットワークへ発行されている状態」と定義しMUST。まだ発行していない保留チャンネルは配信中に含めてはならないMUST NOT。したがって、保留チャンネルしか無い(または何も無い)状態での最初の選択は常に許可されMUST。
- **FR-009**: システムは、配信が終了し発行中チャンネルが無くなった時点で、選択に関する制限を解除しMUST。異常終了(切断・強制終了)でも制限が取り残されてはならないMUST NOT。
- **FR-010**: システムは、現在の選択中ペルソナを、ペルソナ管理画面とチャンネル一覧画面の双方で人間可読に明示しMUST。チャンネル一覧画面での表示は読み取り専用でMUST。
- **FR-011**: システムは、選択中ペルソナが後からアーカイブまたは利用不可になった場合、それを未選択相当として扱い、選択中表示を警告状態にして再選択を促しMUST。この間の掲載は保留に落ちるMUST。
- **FR-012**: システムは、選択中ペルソナの切り替えに際して確認ダイアログを要求してはならないMUST NOT(常時明示表示により誤爆を防ぐ方針のため)。既存の破棄・秘密鍵エクスポートの確認は維持しMUST。
- **FR-013**: システムは、ペルソナが0個のとき、選択対象が無い旨と作成への導線を示しMUST。
- **FR-014**: 本機能は、チャンネル単位のペルソナ割り当て(特定チャンネルへの個別割り当て)を提供しないMUST NOT。選択の単位はグローバルな選択中ペルソナのみとするMUST。
- **FR-015**: 本機能は、掲載パイプラインの発行契機(PCP 到着での自動発行、周期再発行、終了時の最終発行)の挙動を変更してはならないMUST NOT。

### Key Entities *(include if feature involves data)*

- **ペルソナ (Persona)**: ユーザーが名乗る公開鍵アイデンティティ。状態として active / archived / 利用不可(鍵復号失敗)を持つ。表示名(ラベル)と識別子(公開鍵)を持つ。
- **選択中ペルソナ (Selected Persona)**: 次に掲載するチャンネルの署名に用いられるグローバルな既定ペルソナ。高々1つ。有効かつ利用可能なペルソナのみが設定対象。
- **配信中状態 (Broadcasting State)**: 自ノードが1つ以上のチャンネルを実際にネットワークへ発行しているか否かの真偽。選択関連操作のロック可否を決める。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 何も配信していないユーザーは、有効なペルソナを1回の操作(1クリック)で選択中に切り替えられる。切り替え要求のバックエンド処理(ロック取得+SQLite 1 行更新)はミリ秒未満で完了し(plan Performance Goals)、UI は次のポーリング周期(既存 5 秒)以内、選択操作直後の応答では即時に全画面の選択中表示へ反映される。
- **SC-002**: 配信中のユーザーは、選択中ペルソナの切り替え・破棄・アーカイブを一切成立させられない(UI 経由・直接 API 経由のいずれでも成功率 0%)。
- **SC-003**: 配信を停止して全チャンネルが終了した後、ユーザーは追加の手動リセットなしに再びペルソナ選択操作を行える。
- **SC-004**: ユーザーは、掲載中のチャンネルを見る画面上で、現在どのペルソナで名乗っているかを常に判別できる(選択中ペルソナが未表示になる状態が発生しない。ただし未選択・警告状態は明示される)。
- **SC-005**: 配信中に名乗るペルソナが別ペルソナへ入れ替わる事象が発生しない(=同一チャンネル上で旧ペルソナ終了→新ペルソナ開始というリンク推定シグナルを生じない)。

## Assumptions

- 選択の単位はグローバルな「選択中ペルソナ」のみであり、チャンネル単位のペルソナ割り当ては提供しない。これは上流(PeerCastStation / peercast-yt)がチャンネル識別子を明示的に選ばせる機能を持たず、識別子が固定でもないため、チャンネル単位割り当てが機能として成立しないことによる。
- 掲載前に到着チャンネルを保留して手動確認させる「発行ゲート」は設けない。掲載のたびに確認を挟むと配信 UX を損なうため、事前選択+常時明示表示で誤爆を防ぐ方針を採る。
- 「配信中」の判定は、掲載エンジンが保持する「現在発行中のチャンネル」状態を真実源とする。上流の配信状態変化(開始・停止・異常切断)はこの状態に確実に反映される既存の仕組みに依拠する。
- 選択中ペルソナが利用不可になった場合の復元手段は提供しない。事前の秘密鍵(nsec)エクスポートが唯一のバックアップであるという既存方針を踏襲する。
- チャンネル一覧画面での選択中ペルソナ表示は、既存のペルソナ一覧取得手段から導出でき、新たなデータ項目の追加を必須としない。
- 本機能は既存の複数ペルソナ運用に伴うリンク推定リスク(同一ノード・同一トラッカーからの複数ペルソナ掲載)に関する既存の注意方針(ADR-0004)と整合する。
