# Contract: PEX 破棄分類とセキュリティイベント記録規則

本契約は feature 001 の `contracts/p2p-gossip.md` 検査5 を、破棄理由の分類と `pex_rejected`
記録条件の観点で精緻化するものである。検査5 の**破棄判定そのもの(何を破棄するか)は不変**。

## C1. 破棄理由の分類

受信 `PEERS` の各破棄は、次のいずれかに一意に分類される。

| 破棄理由 | 分類 | 記録への影響 |
|---------|------|-------------|
| 件数 > 64(バッチ全体破棄) | 不審(suspicious) | `pex_rejected` 記録 |
| `parse_addr` 失敗(形式不正・長さ > 256・ブラケットなし複数コロン) | 不審 | `pex_rejected` 記録 |
| ホスト名形式(名前空間分離違反 — ADR-0010) | 不審 | `pex_rejected` 記録 |
| 自己アドレス一致 | 良性(benign) | 記録しない(debug のみ) |
| バッチ内 canonical 重複(初出以降) | 良性 | 記録しない(debug のみ) |

## C2. セキュリティイベント記録規則(MUST)

- 受信 `PEERS` の破棄に**不審が 1 件以上**含まれる場合、`pex_rejected` セキュリティイベントを
  記録しなければならない (MUST)。
- 破棄が**良性のみ**(自己アドレス・重複のみ)である場合、`pex_rejected` を記録してはならない
  (MUST NOT)。代わりに debug ログを出力してよい (MAY)。
- 破棄が無い場合、いずれのログも出さない。

## C3. 防御の不変性(MUST NOT)

- 良性/不審いずれの破棄も、**候補として登録しない**破棄挙動は変更前と完全に同一でなければならない
  (MUST NOT change)。本契約はログ分類のみを規定する。

## C4. 情報漏洩の禁止(MUST NOT)

- `pex_rejected` の `source`・`detail` は従来どおり内部情報を漏洩してはならない (MUST NOT)。
- debug ログに自ノードアドレスを含める場合、自ノード自身のアドレスに限る (MUST)。

## テスト観点(ネガティブ必須 — Principle IV)

契約テスト(`tests/contract/pex.rs`)および Gherkin(`tests/features/security.feature`)で以下を固定する:

1. **良性のみ(自己アドレス)**: 自己アドレスのみを含む `PEERS` → 破棄されるが `pex_rejected` 無記録
2. **良性のみ(重複)**: 同一 canonical を重複させた(不審なし)`PEERS` → 破棄されるが無記録
3. **不審(件数超過)**: 65 件の `PEERS` → 全破棄 + `pex_rejected` 記録
4. **不審(形式不正/ホスト名)**: 不正エントリを含む `PEERS` → `pex_rejected` 記録
5. **混在**: 自己アドレス(良性)+ 形式不正(不審)→ `pex_rejected` 記録
6. **回帰**: いずれのケースでも `accepted`(候補登録)集合が変更前と同一
