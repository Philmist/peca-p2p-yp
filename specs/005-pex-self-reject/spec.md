# Feature Specification: PEX 自己アドレス拒否の良性化

**Feature Branch**: `005-pex-self-reject`

**Created**: 2026-07-08

**Status**: Draft

**Input**: User description: "自己アドレスのPEX拒否を良性として扱い、デバッグログに格下げしたいです。"

## 背景 *(context)*

受信 `PEERS`(ピア交換)検証(contracts/p2p-gossip.md 検査5)は、自アドレス・重複・件数超過・
形式不正・長さ超過・ホスト名を破棄し、破棄が 1 件でもあると `pex_rejected` セキュリティイベントを
記録する。健全な網では「自ノードを知っている全ピア」が PEX で自ノードのアドレスを送り返すため、
**自己アドレス反射は接続のたびに常時発生する正常な現象**である。にもかかわらず現状はこれを
`pex_rejected`(WARN のセキュリティイベント)として記録するため、セキュリティログが良性イベントで
埋もれ、真に不審な内容(件数超過・形式不正・ホスト名等)の検知信号としての価値が損なわれている。

本機能は、**拒否理由が良性(自己アドレス・重複)のみである場合はセキュリティイベントを記録せず、
デバッグレベルの観測ログへ格下げする**。破棄そのもの(候補に登録しない防御)は維持し、真に不審な
拒否理由が含まれる場合は従来どおりセキュリティイベントを記録する。

「良性」に自己アドレスと重複を含めるのは、両者とも健全な網で日常的に発生する現象だからである。
とくに dual-stack 環境では同一ノードが複数表記(IPv4 / IPv6)で候補化され、バッチ内重複が
自然に生じうる。件数超過・形式不正・長さ超過・ホスト名は引き続き不審として扱う。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 良性な自己反射・重複でセキュリティログが汚染されない (Priority: P1)

運用者として、健全な網の通常運用中に、自ノードのアドレスが PEX で送り返されたり同一ノードが
重複表記で送られてくるたびに `pex_rejected` セキュリティイベントが記録されるのを止めたい。
セキュリティログを本当に注意すべき事象(不正・攻撃の兆候)だけに保ちたい。

**Why this priority**: セキュリティログの信号対雑音比を回復するのが本機能の主目的。偽陽性が
常時鳴ることは、Principle I/II のセキュリティ監視を実質的に無力化する。

**Independent Test**: 3 ノードの健全な網で相互接続させ、自己反射・重複のみが起きる状況で
`pex_rejected` セキュリティイベントが 0 件であることを確認する。

**Acceptance Scenarios**:

1. **Given** 自ノードが自己アドレスを学習済みで、ピアが PEX で自ノードのアドレスのみを送り返す、**When** その `PEERS` を検証する、**Then** 該当アドレスは候補登録されず破棄され、かつ `pex_rejected` セキュリティイベントは記録されない
2. **Given** ピアが同一 canonical を重複させた `PEERS`(かつ不審な理由は含まない)を送る、**When** 検証する、**Then** 重複分は破棄されるが `pex_rejected` セキュリティイベントは記録されない
3. **Given** 上記いずれか、**When** 破棄が発生する、**Then** その事実は debug レベルのログとしてのみ観測できる(source と破棄件数が分かる)

---

### User Story 2 - 真に不審な PEX 内容は引き続き検知される (Priority: P1)

運用者/セキュリティ担当として、件数超過・形式不正・ホスト名・長さ超過といった不正な PEX 内容は
引き続き `pex_rejected` セキュリティイベントとして記録され、見逃されないようにしたい。

**Why this priority**: 良性化のスコープを誤って広げ、実際の攻撃兆候まで隠すことは Principle I/II
違反となる。US1 と対で成立して初めて安全な変更となる。

**Independent Test**: 件数 65 件の `PEERS` や形式不正エントリを含む `PEERS` を投入し、`pex_rejected`
セキュリティイベントが記録されることを確認する。

**Acceptance Scenarios**:

1. **Given** ピアが 64 件を超える `PEERS` を送る、**When** 検証する、**Then** バッチ全体が破棄され `pex_rejected` セキュリティイベントが記録される
2. **Given** ピアが形式不正・ホスト名・長さ超過のエントリを含む `PEERS` を送る、**When** 検証する、**Then** `pex_rejected` セキュリティイベントが記録される
3. **Given** ピアが自己アドレス・重複(良性)と形式不正エントリ(不審)を混在させた `PEERS` を送る、**When** 検証する、**Then** 不審な拒否が 1 件でもあるため `pex_rejected` セキュリティイベントが記録される

---

### User Story 3 - 契約・ドキュメントとの整合 (Priority: P2)

保守者として、変更後の挙動が contracts/p2p-gossip.md 検査5 および data-model の SecurityEvent
記述と一致し、コードとドキュメントの乖離が残らないようにしたい。

**Why this priority**: Principle VI(追跡可能性)。乖離は将来の実装・レビューの誤りを招く。

**Independent Test**: 契約・data-model の該当記述を読み、実装の挙動(良性=無記録、不審=記録)と
一致していることを確認する。

**Acceptance Scenarios**:

1. **Given** 変更を適用した、**When** contracts/p2p-gossip.md 検査5 と data-model の `pex_rejected` 記述を参照する、**Then** 「自己アドレス・重複のみの破棄はセキュリティイベント対象外」であることが明記されている
2. **Given** 変更を適用した、**When** 設計判断を参照する、**Then** ADR に良性化の根拠と維持すべき線引きが記録されている

---

### Edge Cases

- **良性と不審の混在**: 1 バッチに良性(自己アドレス・重複)と形式不正等(不審)が混在する場合は、
  不審な拒否が 1 件でも存在するためセキュリティイベントを記録する(US2 シナリオ3)。
- **自己アドレス未学習**: 起動直後で自ノードが自身のアドレスを未学習の場合、自己反射は破棄されず
  候補登録される(既存挙動)。本機能の対象外(良性化はあくまで「自己アドレス・重複として破棄された」
  ケースに限る)。
- **重複の判定**: 重複は 1 バッチ内で同一 canonical が 2 回以上現れた場合の 2 件目以降を指す
  (検査5 の既存定義)。異なる表記でも canonical 化後に一致すれば重複として良性扱いになる。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムは、受信 `PEERS` の破棄理由が**良性(自己アドレス・重複)のみ**である場合、`pex_rejected` セキュリティイベントを記録してはならない (MUST NOT)。
- **FR-002**: 良性(自己アドレス・重複)による破棄は、debug レベルのログとして観測可能でなければならない (MUST)。ログには少なくとも接続元(source)と破棄件数を含める。
- **FR-003**: 受信 `PEERS` に良性以外の不審な破棄理由(件数超過・形式不正・長さ超過・ホスト名)が **1 件でも**含まれる場合、システムは従来どおり `pex_rejected` セキュリティイベントを記録しなければならない (MUST)。
- **FR-004**: 良性化は**ログ分類のみ**の変更であり、良性エントリ(自己アドレス・重複)を候補登録しない破棄挙動(防御)は一切変更してはならない (MUST NOT)。
- **FR-005**: 変更後の挙動を contracts/p2p-gossip.md 検査5 および data-model の SecurityEvent(`pex_rejected`)記述へ反映し、コードとドキュメントの乖離を残してはならない (MUST)。設計判断は ADR に記録しなければならない (MUST)。
- **FR-006**: 良性ケース(自己アドレスのみ→無記録・重複のみ→無記録)と不審ケース(不正エントリ含む→記録)、および混在ケースを Gherkin ネガティブシナリオとして記述し、自動テストで検証しなければならない (MUST)（Principle IV）。
- **FR-007**: セキュリティイベントに出力するアドレス・詳細は、従来どおり内部情報を漏洩してはならない (MUST)（Principle II）。debug ログに含める自ノードアドレスは自ノード自身のものに限る。

### Key Entities *(include if feature involves data)*

- **SecurityEvent (category=`pex_rejected`)**: PEX 内容違反の記録。本機能により「良性(自己アドレス・重複)のみ」の破棄は記録対象から外れ、「不審な破棄理由を含む」場合にのみ記録される。
- **受信 PEERS 検証結果**: 採用候補と破棄エントリに加え、**破棄理由の分類(良性=自己アドレス・重複 / 不審=それ以外)**を区別できる情報を持つ。この分類がセキュリティイベント記録要否の判断根拠となる。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 健全な網(自己反射・重複のみが発生する接続)において、`pex_rejected` セキュリティイベントの記録件数が 0 件になる。
- **SC-002**: 不審な内容(件数超過・形式不正・ホスト名・長さ超過)を含む `PEERS` を受けた場合、`pex_rejected` セキュリティイベントが 100% 記録される(良性混在時も含む)。
- **SC-003**: 良性(自己アドレス・重複)による破棄の発生を、debug ログ有効時に接続元と件数の粒度で追跡できる。
- **SC-004**: 破棄そのものの挙動(良性エントリを候補登録しない)は変更前後で同一である(回帰なし)。

## Assumptions

- 「自己アドレス」の判定は既存の自己アドレス集合(nonce 駆動の自己接続検出で学習)を用いる。判定ロジック自体は変更しない。
- ログの既定レベルは INFO であり、debug ログは `RUST_LOG` 等で明示的に有効化した場合にのみ出力される(既存のログ初期化方針を踏襲)。
- 本機能はログ分類とドキュメント整合のみを対象とし、PEX の選定規則・候補登録・接続管理は変更しない。
- 良性化の対象は自己アドレスと重複の 2 種とする(Clarification 2026-07-08 で確定)。件数超過・形式不正・長さ超過・ホスト名は引き続き不審として `pex_rejected` を記録する。
