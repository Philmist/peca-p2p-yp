# Research: 配信実況スレ(P2P 掲示板)

**Feature**: `006-livechat-thread` | **Date**: 2026-07-12 | **Input**: [spec.md](./spec.md)

spec の Assumptions で plan へ先送りした 3 点(NIP-53 援用可否・announce 容量検証・
互換 API 詳細の基礎)に加え、Technical Context の確定に必要な技術選定を記録する。
形式は 001 research.md に合わせる(Decision / Rationale / Alternatives)。

## R1. レスのイベントスキーマ — NIP-53 kind 1311 の援用

- **Decision**: レスは **kind 1311(NIP-53 Live Chat Message)のイベント封筒を援用**する。
  001 contracts/nostr-events.md が「将来フェーズ・予約」として固定した識別子設計
  (`a` タグで `30311:<author_pubkey>:<channel_guid>` を参照、投稿鍵は掲載ペルソナと独立)を
  そのまま履行する。名前欄・メール欄・スレ参照などの本機能固有の属性は `["peca",...]`
  拡張タグで表現する(30311 の PeerCast 固有拡張タグと同じ流儀)。
- **Rationale**: (1) 001 FR-011 で識別子設計が既に契約化されており、履行が最小差分。
  (2) nostr 援用境界(001 FR-014)= イベントのデータ構造・id 計算・Schnorr 署名のみ、
  伝送は独自プロトコル — という原則をそのまま維持できる。(3) 既存の署名検証・サイズ検証・
  形式検証コード(`event/`)を再利用でき、自前の署名形式を発明しない(Principle II)。
- **Alternatives considered**: 独自 JSON 構造 + 自前署名封筒 — 検証コードの二重化と
  援用境界の曖昧化を招くため却下。NIP-53 の周辺機能(zap・リアクション等)の同時採用 —
  本機能に不要(YAGNI)。

## R2. スレ announce・順序確定情報のイベント種別

- **Decision**: いずれも nostr イベント封筒(id/sig)を援用した **peca 固有 kind** とする:
  - **スレ announce = kind 31311**(addressable 範囲)。置換キー `(31311, pubkey, d)`、
    `d` は固定文字列(板 = 配信者ペルソナ単位でアクティブスレ 1 本 — FR-012 — のため、
    ペルソナあたり announce は常に 1 件に置換される)。`expiration` による鮮度管理は
    30311 と同一規則。
  - **順序確定情報 = kind 21311**(ephemeral 範囲)。gossip には流さず、スレ配送
    セッション内でのみ伝送する。
- **Rationale**: NIP-53 にはスレ announce・順序確定に相当する標準 kind がない。
  イベント封筒を共有することで検証パイプラインを一本化できる。31311/21311 は執筆時点で
  NIP 未割当の番号であり、1311 との対応が読み取れる番号を選んだ。addressable の置換規則が
  「板ごとに最新 announce のみ保持」の要件と一致し、ephemeral の「保存しない」意味論が
  揮発性(FR-015)と一致する。
- **Alternatives considered**: 順序確定情報を素の JSON メッセージ(イベント封筒なし)で
  流す — スレ主署名の検証コードを別途書くことになり却下。announce に kind 30311 の
  タグ拡張で相乗り — チャンネル掲載と実況スレのライフサイクル(鮮度・置換)が絡み合い、
  検証の単純さを失うため却下。
- **確認記録(CHK017)**: nostr-protocol/nips リポジトリの kind レジストリに 21311 / 31311
  の割当がないことを確認(2026-07-12。モデル知識ベースの確認であり、一次資料の再突合は
  未了)。**実装フェーズ開始時に同レジストリを再確認する**こと。割当が生じていた場合の
  改番判断基準は contracts/thread-events.md §NIP 適合に規定した。
- **Risk / follow-up**: 将来 nostr 側が 31311/21311 を別用途に割り当てる可能性はあるが、
  本ソフトウェアはリレーと通信しない(FR-014)ため実害は命名衝突のみ。contracts に
  「peca 固有 kind であり NIP 互換を主張しない」と明記する。また peca イベントは有効な
  nostr イベントであるため第三者が公開リレーへ再公開しうるが、31311 の `tip`(ip:port)は
  30311 の `["peca","tip"]` と同じく元々公開の掲載情報であり追加露出はない(受容 —
  contracts/thread-events.md §NIP 適合に記録)。

## R3. スレ announce のイベント率と容量(001 R16 との整合検証)

- **Decision**: スレ announce は 30311 と同じ再発行 60 秒・鮮度 600 秒に従わせる。
  この前提で **既存容量設計(001 R16)の余裕内に収まる**と判定する。
  - 最悪ケース(全 2,000 live チャンネルが実況スレ併設): announce 追加率 ≈ 2,000/60
    ≈ +33 イベント/秒 → 網全体 ≈ 66 イベント/秒。リンクあたりレート上限 200 msg/s の
    33% で、帯域は典型 ~1KB/announce として +33KB/s(最悪)→ 合算 ~85KB/s < 256KB/s。
  - EventStore: announce は板ごと 1 件に置換されるため保持数は ≤ live 板数(≤ 2,000)。
    ただし既存上限 4,096 は 30311 用の設計値(2,000 live + 余裕)であり、31311 と共用すると
    余裕が消える。**kind 31311 に独立の保持枠(既定 2,048、Settings で調整可)を設ける**
    (data-model.md §Settings)。
  - レス本文・順序確定情報は gossip に載らない(ホスト直結配送のみ)ため、
    発見網への追加負荷は announce のみ。
- **Rationale**: spec Assumptions の検証課題への回答。フラッディングの負荷は kind に
  よらずイベント率で決まるため、R16 の計測(100 ノード実測 p99 2.34 秒、5,000 ノード
  外挿 p99 ≈ 4.3 秒)に対し 2 倍のイベント率でも SC-006(掲載 60 秒以内)の予算内。
- **Alternatives considered**: announce を低頻度(300 秒毎)にする — 鮮度管理の規則が
  30311 と分岐し実装・検証が複雑化する割に、容量上の必要がない。却下。

## R4. スレ配送セッションの受け口(gossip との多重化)

- **Decision**: スレ配送は **既存 P2P 待受(既定 7147)と同一ポート・同一フレーミング**
  (4 バイト BE 長さ前置 + JSON)を共有する。HELLO の `features` に `livechat1` を掲げ、
  established 後の最初のメッセージ(`THREAD_JOIN`)でセッション種別を確定する。
  **1 TCP 接続 = 1 用途**(gossip セッションとスレセッションは別接続)とする。
- **Rationale**: (1) スレ主 = 配信者は 7147 を既に開放済みであり、追加のポート開放を
  要求しない(spec Assumptions「配信者は着信可能」に閉じる)。(2) フレーミング・
  フレーム長上限・レート制限・切断コードの実装を再利用できる。(3) `features` の
  「未知の値は無視 (MUST)」規則が 001 で確立済みで、旧バージョンノードとの前方互換が
  自然に成立する(livechat1 を知らないノードは THREAD_JOIN を不正フレームとして扱う
  だけで、gossip には影響しない)。
- **Alternatives considered**: 専用ポート — 配信者に追加の開放・NAT 設定を求め UX を
  損なうため却下。gossip セッション内での論理多重化 — セッション状態機械が複雑化し、
  スレ配送のフロー制御が gossip の伝搬と干渉するため却下(Principle V の検証対象も
  不必要に拡大する)。

## R5. 互換 API の方言・文字コード・受け口

- **Decision**:
  - 方言は伝統的 2ch 形式(monazilla 系専ブラの前提): `GET /{board}/subject.txt`、
    `GET /{board}/dat/{key}.dat`、`GET /{board}/SETTING.TXT`、`GET /{board}/head.txt`
    (ローカルルール)、`POST /test/bbs.cgi`(書き込み)。**模倣対象の参照実装は
    bbs.jpnkn.com(BBSJPNKN)・EXぜろちゃんねる等の 0ch 系互換実装**(板設定 4 項目の
    援用元 — spec Clarifications)とし、詳細は contracts/compat-api.md に固定する。
  - 文字コードは **Shift_JIS(CP932、`encoding_rs`)固定**。変換不能文字は
    **数値文字参照(`&#dddd;`)で保全**する(現代互換実装・専ブラの通例。参照は
    `<>` を含まないため区切りを壊さない)。bbs.cgi 受理時は数値文字参照を展開して
    本文へ格納する。※ 001 R5 の index.txt(`?` 置換)とは方針が異なる — dat は
    実体参照・数値文字参照の解釈が専ブラ側に実装済みであるため保全を優先する。
  - 現代専ブラの差分取得・更新チェックに対応する: dat の **追記不変性**(一度返した
    バイト列は以後変化しない)を契約の不変条件とし、`Range`(206)・
    `If-Modified-Since`(304)・charset 付き Content-Type を提供する。gzip 等の
    内容符号化は行わない(contracts/compat-api.md §HTTP メタデータ)。
  - 受け口は **専用の loopback 専用リスナー**(Settings `compat_bbs_bind`、既定
    `127.0.0.1:7183`、空文字で無効化)。Host 検証・レート制限・ボディ上限・定型エラーは
    `web/` の保護層を再利用する。トークン認証は課さない(既存クライアントが送れない —
    FR-026 の保護方針)。
- **Rationale**: 専ブラ・コメントビューワの dat 読み書きは SJIS 前提が支配的。既存
  ローカル API(7180)は変更系に `X-Api-Token` を必須とする保護設計であり、互換 API の
  「トークンなしの書き込み」を同居させると保護層に例外を掘ることになる。リスナー分離が
  最も単純で、ADR-0012(index.txt の第 2 受け口)と同じ構成パターンで監査もしやすい。
  バインド値は loopback のみ受理(非 loopback は起動拒否 — ADR-0006 決定 4 の原則)。
- **Alternatives considered**: 7180 への同居 — 変更系トークン必須の例外化が必要となり
  却下。UTF-8 既定 — index.txt では実機検証で UTF-8 が正解だったが、dat/bbs.cgi の
  専ブラ実装は SJIS デコードが標準実装であり事情が異なる。保守的に SJIS 固定とする。
- **Risk / follow-up**: 001 R5 と同型の「実機確認までは仮説」。SC-007(利用者所有の
  実況ツール一式での実機確認)を受け入れテストに含め、結果次第でエンコーディング設定の
  追加を検討する。

## R6. 新規板鍵の初回書き込み PoW

- **Decision**: **NIP-13(`nonce` タグ)を再利用**する。ホストは受信レスの板鍵が
  当該板で未知(初見またはローテーション直後)の場合に板設定 `first_post_pow_bits`
  (既定 20、範囲 0〜32。**これを唯一の正式名とし、ノード Settings には置かない** —
  板設定として配布・永続する)を要求し、既知の板鍵には通常しきい値(既定 0)を適用する。
  要求難易度は板設定と共に参加者へ配布し、クライアントは送信前にローカルで nonce を
  計算する。強制はホスト側のみで行う(書き込みは常にホスト直送 — FR-007)。
- **Rationale**: 既存の NIP-13 検証実装(001 検査 #6)を再利用し自前の仕組みを発明しない
  (Principle II)。書き込み経路がホスト集約であるため、網全体での PoW 検証は不要で
  ホスト単独の判定で完結する。既定 20 bits は平均 2^20 ハッシュ ≈ 一般的 CPU で 1 秒弱 —
  正規利用者のローテーションには許容可能、鍵を使い捨てる荒らしには累積コストとなる。
- **Alternatives considered**: 独自のパズル形式 — 却下(自前暗号の禁止に抵触しうる)。
  全レスに高 PoW — 通常の実況テンポ(バースト 30 レス/分)を阻害するため却下。

## R7. ローカルルール Markdown の描画(安全なサブセット)

- **Decision**: **pulldown-cmark** で解析し、raw HTML(インライン・ブロックとも)を
  破棄した安全なサブセット(段落・見出し・強調・リスト・引用・コード・リンク)のみを
  自前 Web UI で描画する。リンクは scheme が http/https 以外なら無効化(001 FR-012 の
  URL 安全性規則と同じ)。互換 API(`head.txt`)へは Markdown を平文のまま出力する
  (描画しない — 従来クライアント側の解釈に委ねない)。
- **Rationale**: 純 Rust・依存が小さい・イベントストリーム型 API のため「HTML を
  最初から生成しない」フィルタリングが確実に書ける(FR-025)。サニタイズ後追い
  (HTML 生成 → ammonia 等で除去)より攻撃面が小さい。
- **Alternatives considered**: comrak — GFM 全部入りで依存が重く不要。生 HTML 許可 +
  サニタイザ — 許可リスト管理の失敗がそのまま XSS になるため却下。

## R8. 板鍵の保管と非リンク性

- **Decision**: 既存の keystore 抽象(ADR-0003 DPAPI / ADR-0009 Linux エンベロープ)で
  秘密鍵を暗号化し、`store/` の新テーブル(board_keys)に保存する。ペルソナテーブルとは
  スキーマ・識別子・外部キーを一切共有しない。板鍵のエクスポート(nsec 表示)機能は
  提供しない(板鍵は端末ローカルの使い捨て身元であり、持ち出し需要より漏洩・リンク
  リスクが上回る)。
- **Rationale**: FR-016(既存ペルソナと別系統・相互リンク禁止)をストレージ層でも
  構造的に分離して満たす。鍵保護の実装は実績ある既存経路の再利用。
- **Alternatives considered**: personas テーブルへの種別カラム追加 — 誤結合・誤表示の
  リスク(ADR-0011 が防いだ類のリンク事故)を作るため却下。

## R9. Principle V 判定(形式的検証)

- **Decision**: **該当** — ホスト = シーケンサの中核状態機械(採番・順序確定情報の配布・
  レス上限到達時の次スレ移行・凍結/クローズ)は 3 基準をすべて満たす:
  (1) 新規設計(Winny2 の先行事例はあるが仕様・実装とも流用しない)、
  (2) 同時書き込み × 上限到達 × 切断/再接続の競合はテストで再現困難、
  (3) 失敗は全端末のレス番号・アンカー不一致 = データ整合性(Principle I)に直結。
  PlusCal モデル `docs/formal/livechat_sequencer.tla` を実装前ゲートとして作成し、
  少なくとも以下を検査する: **採番の一意性・欠番なし単調増加**(安全性)、
  **上限超過採番なし・移行境界での二重採番なし**(安全性)、
  **確定情報の全接続参加者への到達**(ライブネス)。判定理由と検証結果は
  **ADR-0014** に記録する(MUST — Principle V / VI)。
- **非該当と判断した部分**: 互換 API(HTTP リクエスト/レスポンスの単純写像)、板設定の
  配布(単一送信者の置換配布)、announce の伝搬(既存 gossip 状態機械 — ADR-0005 で
  検証済み — に kind を追加するのみ)。理由は ADR-0014 に併記する。
- **Rationale**: ADR-0005(gossip 伝搬の「該当」判定)の前例に従う。シーケンサは
  本機能の存在理由(SC-002: 不一致 0 件)そのものであり、設計段階での競合の根絶が
  テストより安価。

## 解決済み確認

R5 = 互換 API の基礎)。Technical Context に NEEDS CLARIFICATION は残っていない。
