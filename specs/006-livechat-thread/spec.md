# Feature Specification: 配信実況スレ(P2P 掲示板)

**Feature Branch**: `006-livechat-thread`

**Created**: 2026-07-11

**Status**: Draft

**Input**: User description: "配信実況スレ機能(P2P 掲示板)。PeerCast 配信のリアルタイム実況を、中央サーバーなしのスレッドフロート型掲示板として提供する。3 層構成(発見 = 既存 YP gossip / 配送 = ホスト直結星型・ホスト = シーケンサ / 表現 = 端末内で掲示板再構成)。スレ主 = 配信者限定。板 = 配信者ペルソナ単位・アクティブスレ 1 本・レス上限で次スレ。参加者鍵は板単位固定・既存ペルソナと別系統・ローテーション可・新規鍵初回高 PoW。NG/BAN は完全鍵照合。announce 起点の自動接続禁止・チャレンジ認証・レート制限。"

## 背景 *(context)*

PeerCast コミュニティの実況文化は現在も中央サーバー型の掲示板(したらば等)に依存しており、
本プロジェクトが YP(チャンネル発見)で排除した中央依存が実況側に残っている。本機能は配信の
リアルタイム実況を、利用者ノードのみで成立するスレッドフロート型掲示板として提供する。

設計は 3 層に分担する:

1. **発見**: スレの存在(スレ announce)は既存のチャンネル発見網(gossip)に相乗りして伝搬する
2. **配送**: スレ本文はホスト(= スレ主 = 配信者)直結の星型で配送する。ホストは
   **シーケンサ**として全書き込みを採番し、署名付きの順序確定情報を配布する
3. **表現**: 各端末が検証済み本文と確定順序から掲示板(レス番号・アンカー)を再構成する。
   既存の 2ch 互換クライアント(専ブラ・コメントビューワ)向けに、各利用者の自ノードが
   loopback に互換 API を提供する(index.txt の YP ブラウザ互換と同じ思想)

先行事例は Winny2 の BBS 機能(マスターモデル)である。調査
(`docs/research/winny2-bbs.md`)により、(a) スレの正本を単一ノードが持ち書き込みを集約する
方式は Winny2 で実証済み、(b) Winny2 の致命的欠陥はマスター(スレ主)の匿名性が守れない
ことだった、と確認した。本機能は**スレ主を配信者(身元が元々公開の主体)に限定する**ことで
この欠陥を構造的に回避する。匿名のまま任意の利用者がスレ主になれる汎用分散 BBS は、
匿名化中継を含む別の設計課題であり本機能のスコープ外とする。

規模想定は現在の実況文化の実勢(バースト 30 レス/分・1 スレ参加者 100 人弱)に基づき、
星型配送で成立する。木構造化(中継ノード)や gossip 的修復は将来フェーズとし、
プロトコルには拡張余地のみ確保する。

### 用語

| 用語 | 意味 |
|------|------|
| **板** | 配信者ペルソナ単位の掲示板。板の同一性は配信者ペルソナに紐づく(チャンネル GUID や配信セッションには紐づかない) |
| **スレ** | 板の中の一連のレス列。アクティブなスレは板ごとに常に 1 本。レス上限到達または配信者操作で次スレへ移行(スレッドフロート) |
| **レス** | スレへの 1 書き込み。レス番号はホストの採番で一意に確定する |
| **スレ主(ホスト)** | スレを開設・管理する配信者ノード。書き込みの集約・採番・配布を担うシーケンサ |
| **順序確定情報** | ホストが署名して配布する採番結果(チェックポイント)。全端末はこれに基づいてのみレスを表示する |
| **スレ announce** | スレの存在・接続先を知らせる掲載情報。対象チャンネルと同一ペルソナの署名を要する |
| **板鍵** | 書き込みに用いる参加者の鍵。板単位に固定され、既存ペルソナとは別系統。レスの ID 表示に相当 |
| **NG** | 視聴者側のローカル非表示(板鍵単位) |
| **BAN** | スレ主側の採番拒否・接続拒否(板鍵単位・接続単位)。ローカルのみでネットワークへ出ない |
| **板設定** | 板主(= スレ主。板の所有者としての呼称)が設定する運用項目: 板タイトル・スレ最大レス数・名無しのデフォルト名・ローカルルール(最大 2048 文字の Markdown) |
| **互換 API** | 各利用者ノードが loopback に提供する伝統的 2ch 形式の読み書き口(スレ一覧・スレ本文・板設定・書き込み)。既存の専ブラ・コメントビューワ向け |

## Clarifications

### Session 2026-07-12

- Q: 板主(= 配信者)が設定できる項目は? → A: BBSJPNKN の項目を援用し、板タイトル / スレッドの最大レス数 / 名無しのデフォルト名 / ローカルルール(最大 2048 文字の Markdown テキスト)の 4 項目とする
- Q: スレ最大レス数の設定可能範囲は? → A: 100〜4000(既定 1000)
- Q: レスの名前欄は? → A: 名前欄あり(空欄は板の名無しのデフォルト名)。トリップは実装しない。名前欄の `#` 以降は送信前に除去する(トリップ入力の癖によるパスワード類の漏洩事故防止)
- Q: 板設定の変更はいつ反映されるか? → A: 即時反映。ただしスレ最大レス数のみ進行中スレに遡及せず、次スレから適用
- Q: 既存の 2ch/5ch 互換クライアントからの読み書きは必要か? → A: 必要。各利用者の自ノードが loopback に互換 API を提供し、P2P スレへブリッジする(公開サーバーは立てない)
- Q: 互換 API の範囲は? → A: 読み書き両対応(スレ一覧・レス取得・書き込み)
- Q: 互換 API の方言は? → A: 伝統的 2ch 形式(スレ一覧 subject.txt / スレ本文 dat / 板設定 SETTING.TXT / 書き込み bbs.cgi 相当、Shift_JIS)。5ch 新 API(認証・JSON)は不採用(YAGNI)
- Q: レスのメール欄の扱いは? → A: レスの属性として保持・伝搬し、表示互換のみ(age/sage 等の機能的意味は持たせない)
- Q: 互換 API の受け入れ確認基準は? → A: 利用者所有の実況ツール一式での実機読み書き確認(具体名は受け入れテスト時に確定 — 001 R5 の YP ブラウザ実機検証と同型)

## User Scenarios & Testing *(mandatory)*

### User Story 1 - スレの開設・発見・閲覧 (Priority: P1)

配信者として、自分の配信の実況スレを開設し、その存在を視聴者に発見してもらいたい。
視聴者として、チャンネル一覧から実況スレの存在を知り、開いて読みたい。読むだけなら
何の登録も鍵も要らないでほしい。

**Why this priority**: 発見と閲覧が成立しなければ掲示板として存在しない。読み専(ROM)が
多数派という掲示板文化の前提を最初から満たす必要がある。

**Independent Test**: 配信者ノード 1 台 + 視聴者ノード 2 台で、スレ開設 → announce 伝搬 →
視聴者がスレを開く → 既存レスが読める、までを単体で検証できる。

**Acceptance Scenarios**:

1. **Given** 配信者が自分のチャンネルを掲載中、**When** 実況スレを開設する、**Then** スレ announce が発見網に伝搬し、他ノードのチャンネル情報にスレの存在が表示される
2. **Given** スレ announce を受信済みの視聴者ノード、**When** 利用者がスレを開く操作をする、**Then** ホストへ接続し、既存の全レスが確定順序どおりに表示される
3. **Given** スレ announce を受信済みの視聴者ノード、**When** 利用者が何も操作しない、**Then** ホストへの接続は一切発生しない(FR-004、Principle II)
4. **Given** 視聴者が板鍵を持っていない、**When** スレを開いて閲覧する、**Then** 閲覧に鍵の生成・登録は要求されない
5. **Given** 板主が板タイトル・ローカルルール・名無しのデフォルト名を設定済み、**When** 視聴者がスレを開く、**Then** 板タイトルとローカルルールが表示から参照でき、名無しレスの表示名に板の設定が反映される

---

### User Story 2 - 書き込みと全端末一致の確定表示 (Priority: P1)

実況参加者として、レスを書き込み、自分を含む全参加者の画面に同じレス番号・同じ並びで
表示されてほしい。「>>152」と書けば、誰の画面でも同じレスを指してほしい。

**Why this priority**: レス番号とアンカーの全端末一致は掲示板体験の核心であり、
本設計(ホスト = シーケンサ)の存在理由そのもの。

**Independent Test**: 3 ノード(ホスト + 参加者 2)で同時に書き込み、全端末のレス番号・
並び・アンカー解決が一致することを検証できる。

**Acceptance Scenarios**:

1. **Given** スレに接続済みの参加者、**When** レスを書き込む、**Then** 書き込みは自端末に「送信中」として即時表示され、ホストの採番確定後に正式なレス番号付きで全端末に表示される
2. **Given** 複数の参加者がほぼ同時に書き込む、**When** ホストが採番する、**Then** 全端末で同一のレス番号・同一の並び順になる(不一致 0)
3. **Given** レス「>>152」を含む書き込み、**When** 各端末で表示される、**Then** アンカーは全端末で同一のレス(152 番として確定したレス)を指す
4. **Given** 順序確定前のレス本文だけが届いた端末、**When** 表示処理を行う、**Then** そのレスは表示されない(確定表示のみ — 送信中の自分の投稿を除く)
5. **Given** 名前欄を空のまま書き込んだ参加者、**When** レスが確定表示される、**Then** 板の名無しのデフォルト名で全端末に表示される。名前欄に `#` を含めて書き込んだ場合、`#` 以降は送信前に除去されている

---

### User Story 3 - なりすまし・不正情報への耐性 (Priority: P1)

利用者として、偽のスレ announce(他人のチャンネルを騙る・存在しないアドレスや第三者の
アドレスを指す)や、スレ主以外が作った偽の順序確定情報に晒されず、それらが検知・記録
されてほしい。

**Why this priority**: announce は全ノードに伝搬するため、悪用されると網全体が反射攻撃の
母体になる(Principle I / II)。安全性が成立しない機能はリリースできない。

**Independent Test**: 署名不一致の announce・偽の順序確定情報・過大サイズのレスを注入する
モックピアで、すべて不可視 + セキュリティイベント記録となることを検証できる。

**Acceptance Scenarios**:

1. **Given** 対象チャンネルの掲載ペルソナと異なる鍵で署名されたスレ announce、**When** 検証する、**Then** 不可視(保持も再伝搬もしない)とし、セキュリティイベントを記録する
2. **Given** 攻撃者が第三者のアドレスをホストとして記載した announce を伝搬させた、**When** 利用者がスレを開く、**Then** 接続時のチャレンジ検証(スレ主ペルソナの鍵の保持確認)に失敗し、切断・バックオフし、セキュリティイベントを記録する
3. **Given** スレ主以外の鍵で署名された順序確定情報、**When** 参加者が受信する、**Then** 破棄され表示に影響せず、セキュリティイベントを記録する
4. **Given** サイズ上限を超えるレス・レート上限を超える書き込み、**When** ホストが受信する、**Then** 採番せず破棄し、セキュリティイベントを記録する(エラーは内部情報を漏洩しない)

---

### User Story 4 - モデレーションと NG (Priority: P2)

スレ主(配信者)として、荒らしの書き込みを以後採番しない(BAN)ようにしたい。
視聴者として、見たくない書き込み主をローカルで非表示(NG)にしたい。荒らしが鍵を
変えて戻ってきても、無コストではないようにしたい。

**Why this priority**: モデレーション不在の実況スレは荒らしに対して無力だが、
閲覧・書き込み(US1/US2)が成立していれば機能として最低限は成立する。

**Independent Test**: ホスト + 参加者 2 + 荒らし役 1 で、BAN 後の採番拒否・NG 後の
ローカル非表示・鍵ローテーション時の初回 PoW 要求を検証できる。

**Acceptance Scenarios**:

1. **Given** スレ主が特定の板鍵を BAN 済み、**When** その鍵で署名されたレスが届く、**Then** 採番されず、他の参加者には一切配布されない
2. **Given** 視聴者が特定の板鍵を NG 済み、**When** その鍵のレスが確定配布される、**Then** その視聴者の画面でのみ非表示になり、レス番号は欠番として維持される
3. **Given** 利用者が板鍵をローテーションした(または新規参加)、**When** 新しい鍵で初回の書き込みをする、**Then** 通常より高い計算コスト(PoW)を満たさない限り採番されない
4. **Given** NG/BAN 対象の板鍵と短縮 ID 表示が同じ別の鍵、**When** その別の鍵のレスが届く、**Then** NG/BAN は適用されない(照合は完全な鍵で行う)

---

### User Story 5 - スレのライフサイクル (Priority: P2)

配信者として、レスが上限に達したら次スレへ移行したい。配信が終わったらスレを閉じ、
ネットワークに残骸を残したくない。視聴者として、途中から参加しても全ログを読みたい。
ホストが落ちたら書けなくなるのは仕方ないが、読んだ分は読み続けたい。

**Why this priority**: ライフサイクルが未定義でも短時間の実況は成立するが、
スレッドフロート(次スレ)と揮発性は掲示板文化との整合に必須。

**Independent Test**: レス上限を小さく設定した実況スレで、次スレ移行・ホスト切断時の
凍結・明示クローズ時の削除・途中参加の全ログ同期を検証できる。

**Acceptance Scenarios**:

1. **Given** レス数が上限(既定 1000)に達したスレ、**When** 次の書き込みが届く、**Then** ホストは次スレへ移行し、旧スレは書き込み不可となり、新規書き込みは次スレに採番される
2. **Given** 進行中のスレ、**When** 配信者が明示的にスレをクローズする、**Then** 参加者ノードはスレデータを削除する(揮発)
3. **Given** 進行中のスレ、**When** ホストが明示クローズなしに切断した(瞬断・障害)、**Then** スレは凍結され、参加者は取得済みレスを閲覧し続けられるが書き込みはできない
4. **Given** 500 レス進行済みのスレ、**When** 新しい視聴者がスレを開く、**Then** 全 500 レスが確定順序どおりに取得・表示される
5. **Given** 同一の板、**When** ホストが次スレへ移行する、**Then** 参加者の板鍵・NG・BAN は次スレへそのまま引き継がれる(板単位スコープ)

---

### User Story 6 - 既存実況クライアントからの読み書き(互換 API) (Priority: P2)

長年の実況利用者として、使い慣れた専ブラ・コメントビューワを自分のノード(loopback)に
向けるだけで、P2P 実況スレを従来の掲示板と同じように読み書きしたい。

**Why this priority**: 既存ツールの利用者が実況文化の中核であり、互換経路がないと文化ごと
移行できない。ただし本体の読み書き(US1/US2)が成立していることが前提のため P2。

**Independent Test**: 自ノード + 互換 HTTP クライアント(実機ツールまたは互換リクエストの
シミュレーション)で、スレ一覧取得 → レス取得 → 書き込み → 反映確認を単体で検証できる。

**Acceptance Scenarios**:

1. **Given** 自ノードがスレに接続済み、**When** 互換クライアントがスレ一覧を取得する、**Then** 板のアクティブスレが従来形式で返り、板設定(板タイトル・名無しのデフォルト名・スレ最大レス数)も従来の板設定提示形式で参照できる
2. **Given** 互換クライアントがスレ本文を取得する、**When** スレに新しい確定レスがある、**Then** 確定順序どおりのレスが従来形式(名前・メール・ID 表示を含む)で返る
3. **Given** 互換クライアントが書き込みを送信する、**When** 自ノードが受理する、**Then** 板鍵で自動署名され通常経路と同一の検証(名前欄の `#` 除去・PoW・レート制限)を経てホストへ送信され、採番確定後の再取得に反映される
4. **Given** loopback 以外の送信元、**When** 互換 API にアクセスする、**Then** 拒否される
5. **Given** 凍結またはクローズ済みのスレ、**When** 互換クライアントが書き込みを送信する、**Then** 従来クライアントが解釈できる形式のエラーが返り、内部情報は漏洩しない

---

### Edge Cases

- レス上限到達と同時に複数の書き込みが届いた場合、上限を超える採番は行われず、次スレ移行完了までの書き込みは定型応答で拒否される(内部情報を漏洩しない)
- ホストの明示クローズと瞬断は区別される: クローズはスレ主署名付きの通知、通知なき切断は凍結。凍結中の再接続はバックオフ付きで試行され、ホスト再開後の実況は次スレとして扱われる
- 参加上限(既定 128 接続)到達時、ホストは新規接続を定型応答で拒否する。拒否された利用者には「満員」であることのみが表示される
- 順序確定情報に欠落(レス番号の飛び)を検出した参加者は、ホストへ再送を要求できる
- BAN された参加者の書き込みは採番されない。書き込み失敗は定型応答で返し、BAN が理由であることは開示しない(荒らしへの情報提供を避ける)
- 板鍵を紛失した利用者は新規鍵の生成(ローテーションと同経路)で復帰する。初回 PoW が課される
- レスの時刻表示は参考情報であり、正となるのはレス番号(採番順)のみ。端末間の時計ずれは表示順序に影響しない
- 実況スレだけを読む(チャンネルは視聴しない)、視聴だけしてスレは開かない、の両方が可能
- スレ announce の鮮度が切れた(ホストが再発行しなくなった)場合、スレ一覧から除去される。接続済み参加者の扱いは凍結と同じ
- 検証に失敗する板設定(2048 文字を超えるローカルルール等)を受信した参加者ノードは破棄し、セキュリティイベントを記録する
- ローカルルールの Markdown に HTML やスクリプトが含まれていても、描画・実行されない(安全なサブセットのみ描画)
- スレ最大レス数を進行中スレの現在レス数より小さく変更しても、進行中スレには影響せず次スレから適用される
- 互換 API の文字コード変換(Shift_JIS)で変換不能な文字は置換され、従来形式の区切りを壊さない(001 R5 の index.txt と同じ方針)
- 互換クライアントからの連投・過大本文は通常経路と同一のレート・サイズ上限で拒否され、従来クライアントが解釈できる形式のエラーで返る
- 次スレ移行後、互換クライアントにはスレ一覧の再取得で新スレとして見える(旧スレは凍結状態の取得済み内容のみ)

## Requirements *(mandatory)*

### Functional Requirements

**発見・接続**

- **FR-001**: 配信者は自分が掲載中のチャンネルに対して実況スレを開設できなければならない (MUST)。スレの開設・管理(スレ主)は当該チャンネルを掲載した配信者ペルソナに限定しなければならない (MUST)
- **FR-002**: スレ announce は既存の発見網で伝搬し、チャンネル掲載と同様の鮮度管理(定期再発行・期限切れの除去と再伝搬禁止)に従わなければならない (MUST)
- **FR-003**: スレ announce の署名者が対象チャンネルの掲載ペルソナと一致しない場合、その announce を不可視(保持・再伝搬しない)とし、セキュリティイベントを記録しなければならない (MUST)
- **FR-004**: スレ announce の受信のみを契機にホストへ接続してはならない (MUST NOT)。スレへの接続は利用者の明示操作を起点としなければならない (MUST)
- **FR-005**: スレ接続時、相手がスレ主ペルソナの鍵を保持していることをチャレンジにより検証しなければならない (MUST)。検証失敗時は切断し、再試行はバックオフに従わなければならない (MUST)
- **FR-006**: 参加上限到達時、ホストは新規接続を内部情報を漏洩しない定型応答で拒否しなければならない (MUST)

**書き込み・順序確定**

- **FR-007**: 書き込みはホストへ直接送信され、ホストは多段検証(サイズ・形式・署名・PoW・レート)を通過した書き込みにのみレス番号を一意に採番し、署名付きの順序確定情報を全参加者へ配布しなければならない (MUST)
- **FR-008**: 参加者の画面には順序確定済みのレスのみを表示しなければならない (MUST)。例外は自分の未確定投稿で、「送信中」として区別表示しなければならない (MUST)
- **FR-009**: アンカー(>>n)は全端末で同一のレス(番号 n として確定したレス)に解決されなければならない (MUST)
- **FR-010**: 途中参加者は接続時にスレの全既存レスと順序確定情報を取得できなければならない (MUST)
- **FR-011**: スレ主以外の鍵で署名された順序確定情報は破棄し、セキュリティイベントを記録しなければならない (MUST)

**ライフサイクル**

- **FR-012**: 板は配信者ペルソナ単位とし、板ごとのアクティブスレは常に 1 本でなければならない (MUST)
- **FR-013**: レス数が上限(既定 1000、板主が 100〜4000 の範囲で設定可能)に達するか、配信者が明示操作した場合、次スレへ移行しなければならない (MUST)。移行後の旧スレへの書き込みは受け付けてはならない (MUST NOT)
- **FR-014**: ホストとの接続が失われたスレは凍結(書き込み不可・取得済みレスの閲覧は可)としなければならない (MUST)。スレ主署名付きのクローズ通知を受けた参加者ノードは、当該スレのデータを削除しなければならない (MUST)
- **FR-015**: スレデータ(レス・順序確定情報)は永続化してはならない (MUST NOT)(ノード再起動で消える。板鍵・NG・BAN は永続化する)

**匿名性・モデレーション**

- **FR-016**: 書き込みには板単位の参加者鍵(板鍵)を用いなければならない (MUST)。板鍵は既存ペルソナ鍵と別系統とし、相互にリンクされてはならない (MUST NOT)。閲覧のみの参加に鍵を要求してはならない (MUST NOT)
- **FR-017**: 利用者は板鍵を明示操作で再生成(ローテーション)できなければならない (MUST)。新規板鍵での初回書き込みには通常より高い計算コスト(PoW)を課さなければならない (MUST)
- **FR-018**: NG(視聴者側非表示)および BAN(スレ主側採番拒否)の照合は完全な鍵で行わなければならない (MUST)。短縮 ID 表示を照合に用いてはならない (MUST NOT)
- **FR-019**: スレ主は板鍵単位の採番拒否(BAN)および接続単位の切断・接続拒否を行えなければならない (MUST)。NG・BAN はローカル情報であり、ネットワークへ公開してはならない (MUST NOT)
- **FR-020**: NG されたレスの非表示は当該端末のみに適用し、レス番号は欠番として維持しなければならない (MUST)

**防御**

- **FR-021**: スレ単位・板鍵単位・接続単位の受信レートおよびサイズの上限を設け、違反は破棄しセキュリティイベントを記録しなければならない (MUST)。エラー応答は内部情報を漏洩してはならない (MUST NOT)

**板設定・名前欄**

- **FR-022**: 板主(スレ主)は板設定 — 板タイトル・スレ最大レス数・名無しのデフォルト名・ローカルルール(最大 2048 文字の Markdown テキスト) — を設定できなければならない (MUST)。板設定は参加者へ配布され、スレの閲覧時に参照できなければならない (MUST)
- **FR-023**: 板タイトル・名無しのデフォルト名・ローカルルールの変更は即時に参加者へ配布・反映しなければならない (MUST)。スレ最大レス数の変更は進行中のスレに適用してはならず (MUST NOT)、次スレから適用しなければならない (MUST)
- **FR-024**: レスは任意の名前を持てなければならない (MUST)。名前が空のレスは板の名無しのデフォルト名で表示しなければならない (MUST)。トリップは提供せず、名前欄の `#` 以降は送信前に除去しなければならない (MUST)(トリップ入力の癖によるパスワード類の漏洩事故防止)
- **FR-025**: 板設定は受信時に検証しなければならない (MUST)(サイズ・形式)。ローカルルールの Markdown は安全なサブセットとして描画し、HTML の直接埋め込みやスクリプトの実行を許可してはならない (MUST NOT)

**互換 API(伝統的 2ch 形式)**

- **FR-026**: 各利用者ノードは loopback に伝統的 2ch 形式の互換 API(スレ一覧・スレ本文取得・板設定提示・書き込み)を提供しなければならない (MUST)。loopback 以外からのアクセスを受け付けてはならない (MUST NOT)。既存クライアントはトークン認証を送れないため、保護は loopback 限定・Host 検証・レート制限で行う
- **FR-027**: 互換 API の読み出しは確定済みレスのみを従来形式で返さなければならない (MUST)。板設定は従来の板設定提示形式に対応付けて提供しなければならない (MUST)
- **FR-028**: 互換 API 経由の書き込みは、通常の書き込みと同一の経路・検証(板鍵での自動署名・名前欄の `#` 除去・PoW・レート制限)を通らなければならない (MUST)。互換経路だけの抜け道を作ってはならない (MUST NOT)
- **FR-029**: レスはメール欄を属性として持ち、保持・伝搬・表示互換に用いる (MUST)。メール欄に機能的意味(表示順の変更等)を持たせてはならない (MUST NOT)
- **FR-030**: 互換 API のエラー応答は従来クライアントが解釈できる形式で返し、かつ内部情報を漏洩してはならない (MUST NOT)

### Key Entities

- **板**: 配信者ペルソナ単位の掲示板。板鍵・NG・BAN のスコープ。属性: 配信者ペルソナ、アクティブスレへの参照、板設定(板タイトル・スレ最大レス数・名無しのデフォルト名・ローカルルール)
- **スレ**: レスの順序付き列。属性: スレ主ペルソナ、対象チャンネル、スレ番号(次スレの世代)、レス上限、状態(進行中/凍結/クローズ)
- **レス**: 1 書き込み。属性: 本文、名前(任意。空は名無しのデフォルト名で表示)、メール(任意。表示互換のみで機能的意味なし)、板鍵(署名者)、レス番号(確定後)、投稿時刻(参考)、アンカー参照
- **順序確定情報**: スレ主が署名した採番結果。属性: スレ、確定済みレス番号列とレスの対応、スレ主署名
- **スレ announce**: スレの存在と接続先の掲載情報。属性: スレ主ペルソナ署名、対象チャンネル、接続先、鮮度
- **板鍵**: 参加者の板単位の書き込み鍵。属性: 対象の板、鍵、作成日時。既存ペルソナとは独立
- **NG / BAN エントリ**: ローカルの非表示・拒否対象。属性: 対象板鍵(完全鍵)または接続元、種別

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 参加者 100 人・バースト 30 レス/分の条件で、書き込み送信から全参加者での確定表示まで p99 で 5 秒以内
- **SC-002**: 上記条件の多ノードテストで、全端末のレス番号・表示順序・アンカー解決の不一致が 0 件
- **SC-003**: 設定上限最大(4000 レス)まで進行済みのスレへ途中参加した利用者が、参加操作から 15 秒以内に全ログを閲覧できる
- **SC-004**: 署名不一致の announce・スレ主以外の順序確定情報・検証違反の書き込みは 100% 不可視となり、セキュリティイベントとして記録される
- **SC-005**: スレ announce の受信だけでは外向き接続が 1 件も発生しない(自動接続 0 件)
- **SC-006**: 実況機能の追加後も、既存のチャンネル発見の成功基準(掲載から 60 秒以内の一覧反映)が維持される
- **SC-007**: 利用者所有の既存実況ツール(専ブラ・コメントビューワ)が、改修なしで自ノードの互換 API に対しスレ一覧取得・レス取得・書き込みを成立させる(実機確認)

## Assumptions

- スレ主(配信者)ノードは着信可能である(配信者は配信のためにポート開放済みであることが通例)。着信不可能な配信者によるスレ開設は v1 ではサポートしない
- 規模想定はバースト 30 レス/分・1 スレ参加者 100 人弱(現在の実況文化の実勢)。星型配送で成立し、参加上限は既定 128 接続(設定可能)とする
- 配送の木構造化(中継ノード)・gossip 的修復(Plumtree 等)はスコープ外。プロトコルには将来のメッセージ種別追加の余地のみ確保する
- 非配信者(視聴者)がスレ主になる汎用分散 BBS はスコープ外。Winny2 で実証されたマスター匿名性問題(`docs/research/winny2-bbs.md`)を抱えるため、別ソフトウェアの領分とする
- 過去ログの保存・アーカイブ機能は持たない(揮発)。利用者が閲覧中の内容を手元に保存することは技術的に妨げないが、機能としては関知しない
- レスのイベントスキーマに既存の標準(NIP-53 kind 1311 等)を援用するかは plan の research で確定する(nostr 援用境界 — 001 FR-014 の原則に従う)
- 互換 API の詳細形式(スレ一覧・スレ本文の行形式、書き込みパラメータ、エラー文字列、文字コードの既定)は plan の contracts で確定する。適合基準は「利用者所有の既存実況ツールが無改修で動くこと」(SC-007)
- 5ch 新 API(認証・JSON 系)は不採用。必要が生じた場合に別途検討する(YAGNI)
- 実況スレの閲覧・書き込みとチャンネル視聴(PCP)は独立であり、どちらか一方のみの利用も可能
- 既存の発見網に追加されるスレ announce のイベント率は、既存の容量設計(001 R16)の余裕内に収まる(検証は plan で行う)
