# Quickstart: 分散型配信情報共有ネットワーク(YP代替)

機能が end-to-end で動くことを検証する手順。契約の詳細は [contracts/](./contracts/)、
データ定義は [data-model.md](./data-model.md) を参照。

## 前提

- Windows 10/11 x64
- Rust stable ツールチェーン(`rustup`)— 開発時のみ。利用者は配布 exe のみで動作
- 検証用: PeerCastStation(配信側テスト)、任意の YP ブラウザ(index.txt 表示テスト)
- ピア: リレーサーバーは存在しない。ローカル検証は同一 PC 上の複数プロセス
  (ポートを変えて起動)を相互にピア登録する。実網検証は 2 台以上の PC を用意

## 1. ビルドとテスト

```powershell
cargo build --release          # 単一 exe: target\release\peca-p2p-yp.exe
cargo test                     # unit + contract + integration(モックピアでの多ノード伝搬含む)
cargo test --test cucumber     # Gherkin シナリオ(spec.md 対応。実装前は失敗すること)
```

期待結果: すべてのテストがパス(実装完了後)。Gherkin にはネガティブシナリオ
(過大ペイロード拒否・なりすまし検出・大量偽登録耐性)が含まれる。

## 2. 起動と初期設定

```powershell
.\target\release\peca-p2p-yp.exe
```

1. ブラウザで `http://127.0.0.1:7180/` を開く
2. **ペルソナ作成**: ペルソナ画面で新規作成(label は任意。ローカル専用)
3. **ピア登録**: 掲示板/SNS 等で入手したピアアドレス(`host:port`)群をピア画面に
   貼り付けて一括登録(既定シードピアは同梱されない — research R10)
4. 状態表示で着信可否(UPnP の結果)を確認。着信不可でも全機能が使える(FR-016)

期待結果: `GET /api/v1/status` で established ピア数 ≥ 1。
以後はピア交換(PEX)で接続先が自動拡大する(FR-015)。

**ローカル 2 ノード検証の例**(同一 PC):

```powershell
# ノード A(既定ポート)
.\peca-p2p-yp.exe
# ノード B(ポートをずらして起動。設定またはコマンドラインで p2p/http/pcp を変更)
.\peca-p2p-yp.exe --p2p-bind 0.0.0.0:7157 --http-bind 127.0.0.1:7190 --pcp-bind 127.0.0.1:7156
# ノード B の UI(127.0.0.1:7190)でピア「127.0.0.1:7147」を登録 → A-B が established になる
```

## 3. 掲載側の検証(US1 / SC-001)

1. PeerCastStation の YP 設定に `pcp://127.0.0.1:7146/` を追加
2. PeerCastStation で配信を開始(掲載 YP に本ソフトを指定)
3. UI の「掲載中」画面で当該チャンネルと伝搬先ピア数を確認

期待結果: 配信開始から 60 秒以内に、別ノード(手順 4)の一覧に表示される。
チャンネル詳細変更(曲情報等)が 60 秒以内に反映される。配信終了で `status=ended` が発行される。

## 4. 発見側の検証(US2 / SC-004)

別 PC(または手順 2 のノード B)で:

1. 本ソフトを起動し、掲載側ノードと(直接またはピア交換経由で)接続する
2. UI の一覧に手順 3 のチャンネルが表示されることを確認(接続時同期 SYNC_REQ により
   接続直後でも取得できる。5 秒以内)
3. YP ブラウザの取得先に `http://127.0.0.1:7190/index.txt`(ノード B の HTTP ポート)を設定し、
   一覧表示と文字化けの有無を確認(Shift_JIS 既定 — research R5 のリスク解消ポイント)
4. 一覧のトラッカー情報(TIP)経由で PeerCastStation の視聴が開始できることを確認

## 5. 継続性の検証(US3 / SC-002)

1. 3 ノード以上(A=掲載、B・C=発見)でチェーン状・メッシュ状に接続し、手順 3〜4 を成立させる
2. 中間ノードまたは接続ピアの 1 つを停止する
3. 掲載・一覧が残りのピア経由で中断なく継続することを確認
4. すべてのピアを停止 → UI に到達不能バナーが出ること、ピア復帰で自動的に掲載再開することを確認

## 6. NAT/着信不可の検証(FR-016 / SC-009)

1. 一方のノードの P2P 待受を無効化(または UPnP を無効化し着信不可の環境に置く)
2. そのノードから外向き接続のみで、掲載(手順 3)と発見(手順 4)の両方が成立することを確認
3. UI の状態表示が「外向き接続のみで参加中」を示すことを確認

## 7. セキュリティシナリオの検証(spec セキュリティシナリオ)

自動テスト(cucumber + モックピア)で実行:

- 64KB 超フレーム / 16KB 超イベント → 切断・破棄+`p2p_oversize`/`event_oversize` ログ(SC-007)
- 署名不正イベント → 一覧に出ない・再伝搬されない+`event_invalid_sig` ログ(SC-005)
- 大量偽チャンネル投入 → ミュート/ピア切離し/PoW 閾値で一覧の実用性維持(FR-008)
- PEX で不正アドレス(過大件数・形式不正・自アドレス)→ 破棄+`pex_rejected` ログ(FR-015)
- `javascript:` 等のコンタクト URL → UI で警告フラグ(FR-012)

## 8. セットアップ時間の計測(SC-006)

初見ユーザーが「exe 入手 → 起動 → ピアアドレス貼り付け → 一覧閲覧」を 15 分以内に
完了できることを、手順書(README)に沿って計測する。
