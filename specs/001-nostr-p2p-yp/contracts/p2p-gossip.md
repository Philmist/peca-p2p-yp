# Contract: P2P gossip ワイヤプロトコル(ノード間)

**Role**: 利用者ノード同士が署名済みイベント(contracts/nostr-events.md)を直接交換する
自前プロトコル。リレーサーバーは存在しない(FR-002, FR-014)。
**本契約はインターネットに露出する最大の信頼境界**であり、すべての受信データは
多段検証を通す(Principle II — trust nothing from the network)。

**設計判断の記録**: research R13(方式選定)、R14(PEX)、R16(規模導出)。
伝搬・重複抑制・同期の状態機械は Principle V の形式的検証候補(判定 ADR は tasks 先頭フェーズ)。

## トランスポート

- TCP。既定待受 `0.0.0.0:7147`(設定変更・無効化可 — 無効時は外向き接続のみ、FR-016)
- フレーム = `長さ(4 バイト BE、ペイロードのバイト数)` + `ペイロード(JSON、UTF-8)`
- トランスポート暗号化なし(完全性・真正性はイベント署名で担保。掲載情報は公開データ。
  判断は ADR に記録 — plan.md Constitution Check)
- keepalive: 60 秒間隔で `PING`、120 秒無応答で切断

## メッセージ種別

ペイロードは `{"type":"<TYPE>", ...}` の JSON オブジェクト。

| type | 方向 | フィールド | 意味 |
|------|------|-----------|------|
| `HELLO` | 発→受 | `version`(u32、v1=1)、`listen_port`(u16、待受なしは 0)、`features`(文字列配列)、`nonce`(u64、起動時生成の乱数 — 自己接続検出用)、`ts`(unix 秒 — 送信時点のローカル時刻。受信側の時計ずれ自己診断用) | 接続開始。最初のフレームでなければ切断 |
| `HELLO_ACK` | 受→発 | 同上 | 受理。バージョン非互換は `CLOSE` して切断 |

- `version` の互換判定は**完全一致**(v1 = 1 のみ受理)。不一致は `CLOSE`(reason: `incompatible`)
- `features` は v1 では定義済みの値を持たない(空配列を送る)。**未知の feature 値は無視しなければならない (MUST)**(前方互換の規則)
- `listen_port` は自ノードが TCP 待受中のポートを申告する(待受無効 = Settings `p2p_bind` 空文字のときは 0)。**UPnP の成否は申告値に影響しない** — 実際に着信可能かは、受信側が当該 `host:listen_port` へ自ら接続して検証する(verified の定義 — data-model §PeerEndpoint)。申告値は未検証情報として扱う(Principle II)
- `ts` は未検証の申告値である。受信側は**自ノードの時計ずれの自己診断にのみ**用いる(複数の established ピアとの差分の**中央値**が `max_clock_skew_sec` を超える場合に時刻同期を促す通知を表示 — spec Edge Case「時計のずれ」)。他ノードのイベント検証・接続可否の判断に使用してはならない (MUST NOT — Principle II)
| `EVENT` | 双方向 | `event`(nostr イベント JSON) | イベントの伝搬(発行・再伝搬とも同形) |
| `SYNC_REQ` | 双方向 | `since`(unix 秒) | 接続時同期の要求。応答側は自身の EventStore の live かつ鮮度窓内のイベントのうち `created_at ≥ max(since, now − freshness_window_sec)` のもののみを返す(since による絞り込み。非標準の since 値を受けても応答範囲が鮮度窓を超えて拡大しない) |
| `SYNC_DONE` | 双方向 | `count`(u32) | SYNC_REQ への応答完了(応答本体は `EVENT` の列) |
| `GET_PEERS` | 双方向 | — | ピア交換の要求(FR-015) |
| `PEERS` | 双方向 | `peers`(`"host:port"` 文字列配列、≤ 64 件) | **検証済み(verified=1、自ノードが外向き接続に成功した実績のある)ピアのみ**を `last_ok_at` の新しい順に最大 64 件返す(research R14)。外向き接続の成功実績をもって着信可能と判定する |
| `PING` / `PONG` | 双方向 | `nonce`(u64) | keepalive |
| `CLOSE` | 双方向 | `reason`(定型コード) | 正常切断。内部情報を含めてはならない (MUST NOT) |

## セッション状態機械

```text
[outbound] connect → HELLO 送信 → HELLO_ACK 受信 → established
[inbound]  accept  → HELLO 受信 → HELLO_ACK 送信 → established
established → (SYNC_REQ/EVENT/GET_PEERS/PING の交換) → CLOSE または異常切断 → closed
```

- established 前に HELLO/HELLO_ACK 以外を受信したら即切断(`p2p_invalid_frame`)
- established 直後に双方が `SYNC_REQ`(since = now − freshness_window_sec)を送るのが標準フロー
- 切断はいつ発生してもよい(US3: ピア障害時の継続性)。再接続は指数バックオフ
  (初期 5 秒、係数 2、上限 300 秒。接続成功でリセット)
- **ネットワーク分断・再結合(パーティション)**: 分断中の欠落は要件としての回復保証を置かず、
  再接続時の接続時同期(SYNC_REQ)と鮮度期限(FR-006、再発行 60 秒)によって一覧が収束する
  ことを意図した設計とする(分断中に失われた情報は鮮度窓 600 秒以内の再発行で回復する)

## 伝搬規則(gossip)

1. **受信** `EVENT` → 検証パイプライン(下記)を通す
2. **重複判定**: event id が DedupCache にあれば黙って破棄(再伝搬しない)
3. **格納**: EventStore の置換規則 `(kind, pubkey, d)` + last-write-wins で格納。
   旧版しか置換できない(より古い)イベント、および**既に保持している同一 event id の
   イベントは格納も再伝搬もしない**(DedupCache の保持期間(10 分)を超えて同一イベントを
   再受信した場合の第二の防壁 — ループ再発を防ぐ設計上の不変条件)
4. **再伝搬**: 格納に成功したイベントのみ、**受信元を除く** established な全ピアへ `EVENT` を送信
5. **終端保証**: ホップ制限は設けない。重複抑制(2)と鮮度期限(FR-006 / research R2)が
   伝搬の終端を保証する(Principle V 判定対象の中核性質: ループ不在・重複爆発不在・
   live イベントの到達性。判定 ADR ではこの 3 性質を検査すべき不変条件として扱う)
6. **SYNC 応答への適用**: `SYNC_REQ` への応答として受信した `EVENT` にも上記 1〜4 を
   そのまま適用する。既知(格納済み・同一 id)のイベントは (2)(3) で再伝搬されないため、
   接続確立のたびに保持イベント全量が網へ再フラッディングされることはない。
   未知のイベントのみが再伝搬され、これが分断再結合時の回復経路となる

## 受信検証パイプライン(Principle II)

順序どおりに検査し、失敗したら破棄してセキュリティイベントに記録する。
閾値超過は接続の切断と fail_count 反映を伴う:

| # | 検査 | 上限/規則 | 違反時ログ |
|---|------|-----------|-----------|
| 1 | フレーム長 | ≤ 64KB(EVENT を含む全メッセージ) | `p2p_oversize` |
| 2 | 受信レート | ≤ 256KB/秒・≤ 200 メッセージ/秒(1 ピアあたり) | `p2p_rate_limited` |
| 3 | JSON 形式 | パース可能、`type` が既知、未知フィールドは無視 | `p2p_invalid_frame` |
| 4 | EVENT 内容 | contracts/nostr-events.md の受信検証(サイズ→署名→形式→時刻→内容→PoW) | `event_*` |
| 5 | PEERS 内容 | 件数 ≤ 64、各要素は `host:port` 形式・長さ ≤ 256、自アドレス・重複は破棄。IPv6 リテラルは `[addr]:port` のブラケット表記のみ許容(例 `[2001:db8::1]:7147`。ブラケットなしでコロンを複数含むものはパース不能として破棄) | `pex_rejected` |
| 6 | SYNC 応答量 | SYNC_REQ 1 回への応答 EVENT は ≤ event_store_max 件。超過は切断 | `p2p_rate_limited` |

- **SYNC 応答とレート制限の両立**: 検査 2 の受信レート制限は接続時同期の期間中も適用される。
  送信側は SYNC 応答を受信側の上限(256KB/秒・200 メッセージ/秒)以下に平滑化して送信
  しなければならない (MUST)。正当な接続時同期がレート制限に抵触して切断されないことを
  両立条件とする(典型時 ~100 チャンネル ≈ 150KB は 1 秒未満、最悪時 4,096 件でも
  平滑化により完了する — SC-004 の「通常時 5 秒以内」は典型時を基準とする)

- 未検証ピア(接続実績なし)を `PEERS` で再共有してはならない (MUST NOT — research R14)
- エラー・CLOSE の reason に内部情報(パス・スタックトレース)を含めてはならない (MUST NOT)

## 接続管理

- 外向き: `p2p_outbound_target`(既定 8)本を維持。候補は PeerEndpoint から
  `manual 優先 → last_ok_at 新しい順 → fail_count 少ない順`
- 着信: `p2p_inbound_max`(既定 32)本まで。超過は HELLO_ACK 前に CLOSE
- 同一アドレスへの多重接続は 1 本に統合。自己接続(自分の待受への接続)は HELLO の
  `nonce`(起動時生成の乱数 — メッセージ種別参照)が自ノードの値と一致することで検出し、
  切断のうえ当該アドレスを PeerEndpoint から登録拒否する
- **inbound 相手の候補化**: inbound 接続の相手が申告した `listen_port`(> 0)は、
  接続元 host と組み合わせて PeerEndpoint 候補(source=`pex`、verified=0)として登録してよい。
  自ノードから当該 `host:listen_port` への外向き接続に成功して初めて verified=1 となり、
  再接続候補の優先対象・PEX 共有対象になる。`listen_port: 0` は候補登録しない
- **新規(未検証)候補への接続試行は 1 件/秒以下**に制限する(PEX への第三者アドレス注入を
  反射・負荷集中攻撃として悪用されることの緩和 — 下記「脅威と対応範囲」)
- **全ピア到達不能時の自動回復**(US3 シナリオ 3): enabled なすべての既知ピア
  (manual・pex 由来、fail_count 閾値超過で降格済みのものを含む)を再試行対象とし、
  通常の候補順で指数バックオフを継続する。降格は平常時の候補優先度を下げるのみで、
  全ピア到達不能時の再試行からは除外しない。いずれかへの接続が成功し次第、掲載を自動再開する
- UPnP(research R15)は待受可否にのみ影響し、本契約のメッセージには現れない

## 脅威と対応範囲(悪意あるピア — 脅威モデル ADR への入力)

不可用性(停止・切断)以外の悪意ある挙動について、v1 の対応と ADR 委譲範囲を明示する:

| 脅威 | v1 の対応 | 備考 |
|------|-----------|------|
| 旧版イベントの再配信 | 無害化済み: EventStore の置換規則(署名済み `created_at` 比較)と `expiration` により、旧版は格納も再伝搬もされない | 本契約 §伝搬規則 3 |
| イベントの選択的隠蔽(再伝搬拒否)・不完全な SYNC 応答 | 単一ピアの隠蔽は他ピアからの受信で補完される(接続度 8 の多重経路)。完全な遮蔽は Eclipse 攻撃に帰着 | 検出要件は置かない(意図的な非目標)。残余リスクは脅威モデル ADR に記録 |
| Eclipse 攻撃(接続枠の占有による視界遮断) | 緩和: manual ピアの優先維持(§接続管理の候補順)・外向き/着信枠の分離。接続先の多様性確保(サブネット分散等)の要否は**脅威モデル ADR のスコープに含める** | |
| EventStore 追い出し(大量イベント注入で `created_at` 古い順破棄を悪用) | 緩和: イベントサイズ・受信レート上限、任意 PoW。発信者(pubkey)単位・ピア単位のクォータの要否は**脅威モデル ADR のスコープに含める** | data-model §EventStore |
| 制御メッセージ(HELLO/PEERS/SYNC_REQ/CLOSE 等)の経路上改ざん | イベント署名の保護は **EVENT のみ**に及ぶ。特に PEERS 改ざんによるピアリスト毒入れは実接続検証(R14 — 未検証ピアの不使用・再共有禁止)で緩和する。「完全性はイベント署名で担保」の適用範囲外であることを**非暗号化判断 ADR(⑤)に明示する** | §トランスポート |
| PEX への第三者アドレス注入(反射・負荷集中) | 未検証ピアの再共有禁止に加え、自ノードの接続試行自体が反射トラフィックとなるため、新規候補への接続試行レート制限(§接続管理)と fail_count 降格で増幅を抑える。残余リスクの評価は**脅威モデル ADR のスコープに含める** | research R14 |

## 検証方法

- `tests/contract/`: フレーム境界(分割・結合・過大長)、HELLO 順序違反、不正 JSON、
  PEERS 上限超過、SYNC 応答上限のフィクスチャ検証
- **契約フィクスチャの共有**: 上記フィクスチャ(フレーム境界・メッセージ JSON のテストベクタ)は
  本実装とモックピア(research R11 — プロトコル契約の参照実装)の**両方に同一のものを適用**し、
  契約書とモック実装の乖離を検出する
- 統合テスト: インプロセスのモックピアで 3〜10 ノードのトポロジを構成し、
  伝搬(SC-001)・重複抑制(ループ不在)・接続時同期・ピア停止時の継続(US3)・
  PEX 拡大(FR-015)・**churn(ノードの高頻度な参加・離脱の反復下での伝搬・PEX・
  ピアリスト鮮度)**を検証
- Principle V 判定 ADR で該当となった場合: `docs/formal/` の PlusCal モデルで
  ループ不在・到達性・置換の単調性を検査してから実装する
