# Feature Specification: 分散型配信情報共有ネットワーク(YP代替)

**Feature Branch**: `001-nostr-p2p-yp`

**Created**: 2026-07-02

**Status**: Draft

**Input**: User description: "PeerCastはP2P型のソフトウェアでありながらその配信情報が集積されるYellowPage(YP)は基本的には1つのサーバーとして運営され、YPが死んでしまうとPeerCastコミュニティ自体が死んでしまうことが想定される。このため配信者と視聴者間でほぼサーバーを介しない形で配信情報を共有することが可能なP2Pソフトウェア/ソリューションを作りたい。さらに将来的にYPの機能だけではなく、現在実況に用いられるBBSに相当する機能も代替することで中央サーバーに依存せずとも配信/実況を継続できるようにしたい。ただし1からの再設計を避けたいので既に実績があるnostrの技術スタック、具体的にはそのプロトコル定義のNIPを援用し先の要求を満たす(最低でもWindows上で動く)ソフトウェアを作って欲しい。"

## 概要

PeerCast コミュニティの配信情報流通は、現在単一運営の YellowPage(YP)サーバーに
依存しており、YP の停止がコミュニティ全体の停止に直結する単一障害点となっている。
本機能は、配信者と視聴者が特定の中央サーバーに依存せず配信情報
(チャンネル一覧・トラッカー情報)を共有できる分散型の仕組みを、
Windows 上で動作するソフトウェアとして提供する。
既存の PeerCast クライアント(PeerCastStation 等)は改造なしで本ソフトウェアを
従来の YP と同じように利用できることを目標とする。
将来フェーズでは、実況 BBS に相当するコメント共有機能へ同じ分散基盤を拡張する。

## Clarifications

### Session 2026-07-02

- Q: 匿名文化の要件と発信者検証・信頼をどう両立する識別子モデルとするか? → A: ペルソナモデル。利用者は複数の識別子(ペルソナ)を自由に作成・切替・破棄でき、識別子間のリンクはシステム上存在せず本人が明かさない限り紐づけ不可。同一ペルソナ内では継続性(なりすまし防止・信頼)が保たれる
- Q: 履歴のない新規ペルソナのチャンネルを一覧で既定どう扱うか? → A: オープン型。新規ペルソナも既定で一覧に表示し、利用者はミュート/非表示/共有先選別で自衛する(既定非表示による参入障壁は設けない)

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 配信者によるチャンネル掲載 (Priority: P1)

配信者は PeerCast クライアントの掲載先(YP)として本ソフトウェアを指定して
配信を開始するだけで、チャンネル情報(名称・ジャンル・説明・コンタクトURL・
ビットレート・曲情報・トラッカー接続先など)が分散ネットワーク上に公開され、
中央 YP サーバーが存在しなくても他の参加者から発見可能になる。

**Why this priority**: 配信情報の「掲載」が成立しなければ発見も視聴もできない。
本機能全体の存在理由(YP の単一障害点排除)の起点となるため最優先。

**Independent Test**: 配信者環境 1 台で本ソフトウェアを起動し PeerCast
クライアントから配信を開始した後、別の参加者ノードから当該チャンネル情報が
取得できることを確認すれば単独で検証できる。

**Acceptance Scenarios** (Gherkin / Principle IV):

1. **Given** 配信者が本ソフトウェアを起動し、PeerCast クライアントの掲載先として設定している、**When** 配信者が PeerCast クライアントで配信を開始する、**Then** チャンネル情報が分散ネットワークへ公開され、他の参加者が 60 秒以内に取得可能になる
2. **Given** チャンネルが掲載中である、**When** 配信者がチャンネル詳細(ジャンル・説明・曲情報)を変更する、**Then** 変更内容が 60 秒以内に公開情報へ反映される
3. **Given** チャンネルが掲載中である、**When** 配信者が配信を終了する、**Then** チャンネルは終了状態として公開され、他の参加者の一覧から除去される

---

### User Story 2 - 視聴者によるチャンネル発見と視聴開始 (Priority: P1)

視聴者は本ソフトウェアを起動すると、現在配信中のチャンネル一覧
(名称・ジャンル・説明・リスナー数・リレー数・配信時間を含む)を閲覧でき、
選択したチャンネルを既存の PeerCast クライアントでそのまま視聴開始できる。

**Why this priority**: 掲載と対になる発見・視聴が成立して初めて
「YP なしで配信/視聴が継続できる」というコア価値が実証される。

**Independent Test**: 既知のチャンネルが分散ネットワーク上に掲載済みの状態で、
視聴者環境 1 台から一覧取得〜視聴開始(トラッカー解決)が完了することで検証できる。

**Acceptance Scenarios** (Gherkin / Principle IV):

1. **Given** 分散ネットワーク上に配信中のチャンネルが存在する、**When** 視聴者が本ソフトウェアでチャンネル一覧を開く、**Then** 配信中のチャンネルが名称・ジャンル・リスナー数等の情報とともに 5 秒以内に表示される
2. **Given** 視聴者がチャンネル一覧を表示している、**When** 視聴者がチャンネルを選択して視聴を開始する、**Then** 改造していない既存 PeerCast クライアントで当該チャンネルの再生が開始される
3. **Given** 配信終了から一定時間が経過したチャンネルがある、**When** 視聴者が一覧を更新する、**Then** 当該チャンネルは一覧に表示されない

---

### User Story 3 - 共有先障害時の継続性 (Priority: P2)

配信者・視聴者は、配信情報の共有に使っている中継点(共有先)の一部が停止しても、
操作を変えることなく掲載・発見を継続できる。単一の運営者・サーバーの停止が
コミュニティ全体の停止につながらない。

**Why this priority**: 本機能の根本動機(単一障害点の排除)を直接検証する
ストーリー。ただし P1 の掲載・発見が成立していることが前提となるため P2。

**Independent Test**: 複数の共有先を設定した状態で掲載・発見が成立していることを
確認後、共有先の 1 つを停止させ、掲載・発見が中断なく継続することで検証できる。

**Acceptance Scenarios** (Gherkin / Principle IV):

1. **Given** 配信者が複数の独立した共有先へチャンネルを掲載している、**When** 共有先の 1 つが停止する、**Then** 掲載は残りの共有先で継続され、視聴者は引き続きチャンネルを発見できる
2. **Given** 視聴者が一覧を閲覧している、**When** 接続中の共有先の 1 つが応答しなくなる、**Then** 一覧の取得は残りの共有先から継続され、視聴者への影響は表示遅延のみに留まる
3. **Given** すべての共有先へ到達できない、**When** 配信者が掲載を試みる、**Then** 利用者に共有先へ到達できない旨が分かりやすく通知され、到達可能になり次第自動的に掲載が再開される

---

### User Story 4 - チャンネルに紐づく実況コメント (Priority: P3・将来フェーズ)

視聴者は視聴中のチャンネルに紐づくコメント(実況)を投稿・閲覧でき、
実況 BBS に相当する体験を中央サーバーなしで得られる。

**Why this priority**: 依頼上「将来的に」と明示された拡張機能。v1 の範囲外だが、
チャンネル・投稿者の識別体系がこの拡張を妨げない設計であることを v1 で保証する
必要があるため、スコープ境界としてここに記録する。

**Independent Test**: 掲載中のチャンネルに対して 2 つの視聴者環境から
コメントを投稿し合い、相互に一定時間内で閲覧できることで検証できる。

**Acceptance Scenarios** (Gherkin / Principle IV):

1. **Given** 視聴者が配信中のチャンネルを視聴している、**When** 視聴者がコメントを投稿する、**Then** 同じチャンネルの他の視聴者が数秒以内にそのコメントを閲覧できる

---

### Edge Cases

- すべての共有先に到達できない場合、掲載・一覧取得はどうなるか → 明示的な通知と自動再試行(US3 シナリオ 3)
- 他者のチャンネルになりすました偽エントリが流通した場合 → 発信者検証で検出し表示しない/警告する(FR-005, セキュリティシナリオ参照)
- 大量の偽チャンネル登録(スパム・Sybil)が発生した場合 → 緩和策(FR-008)によって一覧の実用性を維持する
- 上限を超えるサイズや不正な形式のデータを受信した場合 → 拒否してセキュリティイベントとして記録する(FR-007)
- 配信者がポート開放できず外部から直接到達できない(firewalled)場合 → 掲載情報にその状態が反映され、視聴可否が視聴者に分かる
- 参加者間の時計のずれが大きい場合でも、掲載の有効期限判定が破綻しない
- 配信者が識別用の鍵(資格情報)を失った/漏洩した場合の扱い → 新しいペルソナでの再掲載は可能とし、失効の扱いは設計時に ADR で定める
- 匿名性(ペルソナ分離)とスパム緩和(FR-008)のトレードオフ → 新規ペルソナが不利益なく参加できることと、大量偽登録への耐性を両立させる必要がある(脅威モデル ADR で確定)

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 配信者は、チャンネル情報(名称・ジャンル・説明・コンタクトURL・ビットレート・コンテンツ種別・曲情報・トラッカー接続先・リスナー数/リレー数・配信経過時間)を分散ネットワークへ掲載できなければならない (MUST)
- **FR-002**: 配信情報の掲載・取得は、単一の中央サーバーや単一の運営者に依存してはならない (MUST NOT)。掲載は複数の独立した共有先へ行われ、任意の 1 つが失われても残りで発見可能でなければならない (MUST)
- **FR-003**: 視聴者は、配信中チャンネルの一覧と各チャンネルの詳細を取得・閲覧できなければならない (MUST)
- **FR-004**: 改造していない既存の PeerCast クライアントが、本ソフトウェアを従来の YP と同様に利用(チャンネル掲載・一覧取得・トラッカー解決)できなければならない (MUST)
- **FR-005**: すべての掲載情報は発信者を暗号学的に検証可能でなければならず (MUST)、検証に失敗した情報(なりすまし・改ざん)は表示してはならない (MUST NOT) — Principle I, II
- **FR-006**: 配信終了したチャンネル、および一定時間更新のないチャンネルは、一覧から自動的に除去されなければならない (MUST)
- **FR-007**: ネットワークから受信するすべてのデータはサイズ・形式・内容の検証を経なければならず (MUST)、不正なデータは拒否し、内部情報を漏らさずにセキュリティイベントとして記録しなければならない (MUST) — Principle II
- **FR-008**: 大量の偽登録やスパムによる一覧の汚染に対する緩和策を利用者に提供しなければならない (MUST)。既定では新規ペルソナのチャンネルも一覧に表示し(オープン型)、緩和は利用者側の手段(ミュート/非表示、共有先の選別、信頼する発信者リスト等のオプトイン)によって行う (MUST)。既定で未知のペルソナを非表示にしてはならない (MUST NOT)。具体的手法は脅威モデル ADR で確定する — Principle I, Security Requirements
- **FR-009**: 本ソフトウェアは Windows(現行サポート版)上で動作し、配信者・視聴者のいずれの役割でも利用できなければならない (MUST)
- **FR-010**: 利用者は、配信情報の共有先の一覧を確認・追加・削除できなければならない (MUST)。初期状態でも設定なしで動作する既定の共有先を持つべきである (SHOULD)。※ v1 ではこの SHOULD から逸脱し既定の共有先を同梱しない(同梱リストが事実上の中央集権点となるため。利用者は掲示板/SNS 等で発見した共有先 URL を貼り付けで一括登録する — 逸脱理由の記録は plan.md「SHOULD からの逸脱記録」参照)
- **FR-011**: チャンネルおよび発信者の識別体系は、将来のコメント(実況 BBS 代替)機能でも同一のペルソナを利用できるものでなければならない (MUST)
- **FR-013**: 利用者は複数のペルソナ(発信者識別子)を作成・切替・破棄できなければならない (MUST)。システムはペルソナ同士を技術的に紐づけてはならず (MUST NOT)、本人が明示的に公開しない限りペルソナ間の関連は第三者から推定できない設計でなければならない (MUST) — 匿名文化の維持(Clarifications 参照)。※ この保証はプロトコル上の情報(イベント・タグ・鍵)に限る。接続元 IP 等のネットワークメタデータ経由で共有先(リレー)運営者が推定しうるリンク可能性の限界と対策範囲は、脅威モデル ADR で定める
- **FR-012**: 掲載情報に含まれるリンクや内容について、悪意あるコンテンツの可能性を利用者に警告する仕組みを持たなければならない (MUST) — Principle I。v1 の警告対象はリンク(URL スキーム検査)に限定し、URL 以外の内容ベース警告の要否と範囲は脅威モデル ADR で確定する

### セキュリティシナリオ (Gherkin / Principle I, II, IV)

```gherkin
Feature: 分散型配信情報共有のセキュリティ

  # Principle II: trust nothing from the network / FR-007
  Scenario: 過大なペイロードの拒否
    Given ネットワークに悪意ある参加者が存在する
    When 上限を超えるサイズのチャンネル掲載データを受信した
    Then そのデータは拒否されなければならない
    And エラー応答は内部情報を漏洩してはならない
    And セキュリティイベントとしてログに記録されなければならない

  # Principle I: なりすまし防止 / FR-005
  Scenario: なりすまし掲載の検出
    Given 配信者Aのチャンネルが掲載されている
    When 第三者が配信者Aを騙る検証不能なチャンネル情報を流通させた
    Then 検証に失敗した情報は一覧に表示されてはならない
    And 検証失敗はセキュリティイベントとして記録されなければならない

  # Principle I / FR-008: Sybil・スパム緩和
  Scenario: 大量の偽チャンネル登録への耐性
    Given 悪意ある参加者が短時間に大量の偽チャンネルを掲載した
    When 視聴者がチャンネル一覧を取得する
    Then 利用者は緩和策(信頼リスト・非表示等)により正当なチャンネルを識別・閲覧し続けられる
```

### Key Entities

- **チャンネル(配信情報)**: 1 つの配信を表す。名称・ジャンル・説明・コンタクトURL・ビットレート・コンテンツ種別・曲情報・トラッカー接続先・リスナー数/リレー数・配信経過時間・掲載時刻を持つ。発信者(配信者識別子)に紐づき、検証可能である
- **ペルソナ(発信者識別子)**: 発信者を暗号学的に識別する識別子。チャンネルおよび将来のコメントの発信者検証に使われる。1 人の利用者は複数のペルソナを作成・切替・破棄でき、ペルソナ間はシステム上リンクされない。同一ペルソナ内でのみ継続性(なりすまし防止・信頼の積み上げ)が保たれる
- **共有先(中継点)**: 配信情報が掲載・購読される独立した中継点。複数が並立し、相互に代替可能。利用者が追加・削除できる
- **チャンネル一覧**: 視聴者が閲覧する、現在配信中のチャンネルの集合ビュー。鮮度(最終更新)に基づき自動的に除去・更新される
- **コメント(将来フェーズ)**: チャンネルに紐づく短文投稿。投稿者識別子により検証可能

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 配信者が配信を開始してから 60 秒以内に、他の参加者のチャンネル一覧に当該チャンネルが表示される
- **SC-002**: 複数の共有先のうち任意の 1 つを停止させても、掲載中の全チャンネルが引き続き発見・視聴可能である(利用者の操作変更なし)
- **SC-003**: 改造していない既存 PeerCast クライアントを使い、掲載から視聴開始までの一連の流れが完了できる
- **SC-004**: チャンネル一覧の取得・表示が通常時 5 秒以内に完了する
- **SC-005**: 発信者検証に失敗したなりすまし・改ざんエントリが一覧に表示される割合が 0% である
- **SC-006**: Windows 環境で、配布物の入手から、共有先(リレー)URL の貼り付け登録を含むチャンネル一覧の閲覧まで 15 分以内に完了できる
- **SC-007**: 不正な形式・過大サイズの受信データの 100% が拒否され、セキュリティイベントとして記録される

## Assumptions

- **プロトコル基盤の指定**: 依頼者の明示的な制約として、既存実績のある nostr の技術スタック(プロトコル定義 NIP)を援用する。どの NIP をどう適用するかは実装計画(plan)フェーズで決定する
- **既存クライアント連携**: 参考資料(PeerCast YP プロトコル解説)が提供されていることから、既存 PeerCast クライアント(PeerCastStation 等)は改造せず、本ソフトウェアが従来 YP と同じ接点(掲載・一覧・トラッカー解決)を提供する方式を前提とする
- **共有先の運用**: 共有先(中継点)は既存の公開運用のものを利用でき、コミュニティ有志が追加運用することも可能。特定の共有先の存続に依存しない(FR-002)
- **スコープ境界**: v1 は YP 代替(掲載・発見・トラッカー解決・継続性)まで。コメント(実況 BBS 代替)は将来フェーズとし、v1 では識別体系の互換性(FR-011)のみを保証する
- **対象プラットフォーム**: v1 の必須対象は Windows のみ。他 OS 対応は排除しないが成功基準に含めない
- **プライバシー**: 配信者のトラッカー接続先(IP 等)は現行 YP と同様に公開情報として扱う。詳細な取り扱い方針は constitution の Follow-up TODO(PRIVACY)に従い要件定義〜設計で ADR 化する
- **鍵(ペルソナ)の管理**: ペルソナの失効・ローテーションの詳細は設計フェーズの脅威モデル ADR で確定する
- **ライセンス**: constitution の暫定方針(許容的ライセンス想定、GPL 結合回避のため既存クライアントとはプロセス間連携に留める)に従う
