# Research: 分散型配信情報共有ネットワーク(YP代替)

**Date**: 2026-07-03 (rev 2) | **Plan**: [plan.md](./plan.md)

Technical Context の未確定事項をすべて解決した。各決定は「Decision / Rationale / Alternatives considered」で記録する。

**rev 2**: spec Clarifications 2026-07-03(リレー排除・スキーマ限定援用・純粋 P2P)に伴い、
R1〜R3・R8・R10〜R12 を改訂し、R13〜R16(gossip・PEX・UPnP・規模)を追加した。

## R1. チャンネル掲載に用いるイベントスキーマ

- **Decision**: NIP-53 Live Activities の **kind 30311(Live Streaming Event、addressable)** の
  データ構造を採用し、PeerCast 固有情報は追加タグで表現する。`d` タグ = チャンネル GUID(hex 32 桁小文字)。
  実況コメント(将来フェーズ)は同じ NIP-53 の **kind 1311(Live Chat Message)** を
  `a` タグ `30311:<pubkey>:<d>` で紐づける。
  **援用はデータ構造(直列化形式・タグ・署名)に限定**し、リレーへの掲載・購読は行わない(FR-014)。
  addressable イベントの置換規則 `(kind, pubkey, d)` は各ノードのローカルイベントストアで実装する。
- **Rationale**:
  - 30 秒〜60 秒周期の情報更新(リスナー数・曲情報)に対し、addressable の置換規則が
    「最新のみ保持」というローカルストアの要件をそのまま定義してくれる。
  - `status`(live/ended)、`starts`、`current_participants`、`title`、`summary`、`t`(ジャンル)など
    必要フィールドの大半が標準タグで揃う(2026-07-02 に NIP-53 原文で確認済み)。
  - 実績ある仕様の援用(依頼趣旨)を、スキーマ設計の再発明回避という形で満たす。
    イベントは標準準拠のため、将来リレーへのブリッジ(スコープ外)も原理的に可能。
  - YP 代替と BBS 代替が同一 NIP スキーマで完結し、FR-011(識別体系の互換)を自然に満たす。
- **Alternatives considered**:
  - 独自イベント形式の新規定義 — 署名・直列化・検証の設計をゼロから行うことになり、
    「1 からの再設計を避けたい」という依頼趣旨に反するため却下。
  - kind 1(テキストノート)への埋め込み — 置換規則がなく古い情報が堆積、構造化も貧弱なため却下。

## R2. 鮮度管理・自動除去(FR-006)

- **Decision**: 掲載側は配信中 60 秒ごとに 30311 を再発行し、**NIP-40 `expiration` タグ**
  (created_at + 600 秒)を付与。受信側は `status=live` かつ `created_at` が 600 秒以内の
  イベントのみを「配信中」と扱い、期限切れイベントはローカルストアから削除し**再伝搬しない**。
  配信終了時は `status=ended` で最終発行。
- **Rationale**: 鮮度判定・期限削除はすべて各ノードのローカル時計基準で完結する
  (リレー非依存 — 「真に信頼できるのは自分だけ」と整合)。期限切れの再伝搬禁止により、
  古い情報がネットワークを循環し続けることを防ぐ。時計ずれ(Edge Case)は 600 秒の余裕幅で吸収する。
- **Alternatives considered**: NIP-09 削除要求の伝搬 — 削除イベント自体の伝搬保証がなく、
  鮮度判定の代替にならない(補助として終了時の `status=ended` で十分)。

## R3. Rust nostr ライブラリ

- **Decision**: rust-nostr の **`nostr` クレート(プロトコル/データ構造層)** を採用。
  イベント構造・直列化・署名・検証(secp256k1 Schnorr)・NIP-13/19 実装に使用する。
  リレークライアント機能を持つ `nostr-sdk` は**使用しない**。
- **Rationale**: nostr の Rust 実装として最も成熟・活発。署名・検証をライブラリに委ねることで
  自前暗号実装の禁止(Principle II)を担保。`nostr` クレート単体はリレー通信を含まない
  データ構造層であり、スキーマ限定援用(FR-014)とモジュール境界が一致する。
- **Alternatives considered**: `secp256k1` + 手組み直列化 — イベント id 計算や署名検証の自作は
  バグ=脆弱性に直結するため却下(Principle II)。`nostr-sdk` 全体 — リレープール等の不要な
  依存とネットワーク経路を持ち込み、FR-014 の境界を曖昧にするため却下。

## R4. Web フレームワーク(UI・index.txt・ローカル API)

- **Decision**: **axum**(+ tower ミドルウェア)。UI 静的アセットは `include_dir` 等でバイナリに埋め込み。
- **Rationale**: tokio と同一エコシステム、tower によるレート制限・タイムアウト・ボディサイズ上限の
  ミドルウェア適用が容易(Security Requirements のレート制限に対応)。単一バイナリ制約と両立。
- **Alternatives considered**: actix-web(別ランタイム系で依存が重い)、tiny_http(ミドルウェア資産なし、
  自前実装が増える)— いずれも却下。

## R5. index.txt のエンコーディング

- **Decision**: 既定 **Shift_JIS(CP932、`encoding_rs`)** で出力し、設定で UTF-8 に切替可能とする。
  変換不能文字は HTML 数値参照ではなく `?` 置換とし、フィールド区切り `<>` を壊さないことを最優先。
- **Rationale**: 従来の日本語 YP エコシステム(YP ブラウザ)は Shift_JIS 前提のものが多い。
  互換性が最重要要件(FR-004)のため保守的な既定を選ぶ。
- **Alternatives considered**: UTF-8 固定 — 一部の既存 YP ブラウザで文字化けの恐れ。
- **Risk / follow-up**: 実際の YP ブラウザ(ユーザー所有)での表示確認を受け入れテストに含める。
  ここは実機確認までは仮説であることを明記しておく。

## R6. ペルソナ秘密鍵の保管

- **Decision**: **Windows DPAPI(`CryptProtectData`、`windows` クレート)** で暗号化した BLOB を
  SQLite に保存。ユーザー単位スコープ(`CRYPTPROTECT_LOCAL_MACHINE` は使わない)。
  UI からのエクスポート(nsec 表示)は明示操作+警告付きでのみ許可。
- **Rationale**: 追加ソフトウェア・追加パスフレーズなしで OS 標準の保護が得られ、
  「自己完結・インストール物を増やさない」というユーザー制約に合致。
- **Alternatives considered**:
  - `keyring` クレート(資格情報マネージャー)— 保存サイズ制限と列挙性の扱いが煩雑なため却下。
  - パスフレーズ派生鍵(argon2)— 起動毎の入力が UX を損なう。将来のオプションとしては排除しない。

## R7. 永続化ストア

- **Decision**: **rusqlite(bundled)** の単一 DB ファイル(`%APPDATA%\peca-p2p-yp\app.db`)。
  対象: ペルソナ、ピアリスト、ミュート、設定。発見チャンネル一覧・イベントストアはメモリ上のみ(揮発)。
- **Rationale**: 単一ファイル・自己完結・トランザクション。チャンネル一覧は本質的に
  エフェメラルであり永続化しない方が鮮度規則(R2)と整合する。ピアリストは再起動後の
  ブートストラップに必要なため永続化する(手動シードの再入力を強いない)。
- **Alternatives considered**: JSON/TOML ファイル群(排他制御と部分更新が脆弱)、sled(安定性・
  メンテ状況に不安)— 却下。

## R8. スパム・Sybil 緩和の初期構成(FR-008、脅威モデル ADR の前提)

- **Decision**: v1 は多層のクライアント側緩和で構成する:
  1. 署名検証必須(検証失敗は不可視 — FR-005)
  2. ペルソナ単位・チャンネル単位のミュート/非表示(ローカル保存のみ、公開はしない)
  3. ピアの追加・削除・無効化(汚染ピア・不良ピアの切り離し)
  4. **任意の NIP-13 PoW フィルタ**(既定 0 = 無効。閾値を上げると低コスト大量掲載を減衰。
     PoW を満たさないイベントは保持も再伝搬もしない)
  5. 受信レート・イベントサイズ・イベントストア容量の上限(1 ピアあたり、全体)
- **Rationale**: オープン型既定(spec Clarifications)を守りつつ、利用者側の自衛手段を積層する。
  PoW は身元登録を要求しないため匿名文化(ペルソナモデル)と両立する唯一のコスト型対策。
  純粋 P2P では自ノードが再伝搬の担い手になるため、「保持しない=拡散に加担しない」という
  フィルタの二重効果がある。
- **Alternatives considered**: Web of Trust 必須化・招待制 — 既定非表示の禁止(FR-008 MUST NOT)に
  抵触するため却下。具体的な閾値・追加手法は脅威モデル ADR(tasks 先頭フェーズ)で確定。

## R9. PCP プロトコル実装の方針とライセンス

- **Decision**: 参考資料 gist(PeerCastStation の YP プロトコル解説)に基づき、
  **プロトコルの事実(atom 構造・ハンドシェイク手順)のみを用いたクリーンルーム実装**を行う。
  PeerCastStation(GPLv3)のソースコードは参照・複製しない。連携はプロセス間 TCP のみ。
- **Rationale**: constitution のライセンス暫定方針(許容的ライセンス想定、GPL 結合回避)に従う。
  プロトコル仕様という事実の利用は結合著作物を構成しない。
- **Alternatives considered**: PeerCastStation へのプラグイン追加 — GPL 結合となり方針に反するため却下。

## R10. 初期シードピアの入手経路

- **Decision**: 既定シードピアは**同梱しない**。UI に「ピアアドレスの貼り付け一括登録」
  (`host:port` を 1 行 1 件)と「現在のピアリストのテキスト書き出し」を用意し、
  掲示板/SNS でのリスト流通を前提とする。接続確立後はピア交換(R14)で自動拡大する。
- **Rationale**: ユーザー指示。同梱リストは事実上の中央集権点となり本機能の動機と矛盾する。
  spec FR-010 の SHOULD からの逸脱として plan.md に理由を記録済み(constitution 準拠)。
- **Alternatives considered**: 既知ピアの同梱、DNS シード — 却下(上記と同根。DNS シードは
  ドメイン運営者という単一運営点を再導入する)。

## R11. BDD テスト基盤

- **Decision**: **`cucumber` クレート**で `tests/features/*.feature` を実行。spec.md の
  受け入れ・セキュリティシナリオと 1:1 対応させ、実装前に失敗を確認する(Principle IV)。
  統合層はインプロセスの**モックピア(gossip プロトコルのテスト用実装)**と PCP 疑似クライアントで
  構成し、3〜10 ノードの多ノード伝搬・切断・PEX シナリオをプロセス内で再現する。
- **Rationale**: Rust ネイティブで Gherkin を実行できる唯一の成熟ライブラリ。外部ランタイム不要。
  gossip は自前プロトコルであるため、モック実装が同時に「プロトコル契約の参照実装」として機能する。
- **Alternatives considered**: Gherkin を doc として扱い通常テストで代替 — シナリオとテストの
  乖離が生じやすく Principle IV の MUST(対応付け)を満たしにくいため却下。

## R12. 待受ポートとバインド

- **Decision**: 既定値 — PCP アナウンス: `127.0.0.1:7146`、HTTP(UI + index.txt): `127.0.0.1:7180`、
  **P2P gossip: `0.0.0.0:7147`**(唯一の外部露出ポート。設定で変更・無効化可。
  無効時は外向き接続のみで参加 — FR-016)。LAN/外部公開が既定なのは P2P のみで、
  PCP/HTTP の LAN 公開は明示的なオプトイン+UI 警告。
- **Rationale**: PeerCastStation の既定 7144(および多重起動時の 7145)との衝突を回避。
  P2P 待受は着信可能ノードを増やし網全体の維持に寄与するため既定有効とするが、
  受信データは contracts/p2p-gossip.md の多段検証を必ず通す(Principle II)。
  PCP/HTTP の loopback 既定は最小権限の原則に基づく攻撃面の最小化。
- **Alternatives considered**: P2P も loopback 既定 — 着信可能ノードが増えず、
  外向き接続のみのノードばかりでは網が成立しないため却下(FR-016 の前提と整合)。

## R13. P2P トランスポートと gossip 方式

- **Decision**: **TCP + 長さ前置(4 バイト BE)JSON フレーム**の自前プロトコル。
  メッセージ種別: `HELLO`/`HELLO_ACK`(バージョン・待受ポート・機能フラグ)、
  `EVENT`(nostr イベント JSON を格納)、`SYNC_REQ`/`SYNC_DONE`(接続時の差分同期)、
  `GET_PEERS`/`PEERS`(ピア交換 — R14)、`PING`/`PONG`。
  伝搬は **eager-push フラッディング + イベント id による重複抑制**(TTL 的なホップ制限は
  設けず、重複抑制と鮮度期限(R2)で終端を保証)。接続確立時は `SYNC_REQ` で相手の保持する
  live イベント一覧を取得して初期一覧を構築する。トランスポート暗号化は行わない
  (完全性・真正性はイベント署名で担保。掲載情報は公開データであり機密性要件がない —
  この判断は ADR に記録)。
- **Rationale**:
  - 想定規模(5,000 ノード・2,000 チャンネル・60 秒再発行)での網全体イベント率は
    最大 ~33 イベント/秒。接続度 8 のフラッディングでもピアリンクあたり最大 ~50KB/s、
    典型時(~100 チャンネル)は ~2.5KB/s であり、最適化(遅延プッシュ・ダイジェスト交換)
    なしで成立する(R16)。
  - JSON フレームは nostr イベントの直列化形式(JSON)と一致し、変換層が不要。
    検証パイプライン(サイズ→形式→署名)を単純に保てる(Principle II)。
  - 自前プロトコルの新規設計であることは認識しており、伝搬・重複抑制・同期の状態機械は
    **Principle V の形式的検証候補**として ADR で判定する(plan.md Constitution Check 参照)。
- **Alternatives considered**:
  - **libp2p(gossipsub)** — 実績ある gossip 実装だが、依存グラフが非常に大きく監査面積が増大、
    Noise 暗号化・マルチプレクサ等の不要機能が付属し「読みやすく保守可能(Principle III)」と
    単一バイナリの軽量性を損なうため却下。設計上の概念(mesh・IHAVE/IWANT)は将来の
    最適化の参考として採用余地を残す。
  - **WebSocket + nostr リレーメッセージ(REQ/EVENT/EOSE)の流用** — リレー/クライアントの
    非対称な役割分担を前提とした語彙であり、対等なピア間通信への流用は意味論の歪みを生む。
    spec Clarifications(スキーマのみ援用、伝送は独自)の決定にも反するため却下。
  - **UDP ベース独自プロトコル** — 再送・輻輳制御の自作が必要になり複雑化するため却下。
    TCP で要件(60 秒以内伝搬)は十分満たせる。

## R14. ピア交換(PEX — FR-015)

- **Decision**: `GET_PEERS` 要求に対し、**最近直接通信に成功した着信可能ピア**の
  `host:port` を最大 64 件返す(`PEERS`)。受信側は既知リストに候補として追加するが、
  **自ら接続に成功するまで「検証済み」とせず、未検証ピアを他ノードへ再共有しない**。
  ピアリスト容量は上限 1,024 件(LRU)。ピアの信頼スコアは v1 では持たず、
  接続成功/失敗の履歴(last_ok_at / fail_count)のみ記録し、失敗超過で候補から降格する。
- **Rationale**: 「真に信頼できるのは自分だけ」— 他ノードの申告(PEX)は participation の
  ヒントに過ぎず、検証(実接続)を経てはじめて自分の資産になる。未検証情報の再共有禁止は
  毒入れ(存在しないアドレス・第三者への反射攻撃)の増幅を防ぐ(Principle II)。
- **Alternatives considered**: 署名付きピアレコード(自己証明アドレス)— なりすまし防止に
  有効だが、IP アドレスの真正性は署名では保証できず(署名者が虚偽アドレスを署名可能)、
  実接続検証の代替にならないため v1 では見送り(脅威モデル ADR で再評価可)。

## R15. UPnP ポートマッピング(FR-016)

- **Decision**: **`igd-next` クレート**(async)で起動時に P2P 待受ポートの UPnP IGD
  マッピングを試行(lease 付き、定期更新)。失敗時は警告なしで**外向き接続のみモード**へ
  フォールバックし、UI の状態表示に着信可否を表示する。UPnP は設定で無効化可。
  TURN 等の中継サーバー型 NAT 越えは実装しない(spec Clarifications / FR-016)。
- **Rationale**: 家庭用ルーターの多くで追加設定なしに着信可能ノードを増やせ、網の維持
  (R12 Rationale)に寄与する。失敗しても全機能が成立する設計(FR-016)のため、
  UPnP は best-effort でよい。
- **Alternatives considered**: NAT-PMP/PCP(Port Control Protocol)併用 — 対応ルーターが
  増えるが v1 では UPnP のみとし、必要になれば追加(YAGNI)。手動ポートフォワード案内は
  ドキュメントで提供。

## R16. 規模想定と容量設計(SC-008)

- **Decision**: 設計目標を **同時 5,000 ノード・2,000 live チャンネル**とする。導出値:
  - 網全体イベント率: 2,000 ch ÷ 60 秒 ≈ 33 イベント/秒(+コメントは将来フェーズで別途)
  - イベントサイズ上限 16KB、典型 ~1.5KB → リンクあたり典型 ~50KB/s(最悪時)、~2.5KB/s(典型 100ch)
  - イベントストア上限: 4,096 イベント(placement: 2,000 live + 余裕)、超過時は古い順に破棄
  - 重複抑制キャッシュ: 直近 10 分の event id(~20,000 件、メモリ ~2MB)
  - ピア接続: 外向き目標 8 / 着信上限 32 — 接続度 8 のランダムグラフは 5,000 ノードで
    直径 ~4–5 となり、hop あたり伝搬遅延 <1 秒として SC-001(60 秒)に十分収まる
- **Rationale**: spec Clarifications 2026-07-03(最大 5,000 ノード)に基づく。全定数は
  Settings(data-model.md)で調整可能とし、実測との乖離は beta で補正する。
- **Alternatives considered**: 構造化オーバーレイ(Kademlia 等)による購読最適化 —
  5,000 ノード規模ではフラッディングで足りるため、複雑性に見合わず却下(YAGNI)。
