# Feature Specification: 読み取り専用 index.txt の LAN 公開(オプトイン)

**Feature Branch**: `feature/lan-index-txt`

**Created**: 2026-07-08

**Status**: Draft

**Input**: User description: "別 PC の YP ブラウザから自宅サーバー上の本ソフトの index.txt を取得するための LAN 公開機能(別セッションの grill-me で確定した 7 分岐の設計判断に基づく)"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 別 PC の YP ブラウザから掲載一覧を取得する (Priority: P1)

利用者は、自宅サーバー(常時稼働機)で本ソフトを動かし、手元の別 PC の YP ブラウザから
`http://<サーバーの LAN アドレス>:<ポート>/index.txt` を直接開いて掲載一覧を閲覧したい。
現状は loopback 強制のため、SSH ポート転送(`-L 7180:127.0.0.1:7180`)などの回避策を
自前で構成する必要がある。新しい設定項目 `index_bind`(既定は空 = 無効)に LAN アドレスを
指定すると、index.txt 配信専用の受け口が追加で起動し、この回避策が不要になる。

**Why this priority**: 本機能の中核価値そのもの。これが動かなければ機能が存在しない。

**Independent Test**: サーバー機で `index_bind` に LAN アドレスを設定して起動し、
LAN 内の別 PC から index.txt を取得して同一ホスト取得分と比較するだけで単独検証できる。

**検証する原則**: Principle II(オプトイン・既定無効による最小権限)。レート制限の維持は
Security Requirements(レート制限)。

**Acceptance Scenarios**:

1. **Given** `index_bind` に LAN のプライベートアドレスが設定され再起動済み、**When** LAN 内の別 PC の YP ブラウザが `http://<LAN アドレス>:<ポート>/index.txt` を GET する、**Then** 同一ホスト(loopback)で取得した場合と同一内容・同一エンコーディングの掲載一覧が返る。
2. **Given** 同じ状態、**When** 別 PC が同 URL へ HEAD リクエストを送る、**Then** 既存の index.txt 配信と同じ規則で GET と整合する応答が返る。
3. **Given** `index_bind` が未設定(既定)、**When** LAN 内の別 PC がサーバーの任意ポートへ index.txt 取得を試みる、**Then** 本ソフトは LAN 上で index.txt を一切提供しない(現行リリースと同一の挙動)。
4. **Given** LAN 公開が有効、**When** 同一の送信元から既存レート制限(毎秒 10 リクエスト)を超える取得が行われる、**Then** 超過分は loopback 側と同じ規則で拒否される(ネガティブシナリオ)。

---

### User Story 2 - index.txt 以外は LAN へ一切露出しない (Priority: P1)

利用者は、掲載一覧を LAN に公開しても、管理 UI・設定 API・掲載操作・PCP 受け口などの
他の機能が LAN からは従来どおり一切利用できないことを保証されたい。LAN 公開の受け口は
index.txt の読み取り専用配信だけを持ち、他のパスは定型の 404 を返す。

**Why this priority**: 攻撃面の限定こそが本機能の安全性の根拠であり、User Story 1 と対で
初めて「安全な LAN 公開」が成立する。これが破れると ADR-0006 決定 4 の緩和自体が成立しない。

**Independent Test**: LAN 公開を有効にした状態で、別 PC から管理 UI・API 各パス・
書き込み系メソッドへのアクセスを試み、すべて拒否されることを確認するだけで単独検証できる。

**検証する原則**: Principle II(最小権限・攻撃面最小化)、Principle IV(ネガティブシナリオの
明示)。エラーが内部情報を漏らさないことは Security Requirements(エラーハンドリング)。

**Acceptance Scenarios**:

1. **Given** LAN 公開が有効、**When** 別 PC から設定 API(`/api/v1` 配下の任意パス)へアクセスする、**Then** 定型の 404 が返り、内部実装の詳細は漏洩しない(ネガティブシナリオ)。
2. **Given** LAN 公開が有効、**When** 別 PC から管理 UI(トップページや静的ファイル)へアクセスする、**Then** 定型の 404 が返る(ネガティブシナリオ)。
3. **Given** LAN 公開が有効、**When** 別 PC から index.txt の URL へ GET/HEAD 以外のメソッドを送る、**Then** 拒否され、掲載内容は一切変更されない(ネガティブシナリオ)。
4. **Given** LAN 公開が有効、**When** 別 PC から管理用 HTTP 受け口や PCP 受け口へ接続を試みる、**Then** 従来どおり loopback 強制のまま到達できない(現行の保護が不変であること)。

---

### User Story 3 - 危険性を理解した上で有効化し、露出状態を常に自覚できる (Priority: P2)

利用者は、LAN 公開を有効化する際にそのリスク(掲載一覧が LAN 内で平文・無認証のまま
取得・改ざんされうること)を明示的に確認し、有効化後は本ソフトの状態表示から
「いま LAN へ露出している」ことをいつでも確認できたい。

**Why this priority**: 誤設定・無自覚な露出の防止。機能自体(US1/US2)が成立した上で、
利用者の判断と自覚を支える安全ゲート。

**Independent Test**: 設定 UI で非 loopback 値の保存を試み、警告と明示確認なしには
保存できないこと、有効起動後にセキュリティイベントの記録と状態表示への反映を確認する。

**検証する原則**: Principle I(誤った安全性の期待を作らない・無自覚な露出の防止)、
Principle II(危険な設定値の構造的拒否)。イベント記録は Security Requirements
(セキュリティログ)。

**Acceptance Scenarios**:

1. **Given** 設定 UI を開いている、**When** `index_bind` に非 loopback 値を入力して保存しようとする、**Then** 「掲載一覧が LAN 内で平文・無認証のまま取得・改ざんされうる」旨の警告 1 項目が提示され、明示的な確認操作なしには保存が完了しない。
2. **Given** 警告を明示確認して保存した、**When** 保存が完了する、**Then** 反映には再起動が必要である旨が伝えられ、再起動までは従来の受け口構成のまま動作する。
3. **Given** `index_bind` に LAN アドレスが設定された状態、**When** 本ソフトを起動する、**Then** LAN 露出中であることを示すセキュリティイベント(カテゴリ `index_txt_lan_exposed`)が 1 件記録され、状態表示(`GET /api/v1/status`)で露出状態(バインド先を含む)を確認できる。
4. **Given** 設定 UI または CLI、**When** `index_bind` に未指定アドレス(`0.0.0.0` / `::`)、グローバルユニキャストアドレス、またはカンマ区切りの複数アドレスを指定する、**Then** 検証で定型エラーとして拒否され、設定は適用されない(ネガティブシナリオ)。
5. **Given** `index_bind` が未設定または loopback 値、**When** 起動する、**Then** LAN 露出のセキュリティイベントは記録されず、状態表示は非露出を示す。

### User Story 4 - LAN 公開に失敗しても本体は止まらない (Priority: P3)

利用者は、LAN 公開の受け口が起動に失敗しても(ポート競合・存在しないアドレスの指定など)、
本体機能(同一ホストでの管理 UI・API・PCP・P2P)が従来どおり稼働し続けることを期待する。
LAN 公開は付加機能であり、その失敗が主機能を巻き込んではならない。

**Why this priority**: 可用性の保護。US1〜US3 が成立した後の運用品質であり、
機能価値の中核ではないが、常時稼働サーバー用途では重要。

**Independent Test**: `index_bind` に競合するポートや存在しないインターフェースの
アドレスを与えて起動し、本体機能の継続稼働と状態表示への失敗理由の反映を確認する。

**検証する原則**: Principle I(可用性の保護 — 付加機能の失敗で主機能を失わない)。

**Acceptance Scenarios**:

1. **Given** `index_bind` のポートが他プロセスに使用されている、**When** 起動する、**Then** 本体機能(loopback の UI/API、PCP、P2P)は正常稼働し、警告ログが記録され、状態表示に LAN 公開の失敗と理由が示される。
2. **Given** `index_bind` にそのホストに存在しないアドレスが設定されている、**When** 起動する、**Then** 同様に本体は継続稼働し、失敗が状態表示から確認できる。
3. **Given** LAN 公開の受け口が失敗した状態、**When** 状態表示を参照する、**Then** 「有効に設定されているが起動に失敗している」ことが「無効」とも「露出中」とも区別して読み取れる。

---

### Edge Cases

- **`index_bind` が既存の管理用受け口と同一(例: `127.0.0.1:7180`)**: loopback 値は検証上は受理されるが、バインド競合により LAN 受け口の起動は失敗する。US4 の縮退継続として扱い、状態表示に失敗理由が示される。
- **IPv6 リンクローカル(`fe80::/10`)でゾーン識別子が欠落・不正**: 書式として解釈可能なら検証は通過し、バインド失敗は US4 の縮退継続として扱う。
- **カンマ区切りなど複数アドレスの指定**: 単一のソケットアドレスのみ受理し、複数指定は検証エラーとして拒否する(複数対応は将来拡張)。
- **ポート欠落・書式不正**: 既存のバインド設定と同じ規則で検証エラーとして拒否する。
- **実行中の `index_bind` 変更**: 保存はできるが再起動まで反映されない(US3 シナリオ 2)。再起動までは変更前の受け口構成が維持される。
- **CGNAT / 共有アドレス空間(`100.64.0.0/10`)の指定**: 検証エラーとして拒否する。露出先が物理 LAN を超えて VPN メッシュ全体に広がりうるため、v1 の受理範囲(private/LAN 限定)には含めない。Tailscale 等の利用者は VPN 側の機能(subnet router 等)で到達でき、解禁が必要になれば将来の ADR 改訂で扱う。
- **Windows で初回の非 loopback バインド**: OS のファイアウォール確認プロンプトが出る。本機能の要件外だが、導入手順書(quickstart)に案内を明記する。
- **レート制限・サイズ上限の一貫性**: URL・ヘッダーのサイズ上限と送信元ごとのレート制限は、loopback 側と LAN 側で同一の規則が適用される(緩和されない)。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムは新しい設定項目 `index_bind`(既定値は空 = 機能無効)を提供し、非空の場合のみ index.txt 配信専用の受け口を追加で起動しなければならない (MUST)。既存のバインド設定(`http_bind` / `pcp_bind` / `p2p_bind`)の意味・検証・既定値は一切変更してはならない (MUST NOT)。
- **FR-002**: 専用受け口は index.txt の GET/HEAD のみを提供し、それ以外のすべてのパス・メソッドには内部情報を漏らさない定型エラーを返さなければならない (MUST)。既存の index.txt 配信に適用されているサイズ上限および送信元ごとのレート制限(毎秒 10 リクエスト)を専用受け口にも同等に適用しなければならない (MUST)。
- **FR-003**: `index_bind` の値は loopback、プライベートアドレス(RFC 1918)、リンクローカル(IPv4 `169.254.0.0/16` / IPv6 `fe80::/10`)、IPv6 ULA(`fc00::/7`)のみ受理しなければならない (MUST)。未指定アドレス(`0.0.0.0` / `::`)、グローバルユニキャストアドレス、および CGNAT / 共有アドレス空間(`100.64.0.0/10` — Tailscale 等のメッシュ VPN が付与)は検証で拒否しなければならない (MUST)。
- **FR-004**: `index_bind` は CLI 引数と設定 UI の両方から設定できなければならない (MUST)。実行中の変更は既存の設定変更と同様に「再起動が必要」として扱い、再起動までは反映してはならない (MUST NOT)。
- **FR-005**: 設定 UI で非 loopback 値を設定する際は、「掲載一覧が LAN 内で平文・無認証のまま取得・改ざんされうる」旨の警告 1 項目を提示し、明示的な確認操作なしに保存を完了してはならない (MUST NOT)。
- **FR-006**: `index_bind` が非 loopback 値で受け口の起動に成功した場合、起動時にセキュリティイベント(新カテゴリ `index_txt_lan_exposed`)を 1 件記録しなければならない (MUST)。また状態表示(`GET /api/v1/status`)で LAN 露出の有無・バインド先・失敗時の理由を確認できなければならない (MUST)。
- **FR-007**: 専用受け口のバインド失敗は致命エラーとせず、警告ログの記録と状態表示への失敗理由の反映を行った上で、本体機能を継続稼働させなければならない (MUST)。既存 3 受け口(管理 HTTP / PCP / P2P)のバインド失敗時の即時終了は変更してはならない (MUST NOT)。
- **FR-008**: LAN 側で配信される index.txt は、loopback 側と同一の内容・同一のエンコーディング設定(`index_txt_encoding` を共有)でなければならない (MUST)。
- **FR-009**: `index_bind` は単一のソケットアドレスのみ受理し、複数アドレスの指定は検証エラーとして拒否しなければならない (MUST)。複数対応は本仕様の範囲外(将来拡張)とする。

### Key Entities

- **設定項目 `index_bind`**: LAN 公開の受け口アドレス。既定は空(無効)。検証済みの private/LAN/loopback アドレス + ポートのみ保持しうる。
- **index.txt 専用受け口**: `index_bind` が非空のときだけ存在する追加の待ち受け。提供機能は index.txt の読み取り(GET/HEAD)のみ。
- **セキュリティイベント `index_txt_lan_exposed`**: 「LAN へ露出中」という運用状態の監査記録。既存カテゴリの多く(違反を弾いた記録)と異なり、利用者が選んだ状態の記録である。
- **露出状態(状態表示)**: 無効 / 露出中(バインド先) / 有効設定だが起動失敗(理由)の 3 状態を区別して提示する。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: `index_bind` 未設定のとき、外部から観測できる振る舞い(全受け口の応答・状態表示・セキュリティイベント)が現行リリースと完全に一致する(回帰ゼロ)。
- **SC-002**: LAN 内の別 PC の YP ブラウザが、SSH ポート転送などの追加構成なしで掲載一覧を取得でき、その内容が同一ホストで取得した掲載一覧と 100% 一致する。
- **SC-003**: LAN 公開が有効な状態で、別 PC からの掲載一覧以外へのアクセス試行(管理 UI・API・書き込み系メソッド・PCP)がすべて拒否される(到達成功率 0%)。
- **SC-004**: インターネット全体へ露出しうる設定値(未指定アドレス・グローバルアドレス)の投入試行が、保存時または起動時の検証で 100% 拒否される。
- **SC-005**: LAN 受け口の起動失敗を注入したすべてのケースで本体機能が継続稼働し、利用者が状態表示から失敗の事実と理由を確認できる。
- **SC-006**: 設定 UI からの非 loopback 値の有効化は、警告への明示確認なしには 1 件も完了しない。

## Assumptions

- **ADR の扱い**: 本機能は ADR-0006 決定 4(LAN 公開オプトイン v1 非実装)を「読み取り専用 index.txt に限り」部分的に supersede する。新規 ADR の起草(決定 4 の将来解禁条件との対応表を含む)と ADR-0006 への追記は、plan フェーズの実装前ゲートとして行う(Principle II/VI)。
- **警告 1 項目の根拠**: contracts/local-api.md §保護方針の LAN 公開警告 2 項目のうち、(2) トークン盗聴は index.txt 配信(トークン不要・読み取り専用)には該当しないため、(1) 攻撃面拡大に相当する 1 項目のみを課す。この整理は新規 ADR に記録する。
- **CLI / API 直接呼び出しからの設定**: `--index-bind` フラグの明示指定、および `PUT /api/v1/settings` の直接呼び出し(自動化クライアント)による明示指定は、それ自体をオプトインの同意とみなし、対話的な確認・警告確認は課さない(検証規則は UI と同一)。FR-005 の警告ゲートは設定 UI 層の責務である。
- **再起動要求の応答形式**: 既存の「再起動が必要な設定変更」と同じ形式に従う。具体形は plan で確定する。
- **警告 UI の具体形**(チェックボックス / 確認ダイアログ等)と **状態表示のフィールド構成**は plan / 実装で確定する。仕様として拘束するのは「明示確認なしに保存不可」「3 状態が区別可能」のみ。
- **OS 固有の案内**(Windows ファイアウォールプロンプト、Linux systemd での LAN バインド例)は quickstart 文書側で扱う。
- **index.txt の内容仕様**は specs/001-nostr-p2p-yp/contracts/http-yp.md(19 フィールド・サニタイズ・エンコーディング)を正とし、本仕様では変更しない。
