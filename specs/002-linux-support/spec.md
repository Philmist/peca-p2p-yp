# Feature Specification: Linux 対応(常時稼働ノード・systemd 親和)

**Feature Branch**: `002-linux-support`

**Created**: 2026-07-06

**Status**: Draft

**Input**: User description: "このアプリケーションはDPAPIを使用しているためWindowsでしか動かないが、常に有効なノードとして動かすにはLinuxのほうが有用であるからLinux上でも動作させる。可能であるならsystemdとも親和性が高い形で実装する。"

## Clarifications

### Session 2026-07-06

- Q: Linux でのペルソナ秘密鍵の「保護鍵」の由来・保管方式は? → A: 乱数マスター鍵ファイル方式(初回起動時に乱数生成した保護鍵を実行アカウントのディレクトリに `0600` で保管し、その鍵で各ペルソナ秘密鍵を AEAD 暗号化して `secret_enc` に格納。DPAPI のユーザースコープ保護に対応する自己完結モデル)
- Q: 保管ファイル(保護鍵/DB)のパーミッションが緩い状態で起動したときの挙動は? → A: 自動是正(検知したら `0600` 相当へ是正して継続し、セキュリティイベントを記録。是正不能な場合は警告して当該ペルソナを利用不可にする)
- Q: 全ペルソナ共有の保管物(保護鍵ファイル・DB)が是正不能な場合の影響範囲は? → A: 影響を受けるすべてのペルソナを利用不可とし、発見・伝搬機能(US1)は継続する
- Q: SC-004 の「所定の停止タイムアウト」の基準値は? → A: systemd 既定の停止タイムアウト(90 秒)以内とする
- Q: 同一ホストでの複数インスタンス稼働は要件か? → A: データディレクトリと待受ポートの個別指定により両立できるべき(SHOULD)として FR-010 に含める

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Linux 上で利用者ノードとして稼働できる (Priority: P1)

Linux サーバーを運用する利用者が、本ソフトウェアを Linux 上で導入・起動し、P2P
ネットワークに利用者ノードとして参加できる。ペルソナを作成・掲載しない「発見・伝搬に
寄与する常時稼働ノード」として、gossip への参加・チャンネル発見・index.txt 提供が
Windows 版と同じように機能する。

**Why this priority**: 「常に有効なノードとして動かすには Linux のほうが有用」という
本機能の動機の中核。Windows 専用機能(DPAPI 等)への直接依存でビルド/起動が失敗する状態を
解消し、Linux 上での基本稼働を成立させることが、他のすべての価値の前提となる MVP。

**Independent Test**: Linux 環境でビルド・起動し、既知ピアへ接続して他ノードが掲載した
チャンネルが一覧に現れること、ローカル HTTP から index.txt を取得できることで検証できる。
ペルソナ鍵管理(US2)や systemd 統合(US3)を実装しなくても単独で価値を提供する。

**Acceptance Scenarios** (Gherkin / Principle IV):

1. **Given** Linux ホストと到達可能な既知ピア、**When** 利用者がノードを起動する、**Then**
   Windows 専用 API への依存で失敗することなく起動し、gossip セッションを確立して
   ネットワークに参加する
2. **Given** 稼働中の Linux ノード、**When** 他ノードが有効なチャンネルを掲載する、**Then**
   多段検証を経て当該チャンネルが自ノードの一覧・index.txt に現れる
3. **Given** 稼働中の Linux ノード、**When** ローカル HTTP エンドポイントへ要求する、**Then**
   Windows 版と同一の応答(index.txt・ローカル API)が得られる
4. **Given** バインド系設定、**When** 起動する、**Then** loopback 強制などの既存の露出方針
   (ADR-0006)が Linux でも同一に適用される

---

### User Story 2 - Linux 上でペルソナ秘密鍵を安全に保管して掲載できる (Priority: P2)

Linux ノードの運用者が、ペルソナ(nostr 鍵ペア)を作成・保管・切替・破棄し、チャンネルを
掲載できる。DPAPI が存在しない Linux でも、秘密鍵は平文で保存されず、同一ホストの他の
利用者アカウントから復号できない水準で at-rest 保護される。ヘッドレス(デスクトップ
セッションなし)の常時稼働ホストでも、対話操作を追加せずにこの保護が成立する。

**Why this priority**: 掲載(発信)機能を Linux で使うために必要。DPAPI 相当の保護が
Linux に無いことが Windows 専用となっている根本原因であり、本機能の技術的核心。ただし
US1(発見・伝搬ノード)は掲載を伴わず単独で価値を出せるため、P1 とは分離する。

**Independent Test**: Linux 上でペルソナを作成し、保管された秘密鍵表現が平文でないこと・
他ユーザーから読み取れないことを確認し、当該ペルソナで掲載したチャンネルが他ノードで
発見されること、nsec エクスポートと破棄が Windows 版と同一の意味論で動くことで検証できる。

**Acceptance Scenarios** (Gherkin / Principle IV):

1. **Given** Linux ノード、**When** 利用者がペルソナを新規作成する、**Then** 秘密鍵は
   平文で永続化されず、保護された表現として保存される
2. **Given** 保管されたペルソナ、**When** 同一ホストの別の利用者アカウントがその保管データを
   読もうとする、**Then** 秘密鍵を復元できない
3. **Given** ペルソナ、**When** nsec エクスポートまたは破棄を要求する、**Then** Windows 版と
   同一の確認要件・非可逆性・ログ非記録の意味論(ADR-0003)で処理される
4. **Given** 保管データが復号できない状態(破損・別アカウント作成・OS 再インストール後・
   他プラットフォームからの持込)、**When** 起動する、**Then** 当該ペルソナのみ「利用不可」と
   明示し、起動と他ペルソナ・発見機能は継続する(ADR-0003 §4 と同挙動)

---

### User Story 3 - systemd サービスとして常時稼働に適した振る舞い (Priority: P3)

運用者が、本ソフトウェアを systemd サービスとして登録し、ホスト起動時の自動開始・異常時の
自動再起動・停止要求時の安全な終了を得る。サービスは提供された unit 定義例で登録でき、
ログはジャーナルで確認でき、設定・状態・ログの配置は Linux/systemd の規約に従う。

**Why this priority**: 「systemd とも親和性が高い形で」という付帯要望。常時稼働の運用性を
大きく高めるが、US1/US2 が動けば手動起動でも常時稼働は可能なため、最優先ではない。

**Independent Test**: 提供 unit 例でサービスを登録し、`systemctl start/stop/status` で
起動・状態確認・安全な停止ができること、停止要求で進行中の gossip セッションを乱さずに
終了すること、ログがジャーナルに出ることで検証できる。

**Acceptance Scenarios** (Gherkin / Principle IV):

1. **Given** systemd と unit 定義、**When** サービスを開始する、**Then** ノードが起動し、
   サービスが「稼働中/準備完了」として扱われる
2. **Given** 稼働中のサービス、**When** systemd が停止を要求する(SIGTERM 相当)、**Then**
   ノードは既存の graceful shutdown 経路で安全に終了し、サービスは正常停止として記録される
3. **Given** サービス実行環境、**When** 設定・状態・ログを配置/出力する、**Then** systemd/
   Linux の標準ディレクトリ規約に沿った場所に置かれ、ジャーナルで参照できる
4. **Given** 異常終了、**When** systemd の再起動方針が有効、**Then** サービスが自動的に
   復帰する

---

### Edge Cases

- ヘッドレスホスト(GUI・デスクトップログインセッションなし)で、デスクトップ用の鍵保管
  デーモン(Secret Service 系)が起動していない場合でも、無人でノードが起動・稼働できるか
- Linux で作成した保管データを別の利用者アカウント・別ホスト・別プラットフォーム(Windows)へ
  持ち込んだ場合の挙動(復号不能 → 当該ペルソナのみ利用不可・全体は継続)
- 秘密鍵の保管ファイル/データのパーミッションが緩い(他ユーザー可読)状態で起動した場合、
  `0600` 相当へ自動是正して継続(是正不能なら警告し、影響ペルソナ — 共有保管物なら全ペルソナ — を
  利用不可とし、発見・伝搬機能は継続)— FR-013
- systemd の停止タイムアウトを超えて終了が長引いた場合(強制終了と graceful shutdown の関係)
- 特権ポートや権限不足でバインドに失敗した場合の、原因が分かる終了・ログ — FR-014
- systemd 由来の実行時ディレクトリ(状態/設定/ログ)指定と、手動起動時の既定配置の両立
- 同一ホストで複数インスタンス(複数サービス)を動かす場合のデータ・ポートの分離 — FR-010

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムは単一のコードベースから Windows と Linux の両方でビルド・稼働できな
  ければならない (MUST)。Windows 専用機能への直接依存によって Linux でのビルドまたは起動が
  失敗してはならない (MUST NOT)
- **FR-002**: Linux 上の稼働時、gossip 参加・チャンネル発見・多段検証・index.txt 生成・
  ローカル API・信頼境界・露出方針(loopback 強制 — ADR-0006)は Windows 版と機能的に同等で
  なければならない (MUST) — Principle II
- **FR-003**: Linux 上でペルソナ秘密鍵を平文で永続化してはならない (MUST NOT)。保管は
  ファイルベース暗号化 — 初回起動時に乱数生成した保護鍵(マスター鍵)を実行アカウントの
  ディレクトリに制限パーミッション(`0600` 相当)で保管し、その鍵で各ペルソナ秘密鍵を AEAD
  暗号化した表現を `secret_enc` に格納する。追加のインストール物・常駐サービス(デスクトップ
  Secret Service 等)を要求しない — によらなければならない (MUST)。保護鍵は DB(`secret_enc`)とは
  別ファイルに置き、DB 単体の持出しでは復号できないこと (MUST)。パスフレーズ派生鍵(argon2 等)は
  将来の任意オプションとしてマスター鍵をラップする形で排除しない(ADR-0003 §1 の Windows 側方針と整合)
  — Principle I, II
- **FR-004**: 秘密鍵の保管は実行アカウント(利用者/サービスアカウント)スコープであり、
  同一ホストの他の利用者アカウントから復号できてはならない (MUST NOT) — Principle I, II
- **FR-005**: 秘密鍵の at-rest 保護は、デスクトップセッション・GUI・対話的なロック解除を
  常時稼働の前提として要求してはならない (MUST NOT)。無人での起動・再起動と両立しなければ
  ならない (MUST) — Principle I(可用性)
- **FR-006**: 保管データを復号できない場合(破損・別アカウント・OS 再インストール後・他
  プラットフォームからの持込)、システムは当該ペルソナを「利用不可」として明示し、起動と
  他ペルソナ・発見機能を継続しなければならない (MUST) — ADR-0003 §4 と同挙動、Principle I
- **FR-007**: nsec エクスポート・破棄の確認要件・非可逆性・ログ/セキュリティイベント非記録の
  意味論は、Windows 版と同一でなければならない (MUST) — ADR-0003 §2・§3、Principle I, II
- **FR-008**: システムは停止要求(SIGTERM 相当)を受けたとき、既存の graceful shutdown 経路で
  安全に終了しなければならない (MUST)
- **FR-009**: システムは systemd の下で稼働状態(準備完了)を通知できるべきである (SHOULD)。
  通知できない場合でもサービスとして正常に稼働できなければならない (MUST)
- **FR-010**: 設定・永続状態・ログの配置は Linux/systemd の標準ディレクトリ規約に従うべきで
  ある (SHOULD)。運用者が配置場所を指定できるべきである (SHOULD)。データディレクトリと待受
  ポートを個別に指定することで、同一ホスト上での複数インスタンスの稼働が両立できるべきである (SHOULD)
- **FR-011**: システムはログを標準出力/システムジャーナルへ出力できるべきである (SHOULD)。
  秘密鍵・nsec をログへ出力してはならない (MUST NOT) — ADR-0003 §2、Principle II
- **FR-012**: 提供物は systemd サービス定義(unit)の例と、Linux での導入・稼働手順を含むべき
  である (SHOULD)
- **FR-013**: 秘密鍵保管ファイル/データ(保護鍵ファイル・SQLite DB)のアクセス権が他ユーザーに
  開いている場合、システムはこれを検知し、`0600` 相当(ファイル `0600`・ディレクトリ `0700`)へ
  自動的に是正して稼働を継続しなければならない (MUST)。是正の事実はセキュリティイベントに
  記録しなければならない (MUST)。是正できない場合(他ユーザー所有・読み取り専用ファイル
  システム等 — 具体条件は plan の契約で列挙)は警告し、影響を受けるペルソナを「利用不可」として扱わなければならない
  (MUST)。保護鍵ファイル・DB など全ペルソナ共有の保管物が是正不能な場合はすべてのペルソナが
  対象となるが、発見・伝搬機能(US1)は継続しなければならない (MUST) — Principle I, II
- **FR-014**: 起動時にポートのバインド失敗・権限不足などで稼働を継続できない場合、システムは
  原因が識別できるエラーメッセージとログを出力して終了しなければならない (MUST)。エラー出力は
  内部実装の詳細を漏洩してはならない (MUST NOT) — Principle II、Security Requirements(エラーハンドリング)

### セキュリティシナリオ (Gherkin / Principle I, II, IV)

```gherkin
Feature: Linux におけるペルソナ秘密鍵の at-rest 保護
  # Principle I, II / FR-003: 平文非保存
  Scenario: ペルソナ作成時に秘密鍵が平文で永続化されない
    Given Linux 上で稼働するノード
    When 利用者がペルソナを新規作成する
    Then 永続化された保管データのどこにも秘密鍵の平文が含まれない
    And 秘密鍵は保護鍵(マスター鍵)による AEAD 暗号化表現として保存される

  # Principle I, II / FR-003, FR-004: 他アカウントからの復号不能
  Scenario: 別の利用者アカウントは保管データから秘密鍵を復元できない
    Given ペルソナが保管された Linux ホスト
    And 同一ホストに別の利用者アカウントが存在する
    When 別アカウントが保管データ(DB・保護鍵ファイル)の読み取りを試みる
    Then パーミッションにより読み取りが拒否される
    And DB 単体を持ち出しても保護鍵ファイルなしでは復号できない

  # Principle I, II / FR-013: パーミッション自動是正
  Scenario: 保管ファイルのアクセス権が緩い状態で起動した
    Given 保護鍵ファイルまたは DB のアクセス権が他ユーザーに開いている
    When ノードを起動する
    Then アクセス権は 0600 相当(ファイル 0600・ディレクトリ 0700)へ自動的に是正され、稼働は継続する
    And 是正の事実がセキュリティイベントとして記録される

  # Principle I / FR-013: 是正不能時の部分劣化
  Scenario: 保管ファイルのアクセス権を是正できない
    Given 保護鍵ファイルが他ユーザー所有であり是正できない
    When ノードを起動する
    Then 警告が記録され、影響を受けるすべてのペルソナが「利用不可」と明示される
    And 発見・伝搬機能は影響を受けずに稼働を継続する

  # Principle I / FR-006: 復号不能データの隔離
  Scenario: 復号できない保管データを持ち込んで起動した
    Given 別ホスト・別アカウント・別プラットフォームから持ち込まれ復号できない保管データ
    When ノードを起動する
    Then 当該ペルソナのみ「利用不可」と明示される
    And 起動・他ペルソナ・発見機能は影響を受けずに継続する

  # Principle II / FR-011: 秘密情報のログ非出力
  Scenario: ログ・ジャーナルへの秘密鍵の非出力
    Given 稼働中のノードでペルソナの作成・エクスポート・破棄を実施した
    When 標準出力・システムジャーナルの全ログ出力を検査する
    Then 秘密鍵・nsec がいかなる形でも出力されていない

  # Principle II / FR-014: エラー時の内部情報非漏洩
  Scenario: バインド失敗時に原因は示すが内部詳細は漏らさない
    Given 待受ポートが権限不足または使用中でバインドできない環境
    When ノードを起動する
    Then 原因が識別できるエラーメッセージとログを出力して終了する
    And エラー出力は内部実装の詳細(スタックトレース・内部パス)を含まない
```

### Key Entities *(include if feature involves data)*

- **保護された秘密鍵表現**: ペルソナ秘密鍵の at-rest 形態。Windows では DPAPI BLOB、Linux では
  本機能で定める保護方式による表現。平文であってはならず、実行アカウントスコープで保護される。
  既存の SQLite 上の `secret_enc` 相当(data-model §Persona)を、プラットフォーム横断で成立させる
- **プラットフォーム実行環境**: ノードが稼働する OS/実行文脈(Windows デスクトップ / Linux
  手動起動 / Linux systemd サービス)。設定・状態・ログの配置と鍵保護方式の選択に影響する
- **サービス定義(unit)**: systemd がノードを管理するための宣言。自動起動・再起動方針・
  停止方式・実行アカウント・ディレクトリを記述する

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 利用者は Linux ホスト上で、文書化された手順に沿って追加の対話操作なしにノードを
  起動し、ネットワークへ参加(既知ピアへの接続と最初のチャンネル発見)を完了できる
- **SC-002**: Windows と Linux の両プラットフォームで、既存の契約テスト・機能(Gherkin)テストが
  同一に通過する
- **SC-003**: デスクトップセッション・GUI の無いヘッドレス Linux ホストで、無人の起動・再起動が
  成功し、ペルソナ秘密鍵が平文で保存されず、保管ファイルが他の利用者アカウントから読めない
- **SC-004**: サービス停止要求から、進行中の処理を破壊せずに安全終了するまでが systemd 既定の
  停止タイムアウト(90 秒)以内に完了する
- **SC-005**: Linux で作成したペルソナで掲載したチャンネルが、他ノード(Windows/Linux 問わず)で
  発見でき、逆方向も成立する
- **SC-006**: 復号できないペルソナが存在しても、ノードは起動し、他ペルソナと発見機能は影響を
  受けずに稼働を継続する

## Assumptions

- 対象は 64bit の一般的な Linux ディストリビューション(systemd を採用するもの)を第一とし、
  systemd を使わない環境でも手動起動での基本稼働は妨げない
- 「常時稼働ノード」の第一のユースケースは発見・伝搬への寄与(US1)であり、掲載(US2)は
  必須ではないため、US1 と US2 を独立スライスとして分離する
- Linux 版でも「自己完結・インストール物を最小化」という既存制約(ADR-0003 research R6)を
  維持し、追加の常駐サービスやデスクトップ環境を稼働前提にしない
- Linux の秘密鍵 at-rest 保護は「初回起動時に乱数生成したマスター鍵を実行アカウントの
  ディレクトリに `0600` で保管し、それで秘密鍵を AEAD 暗号化」する方式を採用(FR-003、
  Clarifications 2026-07-06)。マスター鍵ファイルの具体的な配置パス・AEAD アルゴリズム選定・
  Windows(DPAPI BLOB)と Linux 表現を同一 `secret_enc` に格納する際の識別方式は、plan
  (research/contracts)の決定を正として実装フェーズ冒頭に ADR(ADR-0009)として確定する
  (DPAPI 相当の自己完結性を Linux で成立させることが目的。ADR 確定は実装タスクの先頭ゲート)
- ネットワークプロトコル・イベントスキーマ・信頼境界・脅威モデルはプラットフォーム非依存で
  あり、本機能で変更しない(既存 spec 001 / ADR を正とする)
- トランスポート非暗号化・plain HTTP・loopback 強制(ADR-0006)は両プラットフォームで不変
- 鍵漏洩時の失効表明を v1 非目標とする判断(ADR-0003 §5)はプラットフォームに依らず維持する
- 保護鍵(マスター鍵)ファイルのバックアップ・ローテーション機構は v1 非目標とする。
  nsec エクスポート(FR-007)が唯一のバックアップ手段である(ADR-0003 §2 と同思想)
- 脅威モデルの限界(受容): root 等、データディレクトリ全体(保護鍵ファイル込み)を読める
  主体に対しては at-rest 保護は成立しない。また Linux 方式の「他アカウントからの復号不能」
  (FR-004)はファイルパーミッションに依存し、Windows/DPAPI のユーザー鍵への暗号学的拘束とは
  保証水準が異なる。いずれも自己完結制約(追加常駐なし)とのトレードオフとして受容する
- 新しい保護形式で保管した後に旧バージョンへロールバックした場合の互換はスコープ外
  (旧実装では復号失敗 = 当該ペルソナ利用不可として扱われ、起動・データは破壊されない)
- macOS など他 Unix 系は本機能のスコープ外(将来の拡張余地は排除しない)
